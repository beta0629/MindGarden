# 통합 스케줄링 센터 매칭 목록 필터 개선 기획 요약서

## 1. 제목·목표

- **제목**: 통합 스케줄링 센터 좌측 매칭 목록 필터 개선 — 신규 매칭 우선 노출
- **목표**: "전체" 목록 과다를 줄이고, **신규 매칭건**이 상단에 노출되어 결제·승인 등 **액션 처리**를 빠르게 할 수 있도록 필터 구조와 기본값·정렬 규칙을 정의한다.

---

## 2. 범위

| 구분 | 내용 |
|------|------|
| **포함** | 화면 `/admin/integrated-schedule` 좌측 "매칭 목록" 필터 영역. 필터 옵션 정의, 기본 선택값, 정렬 규칙. |
| **제외** | API 파라미터 추가(서버 사이드 필터)는 본 기획서에서 제안만 하고, 구현 Phase에서 확정. 우측 캘린더·모달·다른 화면 변경 없음. |
| **영향** | `IntegratedMatchingSchedule.js` 필터 상태·UI, (선택) `GET /api/v1/admin/mappings` 쿼리 파라미터. |

## 3. 신규 매칭건 정의 제안

두 가지 정의를 제안하며, **둘 다 반영(AND 또는 조합)** 가능하다.

| 정의 방식 | 설명 | 장점 | 단점 |
|-----------|------|------|------|
| **A. 생성일(createdAt) 기준** | 매칭 **생성일이 최근 N일 이내**인 건 (예: 7일, 14일). | “방금 만든 매칭”이 확실히 포함됨. | N일 지난 미처리 건(결제 대기·승인 대기)은 안 나옴. |
| **B. 상태 기준** | **결제 대기(PENDING_PAYMENT)·승인 대기(DEPOSIT_PENDING)** 등 “액션이 필요한 상태”만. | 액션 필요 건만 모아서 처리하기 좋음. | “신규 생성”과는 무관하게 오래된 미처리 건도 포함됨. |

**권장 제안**

- **우선 제안**: **A + B 조합**  
  - **신규 매칭** = **(createdAt 최근 N일 이내) OR (상태가 결제 대기 또는 승인 대기)**  
  - N은 **7일**을 기본값으로 제안(운영 피드백으로 14일 등 조정 가능).
- **대안 1**: A만 사용 — “최근 N일 이내”만 (구현 단순).
- **대안 2**: B만 사용 — “결제 대기 + 승인 대기”만 (액션 중심).

구현 시 **프론트 필터링**만 할지, **API에 `view=new` 같은 파라미터**를 두고 서버에서 필터·정렬할지는 구현 Phase에서 결정.

## 4. 필터 구조 제안

### 4.1 기존 vs 변경

- **현재**:  
  - **회기** 라디오: 회기 남은 매칭만(기본) | 전체  
  - **상태** 버튼: 전체, 결제 대기, 결제 확인, 승인 대기, 활성, 비활성, 종료됨, 회기 소진, 일시정지
- **문제**: “전체” 선택 시 목록 과다, “회기 남은 매칭만”만으로는 신규 생성 건이 액션보다 뒤로 밀릴 수 있음.

### 4.2 제안 구조

- **첫 번째 필터(라디오)**: 레이블을 **“보기”** 또는 **“우선순위”**로 변경하고 옵션을 아래와 같이 통합 제안.
  - **신규 매칭** (권장 기본값)
  - **회기 남은 매칭**
  - **전체**
- **두 번째 필터**: 기존 **“상태”** 버튼 유지 (전체, 결제 대기, 결제 확인, 승인 대기, 활성, 비활성, 종료됨, 회기 소진, 일시정지).
- **기본 선택값**:  
  - **보기/우선순위** = **신규 매칭**  
  - **상태** = **전체**  
  → 첫 진입 시 “신규 매칭 + 상태 전체”로, 신규·액션 필요 건이 먼저 보이도록 한다.

### 4.3 정렬 규칙

- **신규 매칭** 선택 시: **createdAt 기준 최신순(내림차순)** 정렬 필수.
- **회기 남은 매칭** / **전체** 선택 시: 기존과 동일하게 **createdAt 최신순** 유지 권장(일관성).

## 5. 사용자 시나리오

| 시나리오 | 사용자 행동 | 기대 결과 |
|----------|-------------|-----------|
| 액션 빠르게 처리 | 통합 스케줄링 센터 진입 → (기본값) 신규 매칭 | 최근 생성·결제 대기·승인 대기 등이 상단에 노출되어 결제·승인 처리 용이 |
| 회기 남은 건만 보기 | “보기”에서 “회기 남은 매칭” 선택 | remainingSessions > 0 인 매칭만 표시 (기존 동작 유지) |
| 특정 상태만 보기 | “상태”에서 “결제 대기” 등 선택 | 선택한 상태만 표시 (기존 동작 유지) |
| 전체 목록 보기 | “보기”에서 “전체” 선택 | 모든 매칭 표시 (목록 많을 수 있음은 유의) |

## 6. 필터 옵션·기본값·정렬 요약

| 항목 | 내용 |
|------|------|
| **보기/우선순위 옵션** | ① 신규 매칭 ② 회기 남은 매칭 ③ 전체 |
| **기본 선택** | 보기 = 신규 매칭, 상태 = 전체 |
| **신규 매칭 정의** | (createdAt 최근 N일 이내) OR (상태 = 결제 대기 또는 승인 대기), N=7일 권장 |
| **정렬** | 신규 매칭 선택 시: createdAt DESC. 그 외 뷰도 createdAt DESC 권장 |
| **상태 필터** | 기존 유지 (전체 + 8개 상태 버튼) |

## 7. 의존성·순서

- **선행 확인**: `GET /api/v1/admin/mappings` 응답에 `createdAt` 포함 여부 — **포함됨**(현재 코드·API 확인 완료).
- **구현 순서**: (1) 필터 옵션·기본값·정렬 스펙 확정 → (2) UI 설계(core-designer) → (3) 프론트 필터/정렬 구현(core-coder). API 파라미터 추가는 선택.

---

## 8. 리스크·제약

- **목록 수**: “전체” 선택 시 데이터 많으면 프론트 필터/정렬만으로는 지연 가능. 추후 페이지네이션·서버 필터 도입 검토.
- **N일 기본값**: 7일은 운영 피드백 후 14일 등으로 조정 가능하도록 상수화 권장.
- **멀티테넌트**: 기존과 동일하게 tenantId 기준 데이터만 노출(변경 없음).

## 10. 분배실행 (추후 설계·구현 시)

코드/디자인 수정은 하지 않고 기획 문서만 작성하는 요청이므로, **실제 호출용 분배실행 표**는 추후 설계·구현을 진행할 때 아래를 참고해 채운다.

| Phase | 담당 | 전달할 태스크 설명 요약 |
|-------|------|-------------------------|
| **Phase 1 (설계)** | core-designer | 본 기획 요약서(`INTEGRATED_SCHEDULE_FILTER_PLAN.md`) 및 §0.4(사용성·정보 노출·레이아웃)를 참고해, 통합 스케줄링 센터 좌측 “매칭 목록” 필터 영역만 설계. “회기” → “보기” 또는 “우선순위” 라디오 3옵션(신규 매칭 / 회기 남은 매칭 / 전체), 상태 버튼 유지. B0KlA·unified-design-tokens·AdminCommonLayout 전제. 산출: 필터 블록 레이아웃·라벨·스펙. 코드 작성 없음. |
| **Phase 2 (구현)** | core-coder | 위 기획 요약서와 디자이너 산출물을 참고해 `IntegratedMatchingSchedule.js`(및 관련 CSS)에서 필터 옵션·기본값·필터링·정렬 구현. 신규 매칭 정의(createdAt N일 + 상태 결제 대기/승인 대기), 기본값 신규 매칭, 정렬 createdAt DESC. `/core-solution-frontend`, StandardizedApi·기존 매칭 API 응답 구조 준수. |
