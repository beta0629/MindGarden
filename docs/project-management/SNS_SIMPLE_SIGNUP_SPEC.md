## 1. 목적

소셜 SDK로 **인증은 성공**했으나 테넌트에 **미등록 사용자**인 경우(`requiresSignup: true`), 현재는 특히 **Expo**에서 “관리자에게 문의”로 끊겨 **이탈·CS 부담**이 크다.  
본 문서는 **우리 쪽(MindGarden) SNS 가입 절차를 짧게** 만들되, **멀티테넌트·약관·보안**을 만족하는 구현 방향을 한곳에 정의한다.  
가입 이후 일상적인 재인증은 **동일 SNS 버튼(카카오/네이버)으로 로그인**하는 것을 기본 사용자 모델로 둔다.

## 2. 현황 요약 (코드 기준)

| 영역 | 동작 | 참고 경로 |
|------|------|-----------|
| 네이티브 소셜 로그인 | `POST /api/auth/social-login` — 미가입 시 `requiresSignup: true` + `socialUserInfo` | `OAuth2Controller.java` (`socialLoginWithAccessToken`) |
| Expo 로그인 화면 | `requiresSignup` 시 안내만, 가입 화면 없음 | `expo-app/app/(auth)/login.tsx` |
| 웹 OAuth 콜백 | `requiresSignup=true` 쿼리 시 회원가입 플로로 유도 | `frontend/src/components/auth/OAuth2Callback.js` |
| 웹 소셜 유틸 | `requiresSignup` 반환 처리 | `frontend/src/utils/socialLogin.js` |
| 소셜 회원가입 API | `POST /api/v1/auth/social/signup` + `tenantId`(쿼리 또는 컨텍스트) | `SocialAuthController.java` |
| 가입 서비스 | 이메일·비밀번호(선택·미전송 시 A안 서버 강난수)·`privacyConsent`·`termsConsent` 필수, 휴대폰 선택(형식 엄격) | `SocialAuthServiceImpl.java`, `SocialSignupRequest.java` |
| 가입 성공 응답 | JWT 없음, “다시 로그인” + 웹 로그인 URL | `SocialSignupResponse.java` |

**백엔드 소셜 로그인(네이티브 `social-login`) — 기존 계정 매칭 우선순위:** 아래 **(a)~(d)** 순으로 기존 사용자와 연결을 시도하고, **모두 실패할 때만** `requiresSignup: true`(간편가입 분기)로 응답한다.

| 순서 | 축 | 설명 |
|:----:|----|------|
| **(a)** | 소셜 행 | 동일 `tenantId`에서 `provider` + `providerUserId`(또는 동등 식별자)로 등록된 소셜 연동 행이 있으면 해당 사용자로 로그인(연동 유지·갱신). |
| **(b)** | 전화 | 정규화된 휴대폰 번호로 기존 `User`를 찾아 매칭(테넌트 정책·§10과 정합). |
| **(c)** | 정규화 이메일 | SNS에서 수신·정규화한 이메일과 동일한 기존 `User`가 있으면 해당 계정으로 처리. |
| **(d)** | 이메일에서 파생한 `user_id` | SNS 이메일을 **간편가입과 동일한 규칙**으로 변환한 값이 `users.user_id`와 일치하는 기존 `User`가 있으면 **간편가입으로 보내지 않고** 기존 계정 로그인(소셜 연동)으로 처리. |

**비고(제품 관점):** 위 표의 “비밀번호 필수”는 **현재 DTO·검증 규칙**이다. SNS로만 가입한 사용자는 **일상적으로 비밀번호를 입력하지 않고** 재방문 시에도 **같은 SNS로 로그인**하는 것이 기대 동작이다. DB에 저장되는 비밀번호 해시는 (A안 채택 시) **내부용 난수**로만 채워지고 **사용자가 알 필요·쓸 일이 없음**을 문서·화면·FAQ에서 일관되게 전달한다. 이메일+비밀번호 로그인은 **별도 가입 경로** 또는 레거시 계정에 한해 노출하는 것을 권장한다.

## 3. 설계 원칙

1. **테넌트 격리**: 가입·로그인 모두 `tenantId`(헤더·컨텍스트) 없으면 진행 불가. (`.cursor/skills/core-solution-multi-tenant/SKILL.md`)
2. **카카오·네이버 개발자 센터 정책은 보조**: 동의 UI·스코프 최소화에 활용하되, **서비스·테넌트 약관·개인정보**는 MindGarden 측 **최소 동의 + 버전·시각 저장**을 병행(법무 확정).
3. **하드코딩 금지**: 문구·URL은 설정·i18n·테넌트 브랜딩 정책에 따름.
4. **웹·Expo 정합**: 동일 백엔드 계약; 화면 단계 수만 플랫폼에 맞게 조정.
5. **로그인 채널: SNS 우선**: 가입·가입 완료 안내·설정·고객센터 FAQ에서 **“다음부터는 ○○(카카오/네이버)로 로그인해 주세요”**를 1차 메시지로 둔다. 포털 **이메일·비밀번호 로그인**은 보조이며, SNS 전용 계정에는 비밀번호 찾기·재설정 유도를 **하지 않는다**(정책 B안·계정 유형과 합의 후 UI에서 숨김 가능).

### 3.1 SNS 계정과 “비밀번호” 필드(개념 정리)

| 질문 | 권고 답(본 문서 SSOT) |
|------|------------------------|
| SNS로 가입했는데 DB에 비밀번호(해시)가 있나? | **있을 수 있다.** 레거시 스키마·Spring Security·이메일 계정과의 통합을 위해 `User` 엔티티에 비밀번호 컬럼이 존재할 수 있다. |
| 사용자가 그 비밀번호를 쓰나? | **SNS 우선 정책에서는 “아니오”가 목표다.** 일상 로그인은 **SNS OAuth 재호출**. 저장 값은 (A안) **서버 생성 난수의 해시**로, 사용자는 값을 알지 못한다. |
| 암호화·해시는 의미 있나? | **있다.** DB 유출·내부 접근 시에도 **평문 비밀번호 없음**을 유지하고, 혹시라도 이메일 로그인 경로가 열려 있을 때 **무의미한 강한 비밀번호**로 브루트포스 부담을 줄인다. |
| 왜 사용자에게 비밀번호를 받는 UI가 있나? | **현재 구현(C안)** 때문일 수 있다. 제품 방향이 “SNS만”이면 **A안 또는 B안**으로 API·DTO를 바꿔 **가입 화면에서 비밀번호 입력을 제거**하는 것이 이 문서의 권고와 맞다. |

---

### 3.2 보안·법무에 넘길 메모(한 블록)

- SNS만 쓰는 사용자에게 **비밀번호를 알려주거나** “기억해 두세요”라고 하지 않는다.
- **이메일 찾기·계정 통합** 등에서 이메일 로그인이 필요해지면, 그때만 별도 플로(비밀번호 설정 또는 소셜 연동 확인)를 정의한다.
- 감사 로그에 **가입 채널**(WEB/EXPO), **provider**, 동의 버전을 남긴다.

---

### 3.3 매칭·간편가입 경계

소셜 네이티브 `social-login` 분기에서 **이메일·휴대폰만**으로는 잡히지 않았지만, **이메일에서 파생한 로그인 ID(`users.user_id`)** 가 기존 행과 같으면 **별도 간편가입 화면(`requiresSignup`)으로 보내지 않는다** — §2 표의 **(d)** 단계로 기존 계정에 소셜을 연결하고 세션을 완료한다.  
카카오·네이버 등에서 **이메일 미제공·비동의**인 경우에는 **(c)·(d) 모두 적용할 수 없을 수 있으므로**, 그때는 **(a)(b)** 결과만으로 매칭 여부가 결정된다(이메일 없이도 가입·로그인 정책은 §10·제공자별 폴백과 함께 본다).  
**(d)** 에 쓰는 파생 규칙은 **간편가입 시 `SocialAuthServiceImpl`이 `user_id`를 생성할 때와 동일한 알고리즘**이어야 한다. 구현이 한쪽만 바뀌면 동일 인물이 **가입 직후에는 (d)로 잡히지 않다가** 나중에 잡히는 등 SSOT가 깨지므로, 파생 로직은 **단일 유틸**로 모으고 가입·로그인 양쪽에서 호출한다.  
구현 위치는 코드 인덱스(§9)의 `AbstractOAuth2Service`, `SocialLoginUserIdDerivation` 경로를 따르며, 세부 알고리즘은 소스 주석·테스트로 단일화한다.

## 4. 목표 UX (권고)

### 4.1 공통 (웹·Expo)

1. 소셜 로그인 SDK 성공 → 백엔드 `social-login`.
2. **`requiresSignup: true`** 이면 **가입 전용 짧은 화면**으로 이동(모달 또는 라우트).
3. 화면 구성(권장 최소):
   - **서비스 이용약관·개인정보 처리방침** 동의(체크) + 전문 링크(웹뷰 또는 외부 브라우저).
   - (선택) **마케팅 수신** 동의.
   - (정책상 필요 시만) **휴대폰** — 소셜에서 이미 제공·검증된 값이 있으면 **읽기 전용 표시 + 수정 최소화** 검토.
4. 제출 → `POST /api/v1/auth/social/signup` (본문에 `provider`, `providerUserId`, 이메일·닉네임 등 `socialUserInfo`와 합치).
5. 성공 후 **즉시 다시 `social-login` 호출**해 JWT 수령(또는 백엔드가 가입 응답에 JWT 포함하도록 확장 시 그 경로로 통일).
6. 가입 완료 화면·토스트·푸시(해당 시)에는 반드시 **“이후 로그인은 카카오/네이버 버튼을 이용해 주세요”** 를 포함한다(문구는 i18n·테넌트 정책 키로 관리).

### 4.2 백엔드 비밀번호 정책 옵션 (A / B / C)

현재 서버 구현은 **비밀번호 필수(8자 이상)** 일 수 있다. **사용자를 SNS 로그인으로 유도**하는 제품 목표와 맞추려면 아래 중 **하나를 택해 기획·법무 합의 후** 문서·API·화면을 고친다.

| 옵션 | 사용자 경험 | 서버·스키마 부담 | 비고 |
|------|----------------|------------------|------|
| **A안** | 가입 시 비밀번호 입력 **없음**. 이후 로그인은 **SNS만**. | 서버가 **내부용 강난수**를 생성·**해시 저장**; 사용자는 값을 모름. | **본 문서 권고(제품 정렬).** “암호화되어 있어도 사용자가 안 쓴다”는 질문에 정면으로 부합. 이메일 로그인·찾기 정책과 충돌 시 별도 문서로 분기. |
| **B안** | 동일하게 SNS 중심 | `SOCIAL_ONLY` 등 **계정 유형** 필드, 로그인 UI에서 이메일 폼 숨김 등 | 스키마·권한·감사 범위가 커짐. 장기적으로 가장 명확. |
| **C안** | 가입 시 **8자 비밀번호 1회** 입력 | 변경 최소 | 레거시와 호환되나 **“간편 가입”·SNS 유도 메시지와 상충**하기 쉬움. 과도기 전용으로 두고 축소 권장. |

**본 문서 권고(개정):** 신규·리뉴얼 배치에서는 **A안을 1순위**, 운영·스키마 여건상 필요하면 **B안을 로드맵**으로 둔다. **C안**은 “당장 API를 못 바꿀 때”의 임시책으로만 명시하고, **카피·FAQ에서는 ‘비밀번호는 포털 로그인용이며, 앱에서는 SNS 버튼을 쓰세요’** 식으로 혼선을 줄인다(법무 검토 필수).

### 4.3 로그인 유도(UX 카피·배치 원칙)

다음은 **하드코딩 금지** 원칙 하에 **i18n 키·기획서에 옮길 때의 의미**만 정의한다.

| 접점 | 권고 메시지 방향 |
|------|------------------|
| 로그인 허브 | **주 버튼**: 카카오 로그인 · 네이버 로그인. **보조 링크**: 이메일 로그인(필요한 테넌트만). |
| `requiresSignup` 직후 | “가입이 완료되었습니다. **지금 연결한 ○○ 계정으로** 로그인해 주세요.” |
| 비밀번호 오류 / 400 `PASSWORD_REQUIRED` 등 | “이 계정은 **○○(소셜)로 가입**되어 있습니다. 아래 버튼에서 로그인해 주세요.” (이메일 폼 반복 입력 유도 금지) |
| FAQ / 고객센터 | “SNS로 가입하셨다면 **비밀번호 없이**, 가입에 사용한 **카카오 또는 네이버**에서 로그인합니다.” |

### 4.4 로그인 화면 정보 구조(와이어 시 디자이너 전달)

- **F 패턴 상단**: SNS 2버튼(또는 테넌트가 켠 제공자만).
- **중앙 이하**: 이메일·비밀번호 필드는 **접기** 또는 별도 “이메일로 로그인” 진입으로 **2순위**.
- **회원가입** 링크는 일반 이메일 가입과 SNS 가입 안내를 구분해, SNS 가입자가 **이메일 가입 폼으로 잘못 들어가 이중 계정**을 만드는 것을 방지한다.

## 6. API·계약 (정리)

| API | 용도 |
|-----|------|
| `POST /api/auth/social-login` | SDK 토큰·프로필로 로그인 또는 `requiresSignup` |
| `POST /api/v1/auth/social/signup?tenantId=` | 신규 사용자 생성 + `UserSocialAccount` 연동 |
| (선택 확장) 가입 응답에 `accessToken`/`refreshToken` | 재호출 없이 앱 세션 완료 — **보안 리뷰 후**만 권장 |

**멱등**: 동일 `tenantId + provider + providerUserId` 재가입 시도는 기존과 동일하게 **중복 이메일·계정** 규칙으로 거절.

## 10. 휴대폰 기준 통일·복수 계정 선택·롤별 대시보드 (제품 방향)

본 절은 **이메일보다 휴대폰 번호를 공통 식별 축**으로 삼고, **동일 인물에 가까운 복수 사용자(또는 동일 사용자의 복수 소셜 연동)** 가 있을 때 **로그인 시점에 선택**하며, **내담자·상담사·어드민 등 롤에 따라 대시보드가 달라지는** 요구를 반영한다. 구현 전 **법무(개인정보·식별자)·보안(토큰·세션 고정)** 과 합의가 필요하다.

### 10.1 왜 휴대폰인가 (카카오·네이버 공통)

| 항목 | 권고 |
|------|------|
| **목표** | 같은 사람이 **카카오로도·네이버로도** 들어왔을 때, **이메일 노출/미제공 불일치**보다 **정규화된 휴대폰**으로 매칭·분기하는 쪽을 1순위로 설계한다. |
| **전제** | `tenantId` 단위로 정책을 둔다. 휴대폰은 **테넌트 내**에서의 유일성·매칭 규칙을 정의한다(글로벌 단일 ID는 비범위). |
| **제공자 차이** | 카카오·네이버 동의 항목·동의 시점이 다를 수 있음 → **휴대폰 미수신 시** 폴백(이메일·`providerUserId`만으로 분기)을 문서·API에 명시한다. |
| **정규화** | 기존 DTO와 동일하게 **국내 휴대폰 숫자만 정규화**(`SocialUserInfo` 등)한 값으로 비교한다. |
| **암호화·보관** | 저장·전송·마스킹은 기존 **전화번호 취급 표준** 및 법무 가이드를 따른다. |

### 10.2 “계정이 2개 이상”일 때의 의미와 UI

아래는 **서로 다른 `User` 행**이 동일 테넌트·동일 정규화 휴대폰과 연결되거나, **한 사용자에 여러 소셜 계정**이 붙은 경우 등 실제 스키마에 맞춰 정의를 좁혀야 한다.

| 시나리오 | 사용자에게 보이는 것 | 권고 동작 |
|----------|----------------------|-----------|
| **S1** 동일 `tenantId`·동일 휴대폰으로 **별개 User 2건** (예: 과거에 카카오·네이버로 각각 가입) | “연결된 계정이 2개입니다. 들어갈 계정을 선택해 주세요.” | **계정 선택 화면**: 각 후보에 **역할 뱃지**(내담자/상담사/어드민)·**마스킹된 표시명**·가능하면 **마지막 로그인**을 보여 주고, 선택 후 해당 `userId`로 세션·JWT를 발급한다. |
| **S2** 한 `User`에 카카오·네이버 둘 다 연동 | 보통 선택 불필요. 필요 시 “어느 SNS로 들어오셨는지”만 표시. | 최초 연동 완료 후에는 **마지막 사용 SNS** 또는 **요청 provider**로 통일. |
| **S3** 한 사람이 **동일 테넌트에서 내담자이면서 상담사** 등 **복수 롤** | “역할을 선택해 주세요” 또는 **통합 홈에서 롤 스위처** | 세션의 **activeRole**(또는 동등 개념)을 두고, **대시보드 라우트**를 롤별로 분기한다. 어드민은 별도 서브도메인·경로 정책과 충돌 없게 정한다. |

**기존 필드와의 연결:** 이미 `requiresPhoneAccountSelection`·`phoneAccountSelectionToken`이 있다면, 본 절의 “복수 계정 선택”은 그 계약을 **확장·통합**하는 방향으로 설계한다(중복 플로 방지).

### 10.3 롤별 대시보드

| 롤 | 대시보드 성격(예시) |
|----|---------------------|
| **내담자(CLIENT)** | 예약·웰니스·메시지·결제 등 소비자 홈 |
| **상담사(CONSULTANT)** | 일정·내담자·기록·수입 등 전문가 홈 |
| **어드민(ADMIN)** | 테넌트·콘텐츠·모더레이션 등 운영 화면(기존 어드민 라우팅과 정합) |

- **한 사용자·복수 롤**: 로그인 직후 **기본 진입 롤**(테넌트 설정 또는 가장 최근 사용 롤)을 두고, **헤더/설정에서 롤 전환**이 가능하도록 한다. 전환 시 **라우트 가드**로 잘못된 대시보드 접근을 막는다.  
- **한 휴대폰·User는 1개인데 롤만 여러 개**: “계정 선택”이 아니라 **“역할 선택”** UI만 노출해도 된다.  
- **웹·Expo**: 동일한 **백엔드 계약**(후보 목록·선택 완료 API·JWT 클레임 또는 세션 속성)을 쓴다.

### 10.4 API·계약(초안 — 구현 시 OpenAPI/코드로 확정)

| API / 응답 확장 | 용도 |
|-----------------|------|
| `POST /api/auth/social-login` 응답 | `requiresAccountSelection`, `candidateAccounts[]`(마스킹 표시명, `userId`, 역할 목록, 마지막 로그인 시각 등 **PII 최소**) |
| `POST /api/v1/auth/social/…` (가칭: 계정 확정) | `phoneAccountSelectionToken` 또는 신규 토큰으로 **선택한 `userId`** 확정 후 JWT 발급 |
| JWT / 세션 | 선택된 **activeRole** 또는 클레임 세트(다중 롤 시) |

상세 URL·필드명은 **core-coder** 구현 시 기존 OAuth·소셜 컨트롤러와 충돌하지 않게 정한다.

### 10.5 미결 사항(기획·법무·보안)

- 동일 휴대폰에 **의도치 않은 이중 User**를 **병합**할지, **선택만** 허용할지.  
- **어드민**은 휴대폰 매칭에 포함할지, **별도 2FA·IP 제한**을 유지할지.  
- **다른 테넌트**에 동일 휴대폰이 있을 때의 **노출 금지**(테넌트 경계).
