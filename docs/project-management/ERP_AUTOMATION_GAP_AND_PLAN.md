Phase 1 산출물(ERP_AUDIT_CURRENT_STATE §5.2, §2.1) 기준으로, **현재 수동 트리거(API/화면)만 존재하는 작업**을 모두 나열한다.

| 번호 | 작업 | 설명 | 현재 트리거 |
|------|------|------|-------------|
| 1 | 정기 일 마감(재무) | 일 단위 재무 마감·잔액 확정. 일별 통계와는 별도 "마감" 개념 | 없음(수동) |
| 2 | 정기 주 마감(재무) | 주 단위 재무 마감·집계 | 없음(수동) |
| 3 | 정기 월 마감(재무) | 월 단위 재무 마감·집계 | 없음(수동) |
| 4 | 원장 동기화 | JPA 원장/분개와 집계·리포트 간 일관성 동기화, 또는 원장 스냅샷/잔액 갱신 | 없음(수동) |
| 5 | 재무제표 생성 | 대차대조표·손익계산서·캐시플로우 생성(FinancialStatementServiceImpl) | API/화면 요청 시에만 |
| 6 | 재무 보고서(일/월/년) 생성 | ErpController /finance/daily-report, monthly-report, yearly-report | API 호출 시에만 |
| 7 | 정산 배치 | 정산 규칙 기반 계산·결과 생성(SettlementServiceImpl) | API(calculate, approve) 트리거만 |
| 8 | 할인 회계 적용 | ApplyDiscountAccounting 등 할인 회계 처리 | PlSqlDiscountAccounting API 트리거만 |
| 9 | 환불 처리(할인/부분/세션조정) | ProcessDiscountRefund, ProcessPartialRefund 등 | API/화면 트리거만 |
| 10 | HQ/지점 통합 재무 데이터 갱신 | GetConsolidatedFinancialData, GetBranchFinancialBreakdown 등 집계 캐시/스냅샷 | API 조회 시에만 |
| 11 | 매핑 일괄 동기화(선택) | SyncAllMappings — 매핑 무결성·동기화 | API 트리거만(필요 시 자동화) |

## 2. 수동 작업별 자동화 설계 제안

| 작업 | 권장 주기(cron) | 트리거 조건 | 구현체 제안 | 비고 |
|------|-----------------|-------------|-------------|------|
| 정기 일 마감(재무) | 0 10 0 * * * (매일 00:10) | 전일 마감일(D-1) 기준, 일별 통계 완료 후 | @Scheduled | 일별 통계(00:01) 이후 실행 권장 |
| 정기 주 마감(재무) | 0 15 0 * * MON (매주 월 00:15) | 전주 종료일 기준 | @Scheduled | 주간 집계 SP/로직 필요 시 연동 |
| 정기 월 마감(재무) | 0 20 0 1 * * (매월 1일 00:20) | 전월 말일 기준 | @Scheduled 또는 Quartz | 월간 의존 작업 많을 경우 Quartz로 순서 제어 |
| 원장 동기화 | 0 30 0 * * * (매일 00:30) | 당일 또는 전일 원장 기준 동기화 | @Scheduled | 원장 스냅샷/잔액 갱신 등 정의에 따라 구현 |
| 재무제표 생성 | 0 25 0 1 * * (매월 1일 00:25) | 전월 말일 기준 대차/손익/캐시플로우 | @Scheduled | FinancialStatementServiceImpl 재사용, tenant 순회 |
| 재무 보고서(일/월/년) 생성 | 일: 0 5 1 * * *, 월: 0 10 0 1 * *, 년: 0 15 0 1 1 * | 전일/전월/전년 기준 | @Scheduled | 기존 /finance/*-report 호출 로직 서비스화 후 스케줄 |
| 정산 배치 | 0 0 3 1 * * (매월 1일 03:00) | 전월 데이터 기준 정산 계산·결과 생성 | @Scheduled 또는 Quartz | 월 마감·재무제표 이후 실행 권장 |
| 할인 회계 적용 | 이벤트 또는 0 */15 * * * * (15분) | 미적용 할인 건 배치 적용 | @Scheduled 또는 이벤트 기반 | 트랜잭션 건수에 따라 주기 조정 |
| 환불 처리 | 수동 유지 권장 | 승인·검증 필요 시 자동화 제한 | (자동화 낮음) | 보안·검증 이슈로 배치 자동화는 선택 |
| HQ/지점 통합 재무 갱신 | 0 0 4 * * * (매일 04:00) | 전일 기준 통합·지점별 집계 캐시/스냅샷 | @Scheduled | 조회 성능용 캐시 갱신 목적 |
| 매핑 일괄 동기화 | 0 0 1 * * * (매일 01:00) | 무결성 검사·동기화 필요 건만 | @Scheduled | 필요 시에만 도입 |

- **cron 표현**: Spring @Scheduled 기준(초 분 시 일 월 요일).
- 우선 **@Scheduled**로 통일하고, 부하·순서 제어가 필요해지면 **Quartz** 도입 검토 권장.

## 3. 우선순위 및 Phase별 구현 순서

### 3.1 우선순위 정의

| 우선순위 | 기준 | 해당 작업 |
|----------|------|-----------|
| 높음 | 재무 마감·보고·정산 등 법적/내부 규정·결산에 직결 | 정기 일/주/월 마감, 재무제표 생성, 정산 배치, 재무 보고서(일/월/년) 생성 |
| 중간 | 운영 효율·일관성·성능 | 원장 동기화, HQ/지점 통합 재무 갱신 |
| 낮음 | 선택적·이벤트성·보안 이슈 | 할인 회계 배치 적용, 매핑 일괄 동기화; 환불 처리 자동화는 제한적 검토 |

### 3.2 Phase별 구현 순서 제안

| Phase | 내용 | 작업 목록 | 구현체 |
|-------|------|-----------|--------|
| Phase 3-1 | 재무 마감·재무제표·리포트 자동화 | ① 정기 일 마감 ② 정기 월 마감 ③ 재무제표 생성 ④ 재무 보고서(일/월/년) 생성 | @Scheduled, 기존 서비스 재사용 |
| Phase 3-2 | 주간 마감·정산 배치 | ⑤ 정기 주 마감 ⑥ 정산 배치 | @Scheduled, 월 마감 이후 순서 보장 |
| Phase 3-3 | 원장·통합 재무 갱신 | ⑦ 원장 동기화 ⑧ HQ/지점 통합 재무 갱신 | @Scheduled |
| Phase 3-4 | 선택 자동화 | ⑨ 할인 회계 배치(주기) ⑩ 매핑 일괄 동기화(필요 시) | @Scheduled; 환불은 수동 유지 권장 |

- 실행 순서 의존성: **일 마감(00:10) → 원장 동기화(00:30) → 월/주 마감(월 1일 00:20, 월 00:15) → 재무제표(00:25) → 재무 보고서 → 정산(03:00)** 순으로 시간차를 두면 무리 없음.
- 모든 스케줄러/배치는 **테넌트별 순회 시 TenantContext 설정** 필수.
