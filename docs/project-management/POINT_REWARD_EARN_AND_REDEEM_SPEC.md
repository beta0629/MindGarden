## 1. 비전·원칙

- **쇼핑몰형 체크아웃**에서 내담자는 **“사용 포인트” 입력·슬라이더·전액 사용**으로 **즉시 할인 반영된 최종 결제 금액**을 이해하고, **PG 금액 = 현금 청구분**만 남도록 한다.
- **단일 진실 원천**: 가용 잔액·예약(hold)·확정 차감·적립·clawback은 **append-only 포인트 원장 + 주문·결제 ID**로 추적; 화면 잔액은 **집계 규칙이 명문화**된 읽기 모델(또는 캐시 + 대사).
- **실패 시 복구**: PG 실패·타임아웃·사용자 이탈 시 **hold 해제 또는 역분개**가 자동·멱등으로 수행된다.
- **테넌트 정책 우선**: 적립률·사용 한도·최소 주문금액 등은 **설정 테이블**로만; 하드코딩·운영 수기 우회는 **감사 로그·사유 코드**로만 허용.

## 2. 용어 정의

| 용어 | 정의 |
|------|------|
| **적립(Earn)** | PAID(또는 정책상 확정 이벤트) 시 주문 단위로 포인트가 **가용 대기→가용**으로 전이되는 금액. |
| **사용(Redeem)** | 체크아웃에서 주문에 대해 포인트를 **할인액으로 소비**하는 행위; 원장에는 **DEBIT** 계열. |
| **만료(Expiry)** | 정책상 유효기간 종료로 **가용 잔액이 소멸**되는 이벤트(통지 정책 별도). |
| **소멸(Forfeit)** | 약관 위반·부정 적발·운영 판단 등으로 **강제 차감**(사유·어드민 권한). |
| **가용 잔액** | `총적립 - 총사용 - hold - 만료 - 소멸 ± 조정`을 **동시성 규칙**에 맞게 산출한 사용자 단위 잔액. |
| **예약(Hold)** | 주문 `CREATED`(또는 결제 시도 직전)에 대해 **사용 예정 포인트를 가용에서 잠금**; PAID 확정 시 **확정 차감**, 취소·실패 시 **해제**. |
| **역분개(Reversal)** | 잘못된 차감·중복·환불에 대해 **상쇄 분개**로 원장에 기록(삭제 금지, append-only). |
| **Clawback** | 적립 취소·환불 시 **이미 적립된 포인트 회수**(잔액 부족 시 정책: 마이너스 금지·현금 정산·제한 등). |

## 3. 정책 표 (테넌트별 설정 가능 항목)

| 정책 키 | 설명 | 예시·비고 |
|---------|------|-----------|
| `earn_rate` | 주문 **과세/할인 후 기준 금액** 대비 적립률 | % 또는 고정; **SKU별 예외** 가능 |
| `earn_cap_per_order` | 주문당 적립 상한 | 포인트 단위 |
| `min_order_for_redeem` | 포인트 사용 가능 **최소 주문 금액**(현금+포인트 합산 기준 명시) | 현금화 방지 |
| `max_redeem_per_order` | 주문당 최대 사용 포인트 | 고정 금액 또는 **주문액 대비 %** |
| `allow_pg_mix` | **포인트 + PG 혼합** 허용 | 기본 true |
| `allow_points_only` | **포인트만으로 전액 결제(0원 PG)** 허용 | false면 1원 이상 PG 강제 등 |
| `sku_redeem_blocklist` | **포인트 사용 불가 SKU** | 검사·특가·제휴 등 |
| `discount_stack_order` | **쿠폰·포인트·세금** 적용 순서 | 예: 쿠폰→포인트→부가세 재계산 vs 포인트→쿠폰 (법무·회계와 합의 후 하나로 고정) |
| `hold_ttl_minutes` | 미결제 주문 hold TTL | 만료 시 `EXPIRED` + hold 해제 |
| `refund_points_first` | 부분환불 시 **포인트 복구 우선순위** | §6과 연동 |

## 4. 상태 머신·타이밍

| 구분 | 권고 | 설명 |
|------|------|------|
| **Hold vs 즉시 차감** | **Hold 우선** | 체크아웃 확인 시점에 **가용에서 hold**; `PAID` 시 **확정 DEBIT**로 전환. 즉시 차감은 **PG 성공 전 포인트 유실 리스크**로 비권장. |
| **주문 `CREATED`** | hold 생성·금액 스냅샷 | 사용 포인트·쿠폰·세금 스냅샷 저장; **멱등 키**(사용자+카트+버전) 부여. |
| **`PAID`** | hold → 확정 차감 + (정책) 적립 트리거 | PG 승인 금액과 **스냅샷 대사**; 불일치 시 **실패·롤백** 플로우. |
| **`CANCELLED` / `EXPIRED`** | hold 전량 해제 | 배치 또는 상태 전이 훅에서 **멱등** 해제. |
| **PG 실패** | 포인트 **롤백** | hold 해제 또는 이미 확정분이면 **역분개**; 웹훅 지연 시 **결제 조회 API**와 조정. |

**원장 매핑 (초안)**  
- `ORDER_CREATED` → `POINT_HOLD`  
- `ORDER_PAID` → `POINT_COMMIT` (+ `POINT_EARN` 별도 라인)  
- `ORDER_CANCELLED` / `PAYMENT_FAILED` → `POINT_HOLD_RELEASE` 또는 `POINT_REVERSAL`

## 5. 멱등·경합

| 시나리오 | 대응 원칙 |
|----------|-----------|
| 동일 사용자 **동시 결제** 2건 | 주문·hold·commit에 **유니크 제약**(user+scenario) 또는 **낙관적 락 버전**; 가용 잔액은 **원장 합산 또는 DB 락**으로 레이스 방지. |
| **재시도** | 클라이언트 `Idempotency-Key` + 서버 **멱등 저장소**; 동일 키는 동일 응답 재생. |
| **웹훅 지연·순서 뒤바뀜** | 결제 상태는 **PG 조회 결과 우선**; 포인트는 **주문 ID + 결제 ID** 복합 키로 **중복 commit 거부**. |
| **이중 차감 방지** | `POINT_COMMIT`은 **최대 1회**; 재전달 시 no-op 성공 응답. |

## 6. 환불·부분취소

| 항목 | 정책 초안 |
|------|-----------|
| **사용 포인트 복구** | 환불 금액 산정 시 **현금 환불분 vs 포인트 복구분** 비율을 정책 표 `refund_points_first`에 따름(예: 포인트 사용분 우선 복구). |
| **적립 clawback** | 전액 환불 시 적립 **전액 회수**; 부분 환불 시 **비례 회수**(반올림 규칙 문서화). |
| **PG 부분환불** | PG는 현금만 조정; 포인트는 **원장 CREDIT**로 별도 반영 후 **amount-info·relatedTransactions**와 **대사 리포트**로 정합 확인. |
| **주문 분할 취소** | 라인 단위 취소 시 **라인별 포인트·적립 배분 테이블** 유지 권장. |

## 7. ERP 연계

| 옵션 | 회계·대사 관점 | 시스템 원칙 |
|------|----------------|-------------|
| **A: 할인액 별도 차감** | `amount-info`에 **현금 매출**과 **포인트 할인액** 필드 분리(또는 메타) | 기존 `confirm-payment`·`relatedTransactions`와 **충돌 없이 확장 필드**로만 추가; **중복 INCOME** 방지 로직 재사용. |
| **B: 할인 후 순매출만** | 포인트는 프로모션 비용으로 **ERP 외부** 집계 | 구현 단순; **테넌트 회계 요구**와 맞지 않으면 A로 이행. |

- **원칙**: ERP 쪽 **확정 금액·거래 타입**은 기존 스킬의 **`confirm-payment` / `amount-info` / `relatedTransactions`** 흐름을 **깨지 않고** 메타 또는 별도 라인으로 포인트를 흡수한다.  
- **이중 인식 방지**: 동일 주문에 대해 **INCOME이 두 번** 쌓이지 않도록 **주문·결제·ERP 확정**의 멱등 키를 공유한다(온라인 결제 스펙 5절과 정합).
