## 1. 목표·범위

### 목표
관리자가 **상담 패키지별 가격과 회기 수를 한 화면에서 CRUD**할 수 있도록, 공통코드가 아닌 **전용 "패키지 요금 관리"(또는 "요금·가격 관리") 메뉴**를 두고 해당 화면에서 패키지별 가격·회기 수를 등록·수정·조회·삭제한다.  
현재 공통코드 CONSULTATION_PACKAGE에는 회기 수만 입력 UI가 있고 가격 입력 UI가 없으며, 매핑·결제 등에서 `extra_data`의 가격을 참조하고 있으므로, 가격을 포함한 **일원화된 관리 경험**을 제공하는 것이 목표이다.

### 범위
- **포함**: (1) 관리자 전용 메뉴 추가("패키지 요금 관리" 또는 "요금·가격 관리"), (2) 패키지 목록 조회(패키지명·회기 수·가격·비고 등), (3) 패키지별 가격·회기 수·비고 등록/수정/삭제(CRUD), (4) 기존 CONSULTATION_PACKAGE 기반 플로우(매핑 생성, getPackageOptions 등)와의 연동 유지.
- **제외**: 공통코드 관리 화면 자체의 개편, 결제·ERP·할인 정책 등 다른 요금 로직 변경, 상담사/내담자 역할에 대한 가격 관리 접근 권한(관리자만 접근).
- **영향 영역**: 관리자 메뉴(프론트), 패키지 조회/수정 API 또는 공통코드 API 활용 방식, CONSULTATION_PACKAGE `extra_data` 구조 및 이를 참조하는 프론트(매핑 생성, commonCodeUtils.getPackageOptions 등).

## 2. 사용자 관점·정보 노출·레이아웃 — core-designer 전달용 태스크 설명 초안

아래 블록을 **core-designer 호출 시 전달할 태스크 설명**으로 그대로 사용할 수 있다.

---

### 2.1 사용성 요구(편하게 사용)

- **누가**: 관리자(ADMIN, 또는 패키지 요금 관리 권한을 가진 역할).
- **목적**: 상담 패키지(예: 기본/표준/프리미엄/VIP, 단회기 등)별 **가격과 회기 수를 한곳에서 보고 수정**하기 위함. 신규 패키지 추가·가격 변경·회기 수 변경을 공통코드 화면이 아닌 전용 화면에서 처리.
- **흐름**:  
  - 진입: 관리자 메뉴에서 "패키지 요금 관리"(또는 "요금·가격 관리") 클릭 → 전용 목록 화면.  
  - 목록에서 패키지별로 패키지명(또는 코드 라벨), 회기 수, 가격, 비고(선택)를 한눈에 확인.  
  - **자주 쓰는 동작**: 가격·회기 수 수정. 따라서 목록에서 바로 수정(인라인 또는 행 클릭 후 모달)이 가능하도록 하고, "새 패키지 추가"는 상대적으로 덜 쓰이므로 버튼으로 제공.  
  - 등록/수정 시: 패키지명(또는 코드값·라벨), 회기 수, 가격(원), 비고(선택) 입력. 필수값은 회기 수·가격. 저장 시 유효성(숫자, 0 초과 등) 즉시 피드백.  
  - 삭제(비활성화)는 기존 매핑에서 쓰이는 패키지는 경고 후에만 가능하도록 제약을 두거나, "비활성화"로 두어 목록에서만 숨기는 방식 중 하나를 설계에 반영.
- **제약**: 관리자만 접근. 테넌트별로 자신의 패키지만 조회·수정(멀티테넌트 격리).

### 2.2 정보 노출 범위

- **역할별**:  
  - **관리자(또는 패키지 요금 관리 권한 보유자)**: 패키지 요금 관리 메뉴 접근 가능. 목록·등록·수정·삭제(또는 비활성화) 가능.  
  - **상담사·내담자·기타**: 해당 메뉴 및 API 비노출(접근 불가).
- **노출 필드**:  
  - 목록: 패키지 코드값(또는 코드 라벨), 패키지명(한글명), 회기 수, 가격(원), 비고(선택), 정렬 순서, 사용 여부(활성/비활성).  
  - 등록/수정 폼: 패키지 코드값(신규 시 입력, 수정 시 읽기 전용 가능), 패키지명(한글명), 회기 수, 가격(원), 비고, 사용 여부.
- **비노출·제한**: 다른 테넌트의 패키지, 시스템 내부 ID는 필요 시 관리자용으로만 표시(예: 디버그용). 금액은 원 단위 정수 또는 소수 자리 제한(정책에 따라 설계).

---

### 2.3 레이아웃(배치) 요구

- **화면 구성**:  
  - **상단**: 페이지 제목("패키지 요금 관리" 또는 "요금·가격 관리"), 필요 시 부제목(예: "상담 패키지별 가격과 회기 수를 관리합니다").  
  - **상단 액션**: "새 패키지 추가" 버튼(등록 모달 또는 등록 페이지로 이동).  
  - **본문**: 패키지 목록. 테이블 형태 권장(패키지명, 회기 수, 가격, 비고, 사용 여부, 작업 버튼).  
  - **작업 버튼**: 행별 "수정"(또는 행 클릭) → 수정 모달 또는 슬라이드 패널; "비활성화/삭제"는 정책에 따라 버튼 또는 드롭다운 액션.
- **등록/수정**:  
  - **모달 방식**: 목록 위에 UnifiedModal 사용. 모달 내부에 폼(패키지명, 회기 수, 가격, 비고, 사용 여부). 저장/취소 버튼.  
  - **페이지 방식**: 목록과 별도 라우트(`/admin/package-pricing` 목록, `/admin/package-pricing/new`, `/admin/package-pricing/:id`)로 등록/수정 폼 페이지를 두는 것도 가능. 기존 관리자 설정·공통코드 관리와 일관된 패턴을 따름.
- **블록 순서**: 제목 → 액션(우측 정렬 또는 제목 옆) → 목록 테이블 → (모달은 오버레이).
- **참조**: 기존 관리자 화면(공통코드 관리, 매핑 관리)의 상단 제목·버튼·테이블 스타일, `UnifiedModal` 사용 원칙(`/core-solution-unified-modal`), 아토믹 디자인·디자인 토큰(`/core-solution-atomic-design`, `/core-solution-standardization`).  
- **반응형**: 목록은 가로 스크롤 허용 또는 카드 전환 가능. 모달은 작은 화면에서 전체 화면에 가깝게 표시.

## 4. 데이터·API 정책 제안(선택지와 권고)

가격을 **기존 공통코드 CONSULTATION_PACKAGE와 연동**할지, **별도 API/테이블**로 할지는 아래 두 가지 선택지로 정리할 수 있다.

### 선택지 A: 공통코드 CONSULTATION_PACKAGE와 연동

- **방식**: 패키지 마스터는 기존 `tenant_common_codes`(code_group = 'CONSULTATION_PACKAGE') 유지. 가격·회기 수는 `extra_data` JSON에 `{ "sessions": number, "price": number, "remark": string }` 형태로 저장.  
- **API**: 기존 테넌트 공통코드 조회/수정 API 활용. 목록은 `GET /api/v1/common-codes/tenant?codeGroup=CONSULTATION_PACKAGE`, 등록은 공통코드 생성 API, 수정은 해당 코드의 `extra_data`(및 code_label 등) PATCH.  
- **장점**: 테이블·API 추가 없음, 이미 getPackageOptions 등이 extra_data를 읽고 있어 **연동 일관성** 좋음.  
- **단점**: 공통코드 도메인에 가격이 묶여 있음. 공통코드 화면과 패키지 요금 관리 화면에서 동일 데이터를 다루므로, "가격은 요금 관리에서만 수정" 같은 정책을 운영으로 명확히 해야 함.

### 선택지 B: 별도 API/테이블

- **방식**: `package_pricing`(또는 `consultation_package_pricing`) 같은 전용 테이블을 두고, 패키지 코드값(code_value)과 tenant_id로 CONSULTATION_PACKAGE와 1:1 연결. 가격·회기 수·비고는 이 테이블에만 저장.  
- **API**: `GET/POST/PATCH/DELETE /api/v1/package-pricing`(또는 `/consultation-packages/pricing`) 전용 엔드포인트. 목록 조회 시 공통코드 CONSULTATION_PACKAGE와 조인하여 패키지명 등 표시.  
- **장점**: 요금 도메인이 분리되어 확장(할인, 유효기간 등) 시 유리.  
- **단점**: 테이블·API·마이그레이션 필요. getPackageOptions 등은 "공통코드 + package_pricing 조인" 또는 "패키지 가격 전용 조회 API"로 바꿔야 할 수 있음.

### 권고

- **단기**: **선택지 A(공통코드 연동)** 를 권고한다. 이미 extra_data에 price를 읽는 로직이 있고, 가격 입력 UI만 없으므로, **전용 화면에서 공통코드 API로 extra_data(sessions, price, remark)를 갱신**하면 구현 규모가 작고 기존 플로우와 즉시 호환된다.  
- **중장기**: 요금 정책이 복잡해지면(패키지별 할인, 구간별 가격 등) **선택지 B로 이전**을 검토할 수 있다. 이때 기획서의 "데이터·API 정책"을 B로 갱신하고, Phase 2 코더 태스크에 "전용 테이블·API 설계 및 getPackageOptions 연동 변경"을 포함하면 된다.

## 5. 의존성·순서

- **선행 작업**: 없음. (이미 CONSULTATION_PACKAGE 및 extra_data 구조가 사용 중.)
- **작업 순서**: Phase 1(설계) 완료 후 Phase 2(구현). Phase 2에서 메뉴 추가·라우트·목록·모달·API 연동을 한꺼번에 또는 단계별로 진행 가능.

## 6. 리스크·제약

- **기존 코드**: getPackageOptions, 매핑 생성 모달 등은 현재 extra_data의 price·sessions를 사용하므로, **가격 저장 위치를 바꾸지 않는 한** 기존 동작 유지. 선택지 B로 갱신 시 해당 호출부 수정 필요.
- **멀티테넌트**: 모든 조회·수정은 tenantId 기준으로 격리. API·DB 레이어에서 tenantId 필수(`/core-solution-multi-tenant`).
- **권한**: 메뉴·API 모두 관리자(또는 패키지 요금 관리 권한)만 접근 가능하도록 권한 체크 필요.

## 7. 단계별 완료 기준·체크리스트

### Phase 1 (core-designer)

- [ ] 패키지 요금 관리 화면의 레이아웃(상단 제목·액션, 목록 테이블, 모달 배치)이 문서화되었다.
- [ ] 사용성(관리자 흐름, 자주 쓰는 동작 배치)·정보 노출 범위·역할별 제한이 전달문에 반영되었다.
- [ ] UnifiedModal 사용, 기존 어드민·공통코드와의 일관성이 설계에 포함되었다.
- [ ] 코더가 참조할 수 있는 블록/컴포넌트 구성 또는 시안(또는 스펙)이 산출되었다.

### Phase 2 (core-coder)

- [ ] 관리자 메뉴에 "패키지 요금 관리"(또는 "요금·가격 관리") 메뉴가 추가되고, 해당 경로에서 목록이 노출된다.
- [ ] 목록에서 패키지명·회기 수·가격·비고·사용 여부가 테넌트별로 조회된다.
- [ ] "새 패키지 추가"·행별 "수정" 시 UnifiedModal 기반 등록/수정 폼이 동작하며, 저장 시 extra_data(선택지 A) 또는 전용 API(선택지 B)로 반영된다.
- [ ] tenantId 미존재 시 접근 불가, 권한 없는 역할은 메뉴/API 접근 불가하다.
- [ ] 기존 getPackageOptions·매핑 생성 플로우에서 패키지 가격·회기 수가 정상 반영된다.

## 8. 실행 요청문(부모 에이전트·사용자용)

다음 순서로 서브에이전트를 호출해 주세요.

1. **Phase 1**: **core-designer** 호출  
   - 전달 프롬프트: 본 문서 **§2 전체**와 **§3 Phase 1**의 "호출 시 전달할 프롬프트 요약" 내용을 그대로 또는 보완하여 전달.  
   - 산출물: 패키지 요금 관리 화면의 레이아웃·컴포넌트 구성·디자인 토큰·시안(또는 스펙 문서).

2. **Phase 2**: **core-coder** 호출  
   - 전달 프롬프트: Phase 1 산출물을 참조하라는 점과 **§3 Phase 2**의 "호출 시 전달할 프롬프트 요약"을 전달. 데이터 정책은 **§4 권고(선택지 A)** 또는 확정된 선택지로 명시.  
   - 구현 범위: 메뉴 추가, 목록 페이지, 등록/수정 모달, API 연동(공통코드 또는 전용 API), tenantId·권한 처리.

코드·파일 수정은 Phase 2(core-coder)에서만 수행하며, 기획서는 코드 변경 없이 위 산출만으로 완료합니다.
