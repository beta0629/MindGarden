# 매칭-스케줄 통합 원스톱 화면 (Matching & Schedule Integration) UI/UX 스펙

## 1. 개요 및 배경
가장 빈번하게 발생하는 '내담자-상담사 매칭' 작업과 '스케줄(일정 등록)' 작업이 별도 화면으로 분리되어 있어, 페이지 이동에 따른 업무 효율 저하가 발생했습니다. 이를 융합하여 **기존 매칭 목록**과 **기존 스케줄 캘린더**를 한 화면에 배치하고, 매칭 선택 후 스케줄 등록 모달(자동 채움)만으로 즉시 스케줄링할 수 있는 직관적이고 효율적인 원스톱 화면을 기획합니다.

---

## 2. 창의적 UI 레이아웃 아이디어 제안

### 아이디어 A: 스플릿 뷰 (Split View) — 매칭 목록(좌) + 스케줄 캘린더(우) [채택]
- **구조**: 메인 화면을 좌우 두 개의 패널로 분할합니다. 좌측은 **기존 매칭 관리의 매칭 목록**, 우측은 **기존 스케줄 캘린더**로 구성됩니다.
- **데이터**: 좌측은 `/api/v1/admin/mappings` 실 API 사용, 우측은 `/api/v1/schedules/admin` 등 실 API 사용.
- **상호작용**: 좌측 매칭 목록에서 한 매칭(행/카드)을 선택하거나 "스케줄 등록" 버튼을 누르면 → 기존 스케줄 등록 모달(ScheduleModal)이 열리고, 해당 매칭의 상담사·내담자 정보가 **자동으로 채워진 상태**로 표시됩니다. (드래그 앤 드롭은 필수가 아님. 버튼/선택으로 처리.)
- **장점**: 한 화면에서 매칭 조회·스케줄 등록까지 클릭 최소화로 처리할 수 있어 업무 속도가 향상됩니다.

### 아이디어 B: 칸반 보드 스타일 (Kanban Board)
- **구조**: 상태별(예: 신규 접수, 매칭 중, 스케줄 확정)로 열(Column)을 나누어 카드를 배치합니다.
- **상호작용**: 카드를 다음 단계 열로 드래그하면, '스케줄 확정' 열로 이동할 때 미니 캘린더 팝업이 떠서 일정을 선택하게 합니다.
- **장점**: 전체적인 진행 상태 및 파이프라인 관리에 매우 탁월합니다.
- **단점**: 캘린더 자체의 가시성이 상대적으로 낮아, 상담사별 빈 시간대를 한눈에 파악하기 어렵습니다.

### 아이디어 C: 슬라이딩 드로어 캘린더 (Sliding Drawer Calendar)
- **구조**: 평소에는 매칭 목록이 화면 전체를 차지하고, 카드를 클릭하면 우측에서 캘린더 드로어(Drawer)가 슬라이딩되어 나옵니다.
- **상호작용**: 카드 선택 후 나타난 드로어 내 캘린더에서 빈 시간을 클릭하여 할당합니다.
- **장점**: 좁은 모니터 환경에서도 화면을 넓고 쾌적하게 사용할 수 있습니다.

---

## 3. 최적의 아이디어 선정: 아이디어 A (스플릿 뷰 — 기존 매칭 목록 + 기존 스케줄 캘린더)

**선정 사유**: 상담 센터 관리 업무의 핵심은 **"어떤 내담자를 어느 상담사의 빈 시간에 넣을 것인가"**를 빠르게 결정하는 것입니다. **한 화면에서** 기존 매칭 목록(좌)을 확인하고, 기존 스케줄 캘린더(우)로 일정을 보면서, 매칭 선택 → 스케줄 등록 모달(상담사·내담자 자동 채움) → 저장만 하면 되도록 **클릭 최소화**한 플로우가 업무 효율(One-stop) 극대화에 가장 적합합니다.

---

## 4. 세부 UI/UX 스펙 (마인드가든 디자인 시스템 적용)

본 스펙은 `mindgarden-design-system`(또는 `.pen`), `unified-design-tokens.css`의 토큰과 어드민 대시보드 샘플(https://mindgarden.dev.core-solution.co.kr/admin-dashboard-sample)의 비주얼 언어를 완벽히 준수합니다.

### 4.1. 전체 레이아웃 (Layout)
- **상단 바 (Top Bar)**:
  - 배경: `var(--mg-color-background-main)` (#FAF9F7)
  - 하단 구분선: `1px solid var(--mg-color-border-main)` (#D4CFC8)
  - 내용: 브레드크럼 (`매칭 및 스케줄 관리`), 타이틀 (`통합 스케줄링 센터`), 우측에 필요 시 `+ 신규 내담자 등록` 등 주조색 버튼 배치.
- **본문 영역 (Main Content)**:
  - 패딩: `var(--mg-spacing-24)` (24px)
  - 구조: `display: flex; gap: 24px;` — **좌측 매칭 목록 320px 고정(또는 비율)**, **우측 캘린더 flex: 1**

### 4.2. 좌측 섹션: 기존 매칭 목록 (List View)
- **데이터 소스**: 실 API **`/api/v1/admin/mappings`** 사용. 목업이 아님.
- **UI 기준**: 기존 **MappingManagementPage**의 매칭 목록(카드/테이블 등)과 **동일한 정보**를 표시 — 상담사, 내담자, 상태 등. 통합 화면에 맞게 **컴팩트**하게 표시할 수 있다.
- **섹션 블록 컨테이너**:
  - 배경: `var(--mg-color-surface-main)` (#F5F3EF)
  - 테두리: `1px solid var(--mg-color-border-main)`, `border-radius: 16px`
  - 패딩: `var(--mg-spacing-24)`
- **섹션 제목**:
  - 세로 악센트 바: 폭 4px, `background-color: var(--mg-color-primary-main)` (#3D5246), `border-radius: 2px`
  - 텍스트: "매칭 목록", `font-size: 16px`, `font-weight: 600`, `color: var(--mg-color-text-main)` (#2C2C2C)
- **매칭 항목 (행/카드)**:
  - 배경: `#FFFFFF`, 테두리 `1px solid var(--mg-color-border-main)`, `border-radius: 12px`, 패딩 `16px` (컴팩트 시 축소 가능)
  - 내용: **상담사**, **내담자**, **상태** 등 MappingManagementPage와 동일한 정보. 라벨/캡션: 12px, `color: var(--mg-color-text-secondary)` (#5C6B61).
  - 각 항목에 **"스케줄 등록"** 버튼 또는 행/카드 **선택(클릭)**으로 스케줄 등록 플로우 진입.
  - 좌측 악센트 바: 상태 또는 상담 종류에 따라 주조/보조/포인트 색상 적용 가능(선택).

### 4.3. 우측 섹션: 기존 스케줄 캘린더 (Calendar View)
- **데이터 소스**: 실 API **`/api/v1/schedules/admin`** 등 사용. 기존 스케줄 API와 동일.
- **UI 기준**: **UnifiedScheduleComponent** 또는 **ScheduleCalendarView**와 **동일한** 캘린더·이벤트 표시.
- **섹션 블록 컨테이너**:
  - 배경: `var(--mg-color-background-main)` (#FAF9F7)
  - 테두리: `1px solid var(--mg-color-border-main)`, `border-radius: 16px`
  - 패딩: `var(--mg-spacing-24)`
- **캘린더 헤더 (툴바)**:
  - 상담사 필터(다중 선택 드롭다운), 날짜 이동 버튼, 일/주/월 뷰 전환 탭 (활성 탭: 주조색 #3D5246 텍스트 및 하단 border 적용).
- **캘린더 그리드 및 이벤트**:
  - 기존 캘린더 컴포넌트의 그리드·타임 슬롯·확정 스케줄 블록 스타일을 그대로 사용.
  - 확정된 스케줄 블록: 배경 `var(--mg-color-primary-light)` (#4A6354) 또는 보조색 (#6B7F72), 텍스트 #FAF9F7, `border-radius: 6px`, 타이포그래피 12px.

### 4.4. 연동 시나리오 및 플로우 (클릭 최소화)
1. **매칭 목록 확인**: 좌측에서 실 API 기반 매칭 목록(상담사, 내담자, 상태)을 확인.
2. **매칭 선택**: 한 매칭(행/카드)을 **선택**하거나 해당 항목의 **"스케줄 등록" 버튼** 클릭.
3. **스케줄 등록 모달 열림**: 기존 **ScheduleModal**이 열리고, **선택한 매칭의 상담사·내담자 정보가 자동으로 채워진 상태(Pre-filled)**로 표시됨.
4. **일정 입력 및 저장**: 모달에서 날짜/시간 등만 선택 후 [저장] 클릭 → 스케줄 생성. (드래그 앤 드롭은 필수 아님.)
5. **결과**: 저장 후 캘린더(우측) 갱신, 필요 시 좌측 매칭 목록 상태 갱신.

**정리**: 한 화면에서 **매칭 조회 → 매칭 선택 → 스케줄 등록 모달(자동 채움) → 저장**만 하면 되도록 플로우를 설계한다.

## 5. 서브에이전트 회의 안건 (기술 검토용)

이 화면을 실제 구현할 **core-coder**를 위한 기술적 검토 및 우려 사항(질문 형태)입니다.

1. **기존 컴포넌트 재사용**
   - *"좌측 리스트는 MappingManagementPage의 매칭 목록 UI(카드/테이블)를 재사용하거나 동일 정보를 컴팩트하게 표시할 수 있도록 할 수 있을까요? 데이터는 `/api/v1/admin/mappings`로 통일합니다."*
2. **캘린더 및 스케줄 API**
   - *"우측 캘린더는 UnifiedScheduleComponent 또는 ScheduleCalendarView와 동일한 데이터 소스(`/api/v1/schedules/admin` 등)와 이벤트 표시를 사용할 수 있을까요?"*
3. **ScheduleModal 연동**
   - *"매칭 선택(또는 '스케줄 등록' 버튼) 시 기존 ScheduleModal을 열고, 선택한 매칭의 상담사·내담자 ID(또는 정보)를 모달에 전달하여 Pre-filled 상태로 표시하는 연동이 기존 모달 구조에서 가능한지 확인이 필요합니다."*
4. **성능 및 렌더링**
   - *"매칭 목록과 캘린더 이벤트가 많아질 경우, 목록/캘린더 영역별 로딩·갱신 정책(페이지네이션, 캘린더 범위별 fetch 등)을 어떻게 할지 검토가 필요합니다."*
5. **캘린더 뷰의 가로/세로 스크롤**
   - *"상담사가 많아 가로 스크롤이 생길 경우, 캘린더의 시간축(세로축)과 상단 상담사 헤더(가로축) 고정이 필요합니다. 기존 UI 구조 안에서 스크롤 처리가 가능한지 확인이 필요합니다."*
6. **모바일/태블릿 반응형 처리 정책**
   - *"스플릿 뷰 특성상 화면 폭이 좁은 태블릿(iPad 등)에서는 좌우 배치가 불편할 수 있습니다. 1024px 이하 해상도에서는 슬라이딩 드로어 방식이나 상하 배치로 전환할지, 아니면 이 화면은 PC(데스크탑) 전용으로 고정할지 기술적/기획적 합의가 필요합니다."*
