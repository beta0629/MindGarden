# 급여·세금 로직 검증 및 고도화 회의 설계

**목적**: 급여(salary)와 세금(tax) 로직이 **맞는지 다시 확인**하고, **고도화가 필요한지** 판단하기 위한 회의 설계·분배실행  
**담당**: core-planner(기획 — 오케스트레이션만, 직접 분석·코딩 없음)  
**참조**: `/core-solution-planning`, `docs/standards/SUBAGENT_USAGE.md`, 기존 `SALARY_TAX_INTEGRATION_MEETING_RESULT.md`, `SALARY_TAX_INTEGRATION_LOCATION.md`, `docs/debug/SALARY_TAX_LOGIC_ANALYSIS.md`

---

## 1. 회의 안건 정리

### 1.1 급여·세금 로직 검증 (맞는지 확인)

| 확인 항목 | 설명 |
|-----------|------|
| **계산식** | 급여(등급·회기·기본급·보너스·공제 등) 및 세금(원천징수·지방소득세·부가세·4대보험 등) 계산식이 요구사항·법규와 일치하는지 |
| **기산일** | 기산일 기준 기간 vs 달력월 혼용 여부, 배치·프로시저·API·화면 노출이 일치하는지 |
| **원천징수** | 원천징수 계산·저장·통계 반영 여부 및 항목별 breakdown |
| **세목별 저장** | `salary_tax_calculations` 등 세목별 저장·조회·통계 일관성 |
| **API 일관성** | 경로(`/api/v1/admin/salary/*` 등), 요청/응답 형식, 에러 처리 일관성 |
| **테넌트/권한** | tenantId 필수 적용, 역할별(SALARY_MANAGE, TAX_MANAGE 등) 접근 제어 정합성 |

### 1.2 고도화 여부 판단

| 영역 | 판단 포인트 |
|------|-------------|
| **정확도** | 계산 오류·엣지 케이스·반올림·null/0 처리 등 |
| **성능** | 대량 계산·배치·조회 시 병목·타임아웃 가능성 |
| **UX** | 한 번에 계산 플로우, 기간 선택, 미리보기·세금 breakdown 표시 등 |
| **감사/이력** | 계산 이력·변경 이력·감사 로그 필요 여부 |
| **리포트** | 세금 통계·항목별 리포트·연말정산 대비 데이터 등 |

## 2. 참석 역할(서브에이전트) 배정

| 역할 | 서브에이전트 | 회의에서 담당할 질문/태스크 |
|------|--------------|----------------------------|
| **탐색** | **explore** | 급여·세금 관련 코드 위치, API, 엔티티·서비스·프로시저 구조를 탐색해 **현재 구조 요약** 산출(회의 자료) |
| **로직·예외 검토** | **core-debugger** | 로직 오류·엣지 케이스·예외 처리·500 가능성 관점 검토 → **문제점·개선점 목록** 제안 |
| **컴포넌트·모듈** | **core-component-manager** | 공통 모듈 재사용 여부, 중복·일관성, 적재적소 배치 관점 검토 → **제안서** |
| **구현 품질** | **core-coder** | 구현 품질·표준 준수·유지보수성 관점 검토 → 고도화 시 **수정 방향** 제안 |
| **테스트** | **core-tester** | 테스트 커버리지·시나리오·검증 가능성 검토 → 부족한 부분·고도화 시 **테스트 요구사항** 제안 |
| **UX(선택)** | **core-designer** | 급여·세금 화면 UX·정보 구성 관점에서 고도화 제안(필요 시만 호출) |

## 3. 분배실행 표

실행 순서: **Phase 0(explore) 선행** → 그 결과를 다른 에이전트에 공유하거나, **Phase 1~5는 explore 결과 없이 각자 관점만 조사 시 병렬 호출 가능**.

| Phase | 담당 서브에이전트 | 전달 프롬프트 요약 | 병렬 |
|-------|-------------------|---------------------|------|
| **Phase 0** | **explore** | 급여·세금 관련 백엔드·프론트 경로, API, 엔티티·서비스·프로시저 목록 탐색 → "현재 구조 요약" 문서 산출 | — |
| **Phase 1** | **core-debugger** | 급여·세금 로직의 오류·엣지케이스·예외·500 관점 검토 → 문제점·개선점 목록 보고 | Phase 2~5와 병렬 가능 |
| **Phase 2** | **core-component-manager** | 급여·세금 관련 화면·서버 컴포넌트/모듈 목록, 중복·일관성·적재적소 배치 분석 → 제안서 보고 | Phase 1,3,4,5와 병렬 가능 |
| **Phase 3** | **core-coder** | 급여·세금 구현 품질·표준 준수·유지보수성 검토 → 고도화 시 수정 방향 제안 | Phase 1,2,4,5와 병렬 가능 |
| **Phase 4** | **core-tester** | 급여·세금 테스트 커버리지·시나리오·검증 가능성 검토 → 부족한 부분·고도화 시 테스트 요구사항 제안 | Phase 1,2,3,5와 병렬 가능 |
| **Phase 5** | **core-designer** | (선택) 급여·세금 화면 UX·정보 구성 관점 고도화 제안 | Phase 1~4와 병렬 가능 |

**실행 요청문**: 부모 에이전트는 **Phase 0(explore)을 먼저 호출**한 뒤, 반환된 "현재 구조 요약"을 Phase 1~5 호출 시 컨텍스트로 첨부할 수 있습니다. Phase 1~5는 **동시에 병렬 호출**해도 됩니다. 각 서브에이전트 **결과를 기획(core-planner)에게 보고**하면, 기획이 아래 §6 회의 결과 문서 템플릿을 채워 **사용자에게 최종 보고**합니다.

## 4. 각 서브에이전트 호출 시 복사용 프롬프트 전문

### 4.1 explore

```
【급여·세금 검증 회의용 탐색】

MindGarden 프로젝트에서 급여(salary)와 세금(tax) 관련 코드·API·구조를 탐색해 **현재 구조 요약**을 산출해 주세요. 코드 수정은 하지 말고, 조사·목록화만 합니다.

**탐색 범위**
- 백엔드: 급여·세금 관련 Controller, Service, ServiceImpl, Repository, Entity, DTO, PL/SQL 프로시저/함수 이름·경로
- 프론트: 급여·세금 관련 페이지·컴포넌트 경로, API 호출 경로(엔드포인트)
- API: /api/v1/admin/salary/*, /api/admin/salary/* 등 급여·세금 관련 엔드포인트 목록

**산출 형태**
- "현재 구조 요약" 형태의 보고(마크다운 가능): (1) 백엔드 주요 파일·클래스·프로시저 목록, (2) 프론트 주요 파일·라우트 목록, (3) API 목록(메서드·경로·역할 한 줄 요약). 회의에서 다른 에이전트(core-debugger, core-component-manager, core-coder, core-tester)가 참고할 수 있도록 정리해 주세요.
- 참조해도 되는 기존 문서: docs/planning/SALARY_TAX_INTEGRATION_LOCATION.md, docs/project-management/SALARY_TAX_INTEGRATION_MEETING_RESULT.md, docs/debug/SALARY_TAX_LOGIC_ANALYSIS.md
```

### 4.2 core-debugger

```
【급여·세금 로직 검증 회의 — 로직·예외 관점 검토】

MindGarden의 급여(salary)와 세금(tax) 로직을 **로직 오류·엣지 케이스·예외 처리·500 가능성** 관점에서 검토해 주세요. 코드 수정은 하지 말고, **문제점·개선점 목록**을 보고해 주세요.

**검토 포인트**
1. 계산 로직: 급여(등급·회기·기본급·보너스·공제), 세금(원천징수·지방소득세·부가세·4대보험) 계산식·기산일·기간 정의 불일치 여부
2. 엣지 케이스: null/0/음수, 빈 대상 목록, 기간 역전, tenantId 누락 등
3. 예외 처리: 예외 발생 시 메시지·로깅·클라이언트 전달 일관성, 500 노출 가능 지점
4. 세목별 저장: salary_tax_calculations INSERT 여부, 세금 통계 breakdown 미제공 등

**산출 형태**
- 문제점 목록(증상·위치·심각도)
- 개선 제안 목록(우선순위·core-coder에 전달할 때 참고할 수 있도록)
- (선택) 기존 docs/debug/SALARY_TAX_LOGIC_ANALYSIS.md와의 차이·추가 발견 사항

참조: /core-solution-debug. 가능하면 explore가 산출한 "현재 구조 요약"이 있으면 그 문서를 함께 참고해 주세요.
```

### 4.3 core-component-manager

```
【급여·세금 검증 회의 — 컴포넌트·모듈 관점 검토】

MindGarden의 급여·세금 관련 **화면·서버 컴포넌트/모듈**을 다음 관점에서 검토해 주세요. 코드 직접 작성은 하지 말고, **분석·제안서**만 산출합니다.

**검토 포인트**
1. 공통 모듈 재사용: 급여·세금 화면/API에서 공통 컴포넌트·유틸·상수 재사용 여부, 불필요 중복 여부
2. 중복·일관성: TaxManagement vs ImprovedTaxManagement 등 중복 페이지/API 호출 경로 불일치(/api/admin vs /api/v1/admin) 등
3. 적재적소 배치: 컴포넌트·모듈이 아토믹 디자인·패키지 구조상 적절한 위치에 있는지, 개선 제안

**산출 형태**
- 급여·세금 관련 프론트 컴포넌트·백엔드 모듈 목록(경로·역할)
- 중복·불일치 항목 목록
- 재사용·적재적소 배치에 대한 제안(우선순위·core-coder 전달 시 참고용)

참조: /core-solution-encapsulation-modularization, /core-solution-atomic-design. 가능하면 explore가 산출한 "현재 구조 요약"을 참고해 주세요.
```

### 4.4 core-coder

```
【급여·세금 검증 회의 — 구현 품질·표준 관점 검토】

MindGarden의 급여·세금 **구현 품질·표준 준수·유지보수성**을 검토해 주세요. 이번에는 코드 수정 없이 **검토 결과와 고도화 시 수정 방향**만 제안합니다.

**검토 포인트**
1. 표준 준수: /core-solution-backend, /core-solution-frontend, /core-solution-api, /core-solution-multi-tenant(tenantId 필수) 등 프로젝트 표준 대비 이탈 여부

**산출 형태**
- 표준 대비 이탈 목록(위치·권장 사항)
- 고도화 시 수정 방향 요약(Phase별·담당 레이어)
- (선택) 기존 docs/project-management/SALARY_TAX_INTEGRATION_MEETING_RESULT.md의 체크리스트와 비교·추가 제안

참조: /core-solution-backend, /core-solution-frontend, /core-solution-database-first. 가능하면 explore의 "현재 구조 요약"을 참고해 주세요.
```

### 4.5 core-tester

```
【급여·세금 검증 회의 — 테스트 관점 검토】

MindGarden의 급여·세금 기능에 대한 **테스트 커버리지·시나리오·검증 가능성**을 검토해 주세요. 테스트 코드 작성은 하지 말고, **부족한 부분·고도화 시 테스트 요구사항**을 제안해 주세요.

**검토 포인트**
1. 단위·통합 테스트: 급여 계산 서비스·세금 계산·API에 대한 기존 테스트 유무·커버리지

**산출 형태**
- 현재 테스트 현황 요약(있는 경우 파일·범위)
- 부족한 부분 목록(영역·시나리오)
- 고도화 시 테스트 요구사항(우선순위·검증 기준)

참조: /core-solution-testing. 가능하면 explore의 "현재 구조 요약"을 참고해 주세요.
```

## 6. 회의 결과 문서 템플릿 (기획 취합용)

**사용 방법**: 부모 에이전트가 위 분배실행 표대로 서브에이전트를 호출한 뒤, 각 보고를 기획(core-planner)에게 전달하면, 기획이 아래 템플릿을 채워 **최종 회의 결과 문서**로 저장하고 사용자에게 보고합니다.

## 7. 요약

- **회의 안건**: (1) 급여·세금 로직 검증 — 계산식·기산일·원천징수·세목별 저장·API·테넌트/권한 (2) 고도화 여부 — 정확도·성능·UX·감사/이력·리포트
- **역할 배정**: explore(구조 요약), core-debugger(로직·예외), core-component-manager(컴포넌트·모듈), core-coder(구현 품질), core-tester(테스트), core-designer(선택, UX)
- **실행**: Phase 0(explore) → Phase 1~5 병렬 호출 가능. 각 결과를 **기획에게 보고** → 기획이 §6 템플릿을 채워 **회의 결과 문서** 작성 후 **사용자에게 최종 보고**.
