# ConsultantProfileModal — 급여 프로필 수동 스모크 체크리스트

**대상**: ERP → 상담사 관리(급여) → 상담사 선택 → **ConsultantProfileModal** (급여 프로필)  
**목적**: 등급 동기화(`PUT .../consultants/{id}/grade`), 급여 프로필 POST/PUT 및 `consultant_salary_options` 동기화, GET `optionTypes` 반영을 화면·DB 기준으로 확인한다.  
**관련 자동화**: 백엔드 `SalaryManagementControllerIntegrationTest`에서 API 계약 일부 검증 가능. E2E는 상담사 상세 모달 도달·데이터 의존으로 플레이크 가능 → **본 체크리스트가 필수 게이트**로 둔다.

---

## 사전 조건

- [ ] 급여 관리(`SALARY_MANAGE`) 권한이 있는 계정으로 로그인
- [ ] 테스트용 상담사 1명 이상 (등급·급여 프로필 유무를 바꿔볼 수 있는 계정)
- [ ] 브라우저 개발자 도구 **네트워크** 탭 열기 (필터: `Fetch/XHR`)

## 1) 등급만 변경 후 저장 → 재조회 시 상담사 등급 일치

1. [ ] 모달을 연다 → **수정**(또는 **생성** 후 폼 진입)으로 급여 편집 폼을 연다.
2. [ ] **상담사 등급**만 다른 값으로 변경한다 (기본급 등은 건드리지 않아도 됨).
3. [ ] **저장**한다.
4. [ ] 네트워크에서 **등급 동기화** 요청이 선행되는지 확인: `PUT` … `/api/v1/.../consultants/{id}/grade` (또는 프로젝트에 정의된 등급 갱신 URL) — 성공(2xx).
5. [ ] 이어서 급여 프로필 `POST` 또는 `PUT` … `/api/v1/admin/salary/profiles` 계열이 호출되는지 확인.
6. [ ] 모달을 닫았다가 다시 같은 상담사로 연다(또는 목록 새로고침 후 재진입).
7. [ ] **상담사 등급** 표시가 방금 저장한 등급과 일치한다 (공통코드 라벨이면 라벨·코드값이 기대와 맞는지 확인).

---

## 2) 급여 프로필 — 신규(POST) / 기존(PUT) + 옵션 추가·삭제·빈 배열

### 2-A 신규 프로필 (POST)

1. [ ] 급여 프로필이 **없는** 상담사로 모달을 연다.
2. [ ] **생성** → 필수 필드(급여 유형 등) 입력, **옵션**을 1개 이상 추가한다 (금액은 프론트 필터상 0 초과만 전송되는지 확인하려면 네트워크 페이로드의 `options` 확인).
3. [ ] 저장 후 모달을 다시 열어 **옵션·기본급·유형**이 화면에 반영됐는지 본다.
4. [ ] (가능하면) DB `consultant_salary_profiles` / `consultant_salary_options` 에 해당 `consultant_id` 행이 생겼는지 확인.

### 2-B 기존 프로필 (PUT) — 옵션 추가

1. [ ] 이미 프로필이 있는 상담사로 **수정** 진입.
2. [ ] 옵션을 **추가**하고 저장.
3. [ ] 재조회 시 추가한 옵션이 표시된다(또는 GET 응답 `optionTypes`와 일치).

### 2-C 옵션 삭제 / 빈 배열

1. [ ] 옵션이 여러 개일 때 하나를 제거하고 저장 → 재조회 시 반영.
2. [ ] **모든 옵션을 제거**해 `options: []`(또는 해당 필드 없음)으로 저장되는지 네트워크 페이로드에서 확인.
3. [ ] 재조회 시 옵션이 비어 있거나(또는 서비스 정책상 “유지”인 경우 그 정책대로) **화면·DB가 기대와 일치**하는지 확인. (백엔드 주석: null이면 유지, 빈 배열이면 전부 삭제 — 정책 확인 필수.)

---

## 3) 네트워크 — 등급 PUT 실패 시 급여 저장이 진행되지 않는지

**의도된 동작**: 등급 동기화가 실패하면 급여 프로필 `POST`/`PUT`이 호출되지 않아야 한다.

1. [ ] 개발자 도구 → **네트워크** → 요청 목록 유지.
2. [ ] 등급을 바꾼 뒤 저장한다.
3. [ ] **시나리오 A (권장)**: DevTools **오프라인** 또는 **Request blocking**으로 등급 `PUT` URL만 막거나, 프록시로 등급 API만 500 반환하도록 한 뒤 저장.
4. [ ] **기대**: 에러 토스트(또는 메시지) 표시, **`/api/v1/admin/salary/profiles`에 대한 POST/PUT이 발생하지 않음**.
5. [ ] UI 로딩 스피너가 멈추는지(저장 버튼 재시도 가능한지) 확인 — `finally`에서 `saving` 해제 여부.

## 참고

- 프론트 페이로드: `options` — 옵션 타입·금액·이름 (`ConsultantSalaryOptionItemRequest` 호환).
- 조회 응답: `optionTypes` — `{ type, amount, name }` 목록 (`ConsultantSalaryProfileResponse`).
- ERP E2E 자동화는 메뉴·권한·데이터에 민감하므로, 본 문서가 **테스터 게이트**의 기본 증빙으로 사용된다.
- 백엔드 통합 테스트( MockMvc )는 저장소 내 `SalaryManagementControllerIntegrationTest`가 **세션**(`SessionConstants.USER_OBJECT`, `TENANT_ID`)으로 인증을 모사한다. REST 클라이언트 문서상의 `Authorization` + `X-Tenant-ID`와는 다르지만, 컨트롤러 진입 후 권한·테넌트 처리 경로 검증에는 동등하다.
