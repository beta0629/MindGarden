# MindGarden Phase 1 QA Test Strategy

## 1. 목표와 범위

- **목표:** Phase 1 학원용 MVP(소비자 포털 + 관리자 포털 + 정산 배치)의 품질을 보장하고, 베타 릴리즈 전 안정성을 검증
- **범위:** 프론트엔드/백엔드/API/배치/알림/ERP 연동, 멀티테넌시 관련 시나리오 포함
- **제외:** 모바일 네이티브 앱 고유 기능(추후 Phase 2), 외부 파트너 API(계약 전 단계)

## 2. 테스트 유형 및 커버리지

| 유형 | 주요 항목 | 도구/방법 |
| --- | --- | --- |
| 단위(Unit) | 도메인 서비스, 유틸, PL/SQL 프로시저 | JUnit5, Mockito, utPLSQL |
| 통합(Integration) | API → DB → ERP Stub | Spring Test, Testcontainers |
| E2E(UI) | 소비자/관리자 시나리오 | Playwright, Cypress |
| 회귀(Regression) | 핵심 기능 변화 시 | 자동화 스위트 재실행 |
| 성능/부하 | 정산 배치, API SLA | JMeter, Gatling, AWR |
| 보안 | 인증/세션, XSS/CSRF | OWASP ZAP, 수동 점검 |
| 접근성 | 키보드/스크린리더, 대비 | Axe DevTools, Lighthouse |
| 시각적 회귀 | 디자인 시스템 컴포넌트 | Percy/Chromatic |

## 3. 테스트 계획 일정 (초안)

| 단계 | 기간 | 활동 |
| --- | --- | --- |
| T-6주 | 준비 | 테스트 환경 세팅, 데이터 준비, QA 체크리스트 작성 |
| T-4주 | 기능 테스트 1차 | API/단위/통합 테스트, 버그 triage |
| T-3주 | E2E & 회귀 1차 | 소비자/관리자 주요 플로우 시나리오 실행 |
| T-2주 | 성능/부하 | 정산 배치, 결제 API 부하 테스트 |
| T-1주 | E2E & 회귀 2차 | 버그 fix 검증, 회귀 스위트 재실행 |
| T-3일 | 릴리즈 후보 검증 | Smoke Test, 시각적 회귀, 접근성 검사 |
| T-1일 | UAT | 파일럿 학원/내부 스태프 검증, 승인 서명 |

## 4. 테스트 환경 & 데이터

- **환경:**
  - `qa-web.mindgarden.com` (프론트엔드), `qa-api.mindgarden.com` (백엔드)
  - DB: Oracle 19c QA 스키마(`MG_QA`), Redis QA 인스턴스
  - 배치: QA 서버에서 수동/스케줄 실행 가능
- **데이터:**
  - 기본 테넌트 3개(학원, 테스트, HQ), 회원 샘플, 결제/정산 샘플 세트
  - CSV/SQL 스크립트: `/qa/data/phase1/`
  - Dummy 카드/계좌 정보(결제 PG Sandbox)
- **외부 Stub:** ERP, 알림(SMS/Email) Sandbox, Payment Gateway Sandbox

## 6. 도구 및 리포트

- **이슈 관리:** Jira (프로젝트 키 `MGQA`)
- **테스트 케이스 관리:** TestRail (계정 연결 예정)
- **버전 관리:** GitHub main 브랜치, QA 태그(`qa-phase1-*`)
- **자동화:**
  - CI: GitHub Actions (PR 시 단위/통합 테스트, linters)
  - Nightly: E2E 스위트, 시각적 회귀, 접근성 체크
  - 배치 테스트: 주 2회 정산 배치 실행 후 리포트
- **리포트:** 주간 QA 보고서, 버그 버킷, 릴리즈 체크리스트 공유(Notion)

## 7. 주요 시나리오 매핑 (발췌)

| 시나리오 | 설명 | 커버 테스트 |
| --- | --- | --- |
| 소비자 회원가입/예약/결제 | 신규 고객 플로우 전체 | E2E (Playwright), API 테스트 |
| 관리자 예약 승인/수강 등록 | 상담 → 등록 → 출결 기록 | E2E, 통합 |
| 정산 배치 → 리포트 다운로드 | 배치 실행 후 리포트 검증 | PL/SQL 테스트, API |
| 알림 발송/상태 업데이트 | 예약 확정, 결제 알림 | API, E2E, 통합 |
| 권한/멀티테넌시 검증 | 타 테넌트 접근 차단 | 보안 테스트, API |

## 8. 품질 게이트 & 기준

- **기능:** Critical/High 버그 0건, Medium 이하 잔여 버그는 사용자 영향도 평가 후 승인
- **성능:** 주요 API P95 ≤ 500ms, 정산 배치(10만 건) ≤ 10분
- **보안:** OWASP Top 10 고위험 취약점 0건, 세션/쿠키 정책 준수
- **접근성:** Lighthouse 접근성 점수 ≥ 90, 필수 플로우 키보드 접근 가능
- **문서화:** 릴리즈 노트, Known Issues, 사용자 가이드 초안 완료

## 9. 버그 관리 및 릴리즈 절차

1. 버그 발견 → Jira 등록(MGQA-*) → 우선순위 지정 (Critical/High/Medium/Low)
2. 개발 수정 → PR 제출 → 자동 테스트 → QA 재테스트 → 상태 `Ready for Release`
3. 릴리즈 후보 빌드 → Smoke 테스트 통과 → QA/PO/운영 승인
4. 베타 릴리즈 배포 → 24시간 모니터링 → 이슈 발생 시 롤백 플랜 적용

## 10. 향후 계획

- QA 자동화 커버리지 80% 이상 달성 (E2E 60%, API 90%)
- 정산 배치 자동 비교 스크립트 개발(실제 ERP vs 배치 결과)
- UAT 체크리스트 및 사용자 매뉴얼 제작
- Phase 2 대비 성능/보안 테스트 스크립트 재사용 전략 수립

본 QA 전략 문서는 Phase 1 베타 릴리즈 이후 결과에 따라 업데이트됩니다.
