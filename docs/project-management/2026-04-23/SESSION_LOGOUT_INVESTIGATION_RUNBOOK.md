## 1. 목적·용어

### 1.1 목적

- **무에러 로그아웃** 등(사용자가 “로그아웃”을 누르지 않았는데 세션이 끊기는 현상)을 **주기적으로 스캔**하고, 재현에 필요한 **최소 관측치**를 표준 형식으로 남긴다.
- 동일 패턴이 반복되면 [에스컬레이션](#3-에스컬레이션)과 [Phase 연계](#4-phase-연계)에 따라 원인 분석·수정·검증으로 넘긴다.

### 1.2 용어

| 용어 | 설명 |
|------|------|
| **무에러 로그아웃** | 브라우저 콘솔·네트워크 탭에 **명확한 5xx/클라이언트 예외 없이**, 갑자기 로그인 화면으로 돌아가거나 API 호출이 연쇄적으로 실패하는 현상. (내부적으로는 401·리다이렉트·토큰 만료 등이 있을 수 있음.) |
| **idle** | 사용자 **입력·탭 포커스·내비게이션 없이** 화면만 켜 둔 상태. 제품 정책에 **idle 경고 모달·자동 로그아웃**이 있으면, “모달 직전/직후”를 반드시 기록한다. |
| **401** | HTTP **Unauthorized**. 액세스/리프레시 토큰 만료·무효·누락 시 API가 401을 반환할 수 있으며, 프론트가 이에 반응해 로그인으로 보내는 경우가 많다. |
| **refresh** | (문맥상) **액세스 토큰 갱신** 요청(또는 세션 연장 API). refresh 실패·누락은 조용한 로그아웃과 연관될 수 있다. 브라우저 “새로고침(F5)”과 혼동하지 않도록 보고서에는 **“토큰 refresh”**처럼 구체적으로 쓴다. |

## 2. 주간·격주 체크리스트

**주기:** 주간 또는 격주(격주 시에는 동일 항목을 **격주마다** 채운다).  
**담당:** [운영결정] — 예: 온콜 프론트/백엔드 로테이션, CS 1차 수집 후 개발 합류.

각 건마다 아래를 **한 행(티켓·스프레드시트·인시던트 노트)**으로 남긴다. 비어 있으면 “해당 없음” 또는 “미수집”을 명시한다.

| 항목 | 수집 내용 |
|------|-----------|
| **발생 시각** | 로컬 타임존 + UTC(가능하면). |
| **역할** | 예: ADMIN, CONSULTANT, CLIENT 등(제품 내 표기와 동일). |
| **브라우저 / OS** | 예: Chrome 134 / macOS 15, Safari iOS 18. |
| **모바일 여부** | 예: 예/아니오(태블릿은 별도 표기 권장). |
| **체류 시간** | 로그인 후 해당 화면·탭에 머문 대략 시간(분). idle 구간 포함 여부. |
| **직전 XHR 3건** | 개발자 도구 Network에서 **시간 역순 상위 3건**의 **URL(경로 중심, 쿼리는 민감 시 생략)**·**HTTP status**. 응답 본문은 저장하지 않는다([PII·보안](#6-pii보안)). |
| **멀티탭** | 동일 계정으로 **다른 탭**을 연 상태였는지, 몇 개 탭인지(대략). |
| **tenant** | **마스킹된 테넌트 식별자**만(예: `tenant-***42`). 원문 tenantId·도메인·고객사명 전체는 기록하지 않는다. |
| **idle 모달 직전 여부** | idle 경고/연장 모달이 **뜨기 직전**에 로그아웃되었는지, 모달 후 **확인/연장** 후에도 끊겼는지. |

**추가 메모(권장):** 사용자가 수행한 **마지막 의도적 액션**(저장, 페이지 이동, 파일 업로드 등) 한 줄.

## 3. 에스컬레이션

- **동일 테넌트** 또는 **동일 역할**에서 보고된 “무에러 로그아웃”류 사건이 **주 N건**을 넘으면 **deep-dive**를 시작한다.
- **N 및 집계 창(자연주 vs 롤링 7일)**은 **[운영결정]**으로 확정한다. 초안에서는 문서에 숫자를 고정하지 않는다.
- deep-dive 시에는 [§2](#2-주간격주-체크리스트) 표를 **최소 3건 이상** 모아 공통 분모(시간대, 엔드포인트, idle, 멀티탭)를 정리하고, [§4](#4-phase-연계)에 따라 인벤토리·재현·수정·검증을 태운다.

## 4. Phase 연계

표준 흐름(요지):

1. **explore** — 세션·토큰·idle·라우트 가드·API 에러 처리 관련 **인벤토리** 갱신(경로·갭·우선순위).
2. **core-debugger** — 인시던트 메모와 직전 XHR 3건을 바탕으로 **재현 절차·가설·로그 해석** 정리.
3. **(게이트) core-coder** — 합의된 수정 범위에서만 패치. 범위 밖 리팩터 금지 원칙은 프로젝트 코딩 룰을 따른다.
4. **core-tester** — 단위·통합·E2E 및 회귀(세션·로그인·idle 시나리오)로 검증.

**검증 게이트(한 줄, 출처 고정):**  
`docs/project-management/CORE_PLANNER_DELEGATION_ORDER.md` — **코드 변경이 수반된 배치는 반드시 `core-tester`로 검증**한다(스모크·회귀·콘솔 오류 0건 등). **테스터 통과 전 작업 완료로 보고하지 않는다.**

전체 위임·직접 수정 금지 맥락은 동일 문서의 사용자 강제 규칙 표를 참조한다.

## 5. 운영·인프라 (선택 절)

인시던트가 **특정 시간대·특정 테넌트에만 집중**될 때, 인프라 관점 보조 관측을 한다. **구체 셸 명령·로그 경로는 TBD** — 배포 환경(스테이징/운영)별로 `[운영결정]` 후 이 절에 채운다.

| 영역 | 점검 자리(초안) |
|------|-----------------|
| **nginx 세션** | upstream timeout, `proxy_read_timeout`, keepalive, rate limit과의 상관 — **설정 스냅샷 날짜·환경**만 기록, 비밀·전체 설정 덤프는 금지. |
| **앱 로그 키워드** | 예: `401`, `Unauthorized`, `refresh`, `session`, `expired`, `idle`(제품 로거 접두사가 있으면 그 접두사). **전화번호·이메일·토큰 원문 검색 금지** — 상관 ID·요청 ID 단위로만 추적. |

## 6. PII·보안

- **토큰**(액세스·리프레시·세션 쿠키 값)·**비밀번호**·**전화번호 원문**·**주민번호 등 식별자**는 티켓·슬랙·런북 **어디에도 붙이지 않는다.**
- 네트워크 캡처·HAR는 기본적으로 **수집 금지**. 불가피할 경우 보안 절차에 따라 **마스킹·보관 기간**을 따른다.
- 로그·재현 보고에는 **상관 ID**(요청 ID, trace ID 등)와 **마스킹된 tenant**만 권장한다.
- 사용자 인용 시 **계정 식별 가능한 표현**을 최소화한다.
