# 코어 에이전트 캡슐화·모듈화·컴포넌트 관리 회의 요약

## 1. 회의 목적·반영 사항 요약

**회의 목적**: 다음 두 가지가 프로젝트에 반영되었음을 정리하고, 각 코어 에이전트가 어떻게 협업할지 회의 결과를 문서로 남긴다.

### 반영된 내용

| 항목 | 설명 |
|------|------|
| **캡슐화·모듈화 스킬** | `.cursor/skills/core-solution-encapsulation-modularization/SKILL.md` — 경량화·단순화·반복 제거를 위해 **모든 코어 에이전트**(기획·디자이너·퍼블리셔·코더·디버거·테스터) 작업 시 적용. |
| **컴포넌트 관리 에이전트** | `core-component-manager` — 화면·서버 컴포넌트의 **중복 여부 제안**, **적재적소 배치 제안**, 목록·계층 정리. **core-coder와 한 팀**: 제안 → 코더가 실제 이동·통합·배치 실행. 코드/디자인 직접 작성 없이 분석·제안·문서화만 수행. 에이전트 정의: `.cursor/agents/core-component-manager.md`. |

---

## 2. 캡슐화·모듈화 스킬 적용 원칙

- **목표**: 경량화(불필요한 중복·과도한 추상화 감소), 단순화(한 단위 한 책임·의존 단순), 반복 제거(동일·유사 패턴 공통화).
- **캡슐화**: 경계 명확화, 내부 구현 숨김, 스펙·문서로 단위별 책임 명시.
- **모듈화**: 단위 분리(아토믹 디자인·표준 구조 준수), 의존성 최소화·순환 의존 금지, 공통 추출(두 곳 이상 사용 시 한 번만 정의).
- **단순 반복 제거**:
  - **기획**: 유사 Phase·요구사항 → 공통 Phase 템플릿·체크리스트로 재사용.
  - **디자이너**: 동일·유사 컴포넌트·토큰 → 한 스펙에 통합, 신규는 “기존 XXX 스펙 확장/참조”로만 기술.
  - **퍼블리셔**: 동일 마크업 패턴 → 공통 HTML 조각·클래스 세트 정의·문서화.
  - **코더**: 동일·유사 코드 → 공통 함수·훅·컴포넌트·서비스로 추출.
  - **컴포넌트 관리**: 화면·서버 모듈 목록·재사용·중복은 **core-component-manager**와 협업해 정리.

## 3. core-component-manager 역할·호출 시점

### 역할

- **중복 컴포넌트 제안**: 동일·유사 기능의 중복 컴포넌트/모듈이 있는지 식별하고, 통합·재사용 여부를 제안한다.
- **적재적소 배치 제안**: 각 컴포넌트/모듈이 **어디에 두는 것이 맞는지**(common / ui / 도메인별, Atoms·Molecules·Organisms 계층, 서버 패키지·계층) 제안하고, 잘못된 위치의 이동·재배치 방안을 문서로 제시한다.
- **화면 컴포넌트**: `frontend/src/components/` 하위 목록·계층 정리, 아토믹 디자인·BEM·디자인 토큰 준수 점검, common/ 재사용 검토.
- **서버 모듈**: Controller/Service/Repository 등 단위 목록, 패키지·계층 표준 준수 점검.
- **제한**: 코드·디자인·테스트 코드 직접 작성·수정 없음. **분석·제안·문서화만** 수행. 실행은 **core-coder**가 담당(한 팀).

### core-coder와 한 팀

- **component-manager**: 중복 여부·적재적소 배치를 **제안**하고 문서화.
- **core-coder**: 제안을 받아 **실제 이동·통합·배치** 수행. 작업 후 인벤토리·배치 제안서 갱신을 component-manager에 요청.
- Phase 기획 시 **component-manager + core-coder**를 함께 배정하는 것을 권장.

### 호출 시점

- 컴포넌트 정리·재사용·중복 제거 Phase를 **기획할 때** — 1단계로 “현재 컴포넌트/모듈 목록·중복 분석” 의뢰 후, 결과로 Phase·담당 배정.
- “컴포넌트 인벤토리 1회 작성”, “통합·리팩터 제안 수립” 등 **인벤토리·제안이 필요할 때**.
- 제품에 `core-component-manager` 타입이 없으면 **generalPurpose** 호출 시 프롬프트에 “core-component-manager 역할로 .cursor/agents/core-component-manager.md 및 core-solution-encapsulation-modularization 스킬을 참고하여 컴포넌트/모듈 관리(목록·중복·제안·문서)만 수행해주세요”를 명시.

## 4. 에이전트별 협업 요약

| 에이전트 | 협업 내용 |
|----------|-----------|
| **core-planner (기획)** | 컴포넌트 정리·재사용 Phase 기획 시 **1단계로** core-component-manager에게 “현재 컴포넌트/모듈 목록·중복 분석” 의뢰 → 결과를 바탕으로 Phase·담당 배정. 캡슐화·모듈화 스킬에 따라 Phase 템플릿·체크리스트 재사용. |
| **core-designer (디자이너)** | 새 컴포넌트 설계 시 **기존 공통 컴포넌트 재사용 여부**를 core-component-manager 산출물(인벤토리·제안서)에서 참조. 동일·유사 컴포넌트·토큰은 한 스펙에 통합하고 신규는 확장/참조만 기술. |
| **core-publisher (퍼블리셔)** | 마크업 시 사용할 **공통 마크업·클래스 세트**를 컴포넌트 인벤토리에서 확인. 동일 패턴은 공통 HTML 조각·클래스로 정의·문서화. |
| **core-coder (코더)** | **core-component-manager와 한 팀**. component-manager의 **중복 제안·적재적소 배치 제안**을 받아 실제 코드 이동·통합·배치 수행. 신규 컴포넌트 추가 시 인벤토리·배치 제안서 갱신을 component-manager에 요청. 공통 함수·훅·컴포넌트 추출로 반복 제거. |
| **core-debugger (디버거)** | 컴포넌트/모듈 정리·수정 제안 시 **모듈 경계를 고려**해 수정 제안(영향 범위·의존성 명시). 코드 직접 수정은 core-coder 위임. |
| **core-tester (테스터)** | 컴포넌트 통합·리팩터 후 **영향 받는 테스트 목록** 정리 협업. core-component-manager 제안·인벤토리와 연계해 테스트 범위·시나리오 제안. |

---

## 5. 다음 액션

- **컴포넌트 인벤토리 1회 작성**이 필요할 때: core-planner가 **core-component-manager**를 호출해 “현재 화면·서버 컴포넌트/모듈 목록·중복·재사용 가능 여부 분석 및 문서화”를 의뢰한다. 결과는 `docs/project-management/COMPONENT_INVENTORY.md` 등으로 저장하고, 이후 Phase·디자이너·퍼블·코더가 참조한다.
- **재사용·중복 제거 Phase**를 넣을 때: 기획 단계에서 위 인벤토리·제안을 바탕으로 Phase와 담당(core-coder 등)을 배정하고, 실행 후 인벤토리 갱신을 core-component-manager에 요청한다.
- 모든 코어 에이전트는 작업 시 **캡슐화·모듈화 스킬**(`/core-solution-encapsulation-modularization`)을 참조하고, 컴포넌트/모듈 관련 정리·제안이 필요하면 **core-component-manager** 활용을 전제로 협업한다.
