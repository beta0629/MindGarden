## 1. 제목·목표

- **제목**: 영문 한글화 기획서
- **목표**: 사용자에게 노출되는 영문 텍스트의 범위·우선순위·역할·작업 목록을 정리하고, 한글화 작업 시 참조할 기획 문서를 제공한다. (본 문서는 기획·범위·산출물만 정리하며, 실제 코드 수정·i18n 구현은 별도 Phase에서 수행)

---

## 2. 범위 정의

### 2.1 한글화 대상

- **대상**: **사용자에게 노출되는** 영문 텍스트만 한글화 대상으로 한다.
  - UI 라벨, placeholder, 버튼 문구, 에러/시스템 메시지, 접근성(aria-label, sr-only), 배지·상태 문구, 차트 라벨 등.
- **브랜드 문구**: "Core Solution", "CoreSolution", 로고 alt, 푸터 저작권 등 **브랜드 관련 문구는 한글화 범위에서 제외**하고 유지한다. (선택적으로 4단계에서 톤앤매너 검토 후 한글화 여부를 재검토할 수 있음)

### 2.2 제외 범위

| 구분 | 설명 | 비고 |
|------|------|------|
| 코드 식별자 | 변수명, 함수명, 컴포넌트명, 키 이름 | 한글화 대상 아님 |
| 로그·주석 | 개발자용 로그 메시지, 소스 주석 | 한글화 대상 아님 |
| API 경로·쿼리 파라미터 | `/api/v1/...`, 쿼리 키 | 한글화 대상 아님 |
| 테스트·스토리북 전용 | 테스트/스토리 문구, 디자인 쇼케이스 샘플 문구(사용자 미노출) | 4단계 선택 시 검토 가능, 기본 제외 |
| 외부 서비스 고유명 | PG사·API 필드명 등 업계 관용 표기 유지 가능 | 필요 시 툴팁·도움말로 한글 설명 보강 |

## 3. 우선순위(단계별)

| 단계 | 대상 | 설명 |
|------|------|------|
| **1단계** | 에러 메시지, 접근성(aria-label/sr-only), 공통 노출 메시지 | 사용자 경험·접근성에 직결, 우선 한글화 |
| **2단계** | PG/결제·시스템 설정 등 자주 보이는 폼 라벨·placeholder | 관리자·설정 화면에서 빈도 높은 영역 |
| **3단계** | 관리자 대시보드·역할 설정 placeholder, 이메일 예시 등 | 그 외 관리자·폼 관련 노출 텍스트 |
| **4단계(선택)** | 디자인 쇼케이스·스토리북, 차트 라벨, 브랜드 문구 한글화 | 내부/개발자용 또는 브랜드 정책에 따라 선택 적용 |

## 4. 역할·담당

| 역할 | 담당 내용 |
|------|------------|
| **기획** | 범위·우선순위 확정, 예외 정책(브랜드 유지·제외 항목) 정의, 작업 목록·Phase 정리, 본 기획서 유지·갱신 |
| **디자인** | 브랜드/버튼 문구 한글화 시 톤앤매너·표현 가이드 제안, 필요 시 용어 통일표 작성 |
| **퍼블리셔/코더** | 라벨·placeholder·메시지 치환, 접근성 텍스트(aria-label, sr-only) 한글화, 실제 코드 반영 |
| **i18n 도입 시(선택)** | 리소스 키 설계·파일 구조·다국어 확장 방안은 **별도 i18n 기획**에서 다룸 |

## 5. 작업 목록(카테고리별)

검색 결과 카테고리별 한글화 대상 여부와 우선순위 단계를 표로 정리한다. 파일 경로는 요약 수준으로만 기재한다.

| 카테고리 | 대략 건수 | 한글화 대상 | 우선순위 | 파일/영역 요약 |
|----------|------------|-------------|----------|----------------|
| **브랜드/로고** | 약 25곳 | **N** (유지) | 제외(또는 4단계 선택) | UnifiedHeader, useBranding, GNB, 로그인/홈페이지, 푸터 저작권 등 |
| **placeholder** | 약 25곳 | **Y** | 2~3단계 | PgConfigurationForm, SystemConfigManagement, StaffManagement, ForgotPassword, CommonCodeForm 등 |
| **폼 라벨** | 약 15곳 | **Y** | 2단계 | PG 설정(Merchant ID, Store ID, API Key, Secret Key, Webhook/Return/Cancel URL, API URL), OPS 승인 관리 등 |
| **버튼/UI 샘플** | 약 40곳 | **N** (쇼케이스) | 4단계 선택 | 디자인 쇼케이스·스토리북(Small, Medium, Primary Button 등) — 사용자 미노출 |
| **aria-label/sr-only** | 약 5곳 | **Y** | 1단계 | "Close", "Calendar icon", "Add button" 등 |
| **에러/시스템 메시지** | 1곳 | **Y** | 1단계 | ajax.js — 'Tenant ID is required. Please log in again.' |
| **상태/배지** | 1곳 | **Y** | 3단계 | WellnessNotificationList — "NEW" |
| **차트 라벨** | 3곳 | **Y** | 3~4단계 | ApiPerformanceChart — 'Cache', 'Cache Hit', 'Cache Miss' |
| **테스트 전용** | 약 15곳 | **N** | 제외 | 테스트/스토리 문구 — 사용자 미노출 |

### 5.1 단계별 작업 요약

- **1단계**: ajax.js 에러 메시지 1건, aria-label/sr-only 5곳, 기타 공통 노출 메시지.
- **2단계**: PG 설정 관련 3개 파일 내 폼 라벨·placeholder, 시스템 설정·OPS 관련 폼.
- **3단계**: 관리자 대시보드·역할 설정 placeholder, 이메일 예시, WellnessNotificationList "NEW", ApiPerformanceChart 라벨 등.
- **4단계(선택)**: 스토리북·쇼케이스 문구, 차트 라벨 보강, 브랜드 문구 한글화 검토.

---

## 6. 예외 정책

- **브랜드**: "Core Solution", "CoreSolution", 로고 alt, 푸터 저작권 문구는 **기본 한글화 제외**. 사업/마케팅 정책으로 한글화할 경우 4단계에서 별도 결정.
- **코드·로그·API**: 식별자, 로그, 주석, API 경로는 한글화 대상 아님.
- **테스트·스토리북**: 사용자에게 노출되지 않는 테스트/스토리 문구는 기본 제외.
- **업계 용어**: PG·API 필드명 등 관용 표기를 유지할 경우, 한글 라벨 또는 툴팁으로 설명만 보강할 수 있음.

## 7. 다음 단계

1. **기획 확정**: 이해관계자 검토 후 범위·우선순위·예외 정책 확정.
2. **용어 통일표(선택)**: 디자이너·기획이 자주 쓰는 용어(예: "저장", "취소", "Merchant ID → 가맹점 ID") 정리.
3. **실행 Phase 설계**: 1단계부터 core-coder(또는 퍼블리셔)에게 치환 작업 분배; 필요 시 core-designer에게 버튼/공통 문구 톤앤매너 검토 의뢰.
4. **i18n 도입 검토**: 다국어 확장이 필요하면 리소스 키·파일 구조·적용 범위에 대한 **별도 i18n 기획서** 작성 후 진행.
