# 무중단 배포 도입 시 — 배포·컷오버 직후 검증 체크리스트

**목적**: 블루/그린·카나리·트래픽 스위치 등 **무중단 배포**가 도입된 뒤, **배포 직후·컷오버 직후**에 운영·릴리즈 담당이 빠르게 통과/실패를 판별할 수 있는 **수동·반자동 검증** 순서를 한 곳에 모은다.

**범위**: 문서만. 프로덕션 코드 변경 없음. 아래 URL·호스트·토큰은 **환경별 placeholder**로 치환한다.

**연관 문서**

- 로컬·CI에서 복사 실행하는 명령 순서: [`docs/project-management/NEXT_LOCAL_VERIFICATION_COMMANDS.md`](../../project-management/NEXT_LOCAL_VERIFICATION_COMMANDS.md)
- ERP 재무·환불 허브 등 **수동 스모크** 상세(번호 1~10·`tenantId`/API 패턴): [`docs/guides/testing/ERP_FINANCIAL_HUB_SMOKE.md`](./ERP_FINANCIAL_HUB_SMOKE.md)
- 테스트 피라미드·E2E 위치·헤더 표준: [`.cursor/skills/core-solution-testing/SKILL.md`](../../../.cursor/skills/core-solution-testing/SKILL.md), `docs/standards/TESTING_STANDARD.md`

**보안**: 자격 증명·시크릿은 문서·PR에 넣지 않고, **Secrets·런북 전용 저장소**만 사용한다. E2E 변수 우선순위는 `ERP_FINANCIAL_HUB_SMOKE.md` 「E2E 선행 조건」을 따른다.

## 1. 배포 전 (프리플라이트)

- **헬스 엔드포인트 기준선**: 대상 환경의 API 베이스(`https://{API_HOST}`)에 대해 `curl -sfS "{API_BASE}/actuator/health"` 또는 팀 표준 헬스 URL이 **200**이고 본문이 기대 JSON(또는 `UP`)인지 확인한다. (엔드포인트명은 저장소·운영 런북 SSOT에 맞출 것.)
- **정적 자산·인덱스**: 프론트 배포물의 `index.html`(또는 CDN 진입점)을 `curl -sfSI`로 받아 **ETag / `Cache-Control` / 빌드 해시 주입 스크립트**가 배포 태그·커밋 SHA와 일치하는지 확인한다. (예: `main.*.js` 파일명 해시가 릴리즈 노트와 동일한지.)
- **API 인가 스모크(토큰 없음)**: 보호된 `GET/POST /api/v1/...` 1건에 **Authorization 없이** 호출해 **401/403**이 명세대로 나오는지 확인한다(공개 엔드포인트 오동작 방지).
- **API 정상(토큰 있음)**: 스테이징·프로덕션 **전용 서비스 계정**으로 로그인해 획득한 `Bearer`와 `X-Tenant-ID: {tenantId}`로 **읽기 전용** API 1건이 **200**·`success`·필수 필드가 있는지 확인한다.
- **롤백 스위치·릴리즈 식별자 placeholder**: 로드밸런서/게이트웨이/플랫폼에서 **이전 버전 트래픽 비율**·**타깃 그룹**·**이미지 태그**를 되돌릴 수 있는 위치와 담당자를 런북 한 줄에 적어 둔다. (실제 스위치 이름은 인프라 SSOT.)
- **세션·스티키 정책**: 컷오버 중 **세션 고정**이 필요한 경우 LB 쿠키·TTL이 배포 전후 동일한지, 무중단 전략(드레인·웜업)과 충돌 없는지 확인한다.
- **E2E 선행(선택)**: 컷오버 직전 스테이징에서 `BASE_URL`·백엔드 8080 정합을 맞춘 뒤, [`NEXT_LOCAL_VERIFICATION_COMMANDS.md`](../../project-management/NEXT_LOCAL_VERIFICATION_COMMANDS.md) 절 3처럼 **ERP Playwright 한 스위트**를 돌려 회귀를 고정한다.

## 2. 배포 직후 (새 버전이 트래픽을 받기 직전·직후)

- **헬스 재확인**: 새 인스턴스/팟 그룹에 대해 `curl -sfS`로 헬스가 **연속 2~3회** 성공하는지 확인한다(콜드 스타트·DB 연결 지연 대비).
- **API 401/200 재확인**: 절 1과 동일한 **무토큰 401**·**토큰+`X-Tenant-ID` 200** 조합을 **새 타깃** 호스트명(또는 내부 DNS)으로 반복한다.
- **정적 index·번들 해시**: 사용자가 보는 오리진에서 `index.html`을 내려받아 **스크립트 `src` 해시**가 이번 배포 아티팩트와 일치하는지 확인한다. CDN이면 **캐시 무효화/버전 쿼리** 후 재확인한다.
- **세션 연속성**: 기존 브라우저 세션(또는 테스트 계정)으로 **한 번의 요청**이 새 버전에서도 **로그아웃/403 급증 없이** 이어지는지 확인한다. 이상 시 스티키·쿠키 도메인·SameSite 변경을 의심한다.
- **치명 오류 모니터링**: APM·로그에서 배포 시각 전후 **5xx 급증**, **인증 실패 급증**, DB 커넥션 고갈이 없는지 5~10분 구간을 본다.
- **롤백 준비 확인 placeholder**: 「절 1에서 기록한 스위치」로 **즉시 이전 안정 버전**으로 되돌릴 수 있는지 드라이런(스테이징) 또는 체크리스트 틱만 수행한다.
- **ERP 로그인 스모크 1경로(최소)**: 브라우저로 **관리자·ERP 권한 계정**으로 로그인한 뒤, [`ERP_FINANCIAL_HUB_SMOKE.md`](./ERP_FINANCIAL_HUB_SMOKE.md) 상단 **1~3번**에 해당하는 경로만 수행한다 — `/erp/financial` 진입·로드, `/erp/refund-management` 이동, 다시 `/erp/financial` 복귀 시 **탭/화면 끊김 없음**. 개발자 도구에서 **콘솔 치명 오류 없음**·**`tenantId` 전달 방식 기존과 동일**(해당 문서 8~10번 기준)을 함께 본다.

## 3. 컷오버 후 (트래픽 100%·스위치 완료 후)

- **전 사용자 경로 헬스**: 공개 로그인·랜딩·API 게이트웨이에 대해 외부에서 `curl` 또는 합성 모니터링이 **200/302 기대 패턴**을 유지하는지 확인한다.
- **API 샘플링**: `X-Tenant-ID`를 바꿀 수 있는 환경이라면 **테넌트 A/B** 각각 1건씩 **200**·데이터 격리가 유지되는지 확인한다.
- **정적 자산 최종 일치**: 컷오버 완료 시각 이후 `index.html`과 메인 번들 해시가 **의도한 릴리즈**와 일치하는지 재확인한다(구버전 캐시 잔존 여부).
- **세션·웹소켓(해당 시)**: 장시간 로그인 사용자·관리자 화면에서 **재로그인 강제**·**401 연쇄**가 없는지 샘플링한다.
- **ERP 확장 스모크(선택)**: 여유가 있으면 `ERP_FINANCIAL_HUB_SMOKE.md` **4~7번**(탭 전환·달력 이전/다음·새로고침)까지 확장한다. **9~10번**(`tenantId`·API 패턴)은 변경 PR이 있을 때 필수에 가깝게 본다.
- **롤백 스위치·포스트모템 placeholder**: 컷오버 후 **N분 관찰 창** 동안 지표가 안정이면 스위치 잠금(의도치 않은 롤백 방지) 여부를 기록한다. 이상 시 **절 1의 롤백 절차**로 되돌리고, 타임라인·로그 ID를 포스트모템 템플릿에 남긴다.
- **E2E 회귀(선택)**: 프로덕션 대역이 아닌 **스테이징 동일 태그**에서 `tests/e2e` ERP 스펙 일부를 재실행해 CI와 현장 결과를 맞춘다. (`playwright.config.ts`, `NEXT_LOCAL_VERIFICATION_COMMANDS.md` 절 3.)

| 배포 전 | 헬스 OK, 무토큰 401, 토큰+테넌트 200, index/번들 해시·릴리즈 정합, 롤백 경로 명시 |
| 배포 직후 | 새 타깃 헬스·401/200·정적 해시·세션·ERP 1경로 스모크·5xx/인증 모니터링 |
