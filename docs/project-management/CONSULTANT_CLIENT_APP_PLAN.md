# 상담사·내담자 중심 앱 기획서

> **작성일**: 2026-05-12  
> **작성자**: core-planner  
> **상태**: 기획 완료 → 분배실행 대기

---

## 1. 목표

현재 어드민(ADMIN) 위주로 개발된 MindGarden 플랫폼을 **상담사(CONSULTANT)와 내담자(CLIENT) 중심의 정교한 UX**로 전환한다. 백엔드 API는 이미 충분히 구현되어 있으므로, **프론트엔드 UI/UX 설계·구현**에 집중하여 상담사·내담자가 "매일 쓰고 싶은" 앱 경험을 제공한다.

### 2.5 핵심 문제점 (Gap 분석)

| # | 문제 | 설명 |
|---|------|------|
| 1 | **어드민 레이아웃 의존** | 상담사·내담자도 AdminCommonLayout(LNB·GNB)을 사용 → 어드민 느낌, 모바일 최적화 부족 |
| 2 | **디자인 차별화 없음** | 어드민/상담사/내담자 모두 B0KlA 스타일 → 역할별 고유 경험 부재 |
| 3 | **모바일 퍼스트 아님** | 기존 UI가 데스크톱 기준 → 상담사·내담자는 주로 모바일 사용 예상 |
| 4 | **네비게이션 구조** | 어드민 LNB 기반 → 상담사·내담자에게는 바텀 네비게이션·탭 기반이 적합 |
| 5 | **예약 플로우 미약** | 내담자 예약→결제→확인 플로우가 단일 화면으로 통합되지 않음 |
| 6 | **온보딩 없음** | 첫 사용자 가이드·프로필 설정 흐름 부재 |
| 7 | **실시간 피드백** | 알림·상태 변경에 대한 실시간 UX 부족 |
| 8 | **기존 RN 앱 테넌트 미지원** | `/mobile/` 디렉토리에 React Native 앱이 존재하나 tenantId가 전혀 없음. API URL 하드코딩, FCM 토큰에 tenantId 미포함, 세션에 tenant 개념 없음 → 테넌트 레이어 전면 재설계 필요 |
| 9 | **푸시 알림 기본 수준** | FCM 기본 구현만 존재. 예약 리마인더·결제 알림·상담 시작 알림·웰니스 리마인드 등 도메인 특화 푸시 시나리오 부재 |

### 2.6 기존 React Native 앱 현황 (`/mobile/`)

| 항목 | 현황 | 재활용도 |
|------|------|----------|
| **프레임워크** | React Native + TypeScript | 프로젝트 구조 참고 |
| **FCM 푸시** | `NotificationService.js` — Firebase Messaging 완전 구현 (권한·토큰·포그라운드/백그라운드·클릭 라우팅·히스토리·설정) | **높음** — tenantId 추가만 필요 |
| **네이버 SDK** | `initializeNaverSDK()` — App.tsx에서 초기화 | **중간** — 테넌트 연동 추가 |
| **카카오 SDK** | 연동 흔적 존재 | **중간** — SDK 버전 업 + 테넌트 연동 |
| **바텀 탭** | 상담사 7탭, 내담자 5탭 (Bottom Tab Navigator) | **높음** — IA 참고 |
| **세션 관리** | AsyncStorage 기반 토큰 세션 (SessionContext) | **중간** — tenant 로직 추가 |
| **API 엔드포인트** | AUTH, PUSH, SCHEDULE, MESSAGE 등 전체 매핑 (endpoints.js) | **중간** — `/api/v1/` + tenantId 헤더 추가 |
| **알림 컨텍스트** | 읽지 않은 메시지/시스템 알림 카운트 통합 (NotificationContext) | **높음** — 그대로 활용 |

#### 테넌트 미지원 영역 (새로 구현 필요)

| 영역 | 문제 | 해결 방향 |
|------|------|----------|
| **API Base URL** | `m-garden.co.kr` 환경별 하드코딩 | 테넌트별 서브도메인 동적 URL 또는 공용 도메인 + tenantId 헤더 |
| **FCM 토큰 등록** | `registerTokenToServer(userId)` — tenantId 없음 | `registerTokenToServer(userId, tenantId)` |
| **세션** | user 객체에 tenantId 없음 | 로그인 시 tenantId 포함, 모든 API 호출에 자동 전파 |
| **소셜 로그인** | Naver client ID/secret 환경별 하드코딩 | 공용 앱키 + 백엔드 tenant 매핑 방식 |
| **알림 설정** | AsyncStorage 로컬 저장만 | 서버 동기화 (tenantId 포함) |
| **Push API** | `/api/mobile/push-token/register` — tenantId 파라미터 없음 | 백엔드 API 확장 필요 |

## 3. 기획: 상담사·내담자 중심 앱

### 3.1 범위

**포함**:
- 상담사 전용 앱 UI/UX (모바일 퍼스트)
- 내담자 전용 앱 UI/UX (모바일 퍼스트)
- 역할별 전용 레이아웃 (바텀 네비게이션 기반)
- 디자인 시스템 확장 (역할별 테마·컴포넌트)
- 사용자 시나리오 기반 플로우 설계
- React Native 앱 테넌트 아키텍처 재설계 (기존 `/mobile/` 코드 참고, 테넌트 레이어 새로 구축)
- 푸시 알림 강화 (FCM SDK, 도메인별 푸시 시나리오)

**제외**:
- 백엔드 API 대규모 신규 개발 (추가 콘텐츠용 API는 Phase 2B·3에서 순차 개발)
- 어드민 UI 변경
- ERP·컴플라이언스 등 관리 기능

**영향 영역**:
- 프론트엔드: `frontend/src/components/consultant/`, `frontend/src/components/client/`, `frontend/src/components/layout/`, `frontend/src/styles/themes/`
- 역할: CONSULTANT, CLIENT

### 3.2 핵심 기능 목록

#### 공통 기능 (상담사·내담자 공통)

### 3.4 사용자 시나리오

#### 시나리오 1: 상담사 아침 루틴
1. 앱 열기 → 오늘의 대시보드 표시
2. "오늘 3건 상담, 1건 일지 미작성" 확인
3. 미작성 일지 탭 → 빠르게 작성 완료
4. 스케줄 탭 → 첫 상담 10:00 내담자 정보 확인
5. 상담 시작 → 완료 → 일지 작성 유도

#### 시나리오 2: 내담자 예약 플로우
1. 홈 → "예약하기" 버튼
2. 기존 상담사 또는 새 상담사 선택 (프로필·전문분야·평점 확인)
3. 가용 시간 슬롯에서 선택
4. 보유 회기 차감 또는 결제 진행
5. 예약 완료 → 캘린더에 자동 추가·알림 설정

#### 시나리오 3: 상담 후 평가
1. 상담 완료 → 푸시 알림 "오늘 상담은 어떠셨나요?"
2. 홈 또는 알림에서 "평가하기" 카드 노출
3. 별점(1~5) + 태그(공감적·전문적·편안한 등) 선택
4. 한줄평(선택) → 완료

#### 시나리오 4: 상담사 메시지 전송
1. 내담자 목록 → 특정 내담자 선택
2. 프로필 카드에서 "메시지" 버튼
3. 격려·안내 메시지 작성 → 전송
4. 내담자가 읽음 확인 가능

#### 시나리오 5: 내담자 감정 일기 작성
1. 앱 열기 → 홈 또는 웰니스 탭
2. "오늘의 기분" 카드 노출 → 탭
3. 이모지 5단계 중 선택 (😢😟😐🙂😊)
4. 감정 태그 칩 선택 (불안, 피곤, 평온, 기쁨 등 다중 선택)
5. 한줄 메모 (선택) → 저장
6. 주간 감정 추이 차트에 반영
7. 담당 상담사에게 자동 알림 (설정 시)

#### 시나리오 6: 내담자 자가 심리검사
1. 웰니스 → 자가 심리검사 → PHQ-9 선택
2. 9개 문항에 답변 (0~3점, 칩/버튼 형태)
3. 제출 → 총점 + 해석 카드 표시 ("경미한 우울 증상")
4. "상담사에게 결과 공유" 토글 → 담당 상담사에게 자동 전달
5. 4주 후 재검사 리마인드 푸시 발송

#### 시나리오 7: 커뮤니티 상담사 칼럼
1. 상담사 → 더보기 → 커뮤니티
2. "새 글 작성" → 카테고리 선택 (불안, 관계, 자존감 등)
3. 제목 + 본문 작성 → 미리보기 → 게시 요청
4. 관리자 검수 후 게시 확정
5. 내담자들이 피드에서 열람·좋아요·댓글

| **아이콘** | Lucide 아이콘 유지 (일관성), 필요 시 커스텀 아이콘 |
| **로그인 화면** | 카카오(노란색) + 네이버(초록색) 소셜 버튼 중심, ID/PW는 "다른 방법으로 로그인" 접힘 |
| **입력 최소화** | 모바일에서 키보드 노출 최소화. 예약: 카드 탭·날짜 스와이프·시간 칩. 일지: 태그 칩 선택, 메모만 텍스트. 검색: 최근 검색·추천 태그 우선 |

### 3.8 테넌트 아키텍처 (React Native)

#### 테넌트 식별 방식
모바일 앱에서는 웹과 달리 서브도메인으로 테넌트 식별이 불가능하므로, 다음 방식을 조합:

1. **딥링크/유니버설 링크**: `mindgarden://tenant/{tenantCode}` 또는 `https://app.mindgarden.co.kr/t/{tenantCode}`
2. **초기 테넌트 선택 화면**: 첫 실행 시 기관 코드 입력 또는 QR 스캔
3. **테넌트 코드 캐싱**: AsyncStorage에 저장, 이후 자동 적용
4. **소셜 로그인 연동**: OAuth state 파라미터에 tenantId 포함 (기존 백엔드 로직 활용)

#### API 호출 시 테넌트 전파
```
모든 API 요청 → TenantAwareApiClient
  ├── 헤더: X-Tenant-Id: {tenantId} 자동 삽입
  ├── 토큰: Authorization: Bearer {accessToken}
  └── Base URL: 환경별 공용 URL (테넌트 구분은 헤더로)
```

#### FCM 토큰 테넌트 연동
```
로그인 성공
  → FCM 토큰 획득 (messaging().getToken())
  → POST /api/v1/mobile/push-token/register
    body: { userId, tenantId, token, platform, deviceInfo }
  → 서버에서 (tenantId + userId + deviceToken) 매핑 저장
```

#### 색상 일관성 위반 방지 규칙

1. **토큰 외 색상 사용 절대 금지**: `color: #E07A5F` ✗ → `color: var(--mg-client-primary)` ✓
2. **한 화면에 Primary 색상 3곳 이하**: CTA 버튼 1개 + 활성 탭 1개 + 강조 텍스트 1개 = 최대 3곳
3. **그래디언트 사용 제한**: 앱 헤더 배경, 히어로 카드, CTA 버튼에만 허용. 일반 카드·리스트에 그래디언트 금지
4. **동일 의미에 동일 색상**: 에러는 항상 `--mg-color-error`, 성공은 항상 `--mg-color-success`. 예외 없음
5. **투명도 토큰화**: `rgba(0,0,0,0.05)` 직접 사용 금지 → `var(--mg-opacity-subtle)` 등으로 토큰화
6. **새 색상 추가 시**: 반드시 core-designer 승인 + 토큰 등록 + 문서 업데이트. 무단 색상 추가 불가
7. **코드 리뷰 체크**: PR 시 색상 하드코딩 여부를 린터 또는 grep으로 자동 검사 (`#[0-9a-fA-F]{3,8}` 패턴)

## 갭(Gap) 분석 결과 및 선행 작업

### 디자인 토큰 — 신규 생성 필요

스펙에서 정의한 역할별 토큰이 `unified-design-tokens.css`에 없음. Phase 1 전에 추가 필수.

| 토큰 | 현황 | 조치 |
|------|------|------|
| `--mg-client-primary` (#E07A5F) 등 내담자 색상 | 미정의 | 신규 정의 |
| `--mg-consultant-primary` (#3D5246) 등 상담사 색상 | 참조만 있음 | 값 정의 |
| `--mg-spacing-*` | 기존은 `--spacing-*` 패턴 | 네이밍 매핑 추가 |
| `--mg-radius-*` | 기존은 `--radius-*` 패턴 | 네이밍 매핑 추가 |
| `--mg-shadow-*` | 미정의 | 신규 정의 |
| `--mg-font-size-*` | 미정의 | 신규 정의 |

### 라우팅 — ProtectedRoute 누락

상담사·내담자 라우트 대부분에 ProtectedRoute가 없음. 새 AppShell 적용 시 함께 보호 추가.

### 백엔드 API — 2건 주의

- **C3 내담자 관리**: `GET /consultants/{id}/clients`가 빈 리스트 반환 (스텁). 매핑 API로 우회 가능.
- **L3 내 상담 조회**: `/schedules/client/{id}`가 ADMIN only. `GET /schedules?userId={id}&userRole=CLIENT`로 대체.

### 추가 콘텐츠 백엔드 API 영향

| 콘텐츠 | 필요 API | 현황 |
|--------|----------|------|
| 감정 일기 | CRUD `/api/v1/mood-journals` + 통계 API | **신규 필요** |
| 자가 심리검사 | 검사 목록·실시·결과 저장 `/api/v1/self-assessments` | **신규 필요** (기존 PsychAssessment는 관리자용) |
| 오디오 명상 | 콘텐츠 목록·즐겨찾기·수련이력 `/api/v1/meditations` | **신규 필요** (기존 HealingContent 확장 가능) |
| 심리 교육 | 콘텐츠 CRUD·북마크·추천 `/api/v1/psycho-education` | **신규 필요** |
| 커뮤니티 | 게시글 CRUD·좋아요·댓글·신고 `/api/v1/community` | **신규 필요** |

> ⚠️ "백엔드 신규 개발 제외" 범위에서 벗어남. 추가 콘텐츠는 백엔드 API 신규 개발이 필요하므로, **제외 범위를 수정**하거나 **별도 Phase로 분리** 필요. → §3.1 제외 범위를 "대규모 신규 개발"로 수정하여 추가 콘텐츠용 API는 Phase 2B·3에서 순차 개발하는 것으로 반영함.

### 와이어프레임 — 4화면 부족

스케줄 주간뷰, 내담자 목록, 내 상담 이력, 웰니스 화면이 없음. 코더가 IA 기반으로 구현 시 자연스럽게 해결.

### 소셜 연동

- **카카오·네이버 OAuth2**: 기존 구현 활용 (OAuth2Controller, SocialAuthController)
- **카카오 알림톡**: TenantKakaoAlimtalkSettings 엔티티로 테넌트별 설정 관리 중
- OAuth2 client-id는 앱 전체 하나 (현재 단일 테넌트 운영이면 문제 없음)

#### SDK 기반 소셜 로그인 (React Native)
- **카카오 SDK**: `@react-native-seoul/kakao-login` 또는 Kakao JS SDK WebView 연동
- **네이버 SDK**: `@mj-studio/react-native-naver-login` 또는 기존 `socialLogin.js` 확장
- **플로우**: SDK에서 소셜 토큰 획득 → 백엔드 `/api/auth/social-login` 전송 → 백엔드에서 소셜 토큰 검증 + 사용자 매핑 + tenantId 처리 → JWT 발급 → 앱 저장
- **백엔드 변경**: `SocialAuthController`에 tenantId 파라미터 추가 (현재는 state/session/subdomain에서 추출)

| **담당** | `core-coder` (+ 필요 시 Spring API는 동일 배치 또는 별도 백엔드 배치) |
| **목표** | (1) **마음 날씨** — 짧은 일기·(선택) 음성을 AI로 분석해 감정 키워드·요약을 제시하고, 상담사 전달은 **옵트인**으로 연결. (2) **마음 정원** — 앱명 MindGarden에 맞춰 **개인 정원**이 상담 완료·과제(자기돌봄 행동) 수행에 따라 **꽃·나무 등으로 시각적 성장**하여 내적 성장과 참여 연속성을 돕고, **비경쟁·비순위** 원칙으로 중도 이탈 동기를 보조한다. |
| **적용 스킬** | `/core-solution-frontend`, `/core-solution-api`, `/core-solution-multi-tenant`, 개인정보·동의 카피는 기획·법무 검토 |
| **의존** | Phase 3 완료 |

**차별점 (제품 포지션)**:
- **마음 날씨**: 단순 기록을 넘어, 예: *「오늘 유독 ‘불안’ 키워드가 높았어요. 상담사님께 **이 데이터(요약/키워드)**를 전달할까요?」* — 사용자 **명시 동의** 후에만 상담사 화면·알림·세션 노트 등으로 브릿지.
- **마음 정원**: 상담 세션 완료·과제 완료 등 **합의된 이벤트**마다 정원이 자라며, **세션 횟수만 과도하게 보상**하지 않도록 가중치·주간 캡·「미접속 시 망가짐 없음」 등 정책으로 **건강한 게임화**를 유지한다.

**전달 프롬프트**:
> Phase 4 — (A) AI 「마음 날씨」 (B) 「마음 정원」성장 시스템을 구현해 주세요.
>
> **A. 마음 날씨**
> 1. **입력**: 감정 일기·짧은 메모 텍스트; (선택) 음성 메모 업로드 → STT 또는 서버 전사 후 동일 파이프라인
> 2. **분석 결과**: 감정 키워드(다중), 친근한 한 줄 요약, **진단 아님·참고용** 문구 고정
> 3. **트렌드(선택)**: 주간 대비 특정 키워드 상승 시에만 부드러운 알림 카피
> 4. **상담사 브릿지**: 공유 시 **요약·키워드·날짜** 위주; 원문 공유는 별도 토글·동의로 분리 권장
> 5. **거버넌스**: 동의 로그, 삭제·철회, 테넌트 격리(`X-Tenant-Id`), API는 `/api/v1/` 계약 하에 설계
>
> **B. 마음 정원**
> 6. **이벤트 소스**: 예) 상담 완료(매칭·세션 상태 연동), 과제/자기돌봄 완료(웰니스·일기 등과 이벤트 계약 합의)
> 7. **시각화**: 정원 씬(2D/일러스트/Lottie 등은 `core-designer` 스펙), 성장 단계·수집 요소(꽃·나무)는 **비경쟁**·타 사용자 비교 없음
> 8. **보상 설계**: 과도한 이용 유도 방지(주간 성장 상한·휴식 페널티 없음), 상담사에게는 **정원 레벨 노출 없음** 원칙(선택적 요약만 별도 정책 시)
> 9. **동기화**: 서버 권위 상태 + 오프라인 큐(Expo는 `EXPO_NATIVE_APP_PLAN` Phase 3-G 정합)
>
> **UI**: 웹(내담자) + Expo 앱 중 기획이 지정한 우선 플랫폼부터; 디자인은 `core-designer` 게이트  
> **디자인 퀄리티·토큰**: §3.9·§3.10 동일 적용

## 9. 실행 요청문

다음 순서로 서브에이전트를 호출해 주세요:

1. **Phase 0** → `core-designer` (model: `gemini-3.1-pro`) — 디자인 시스템 확장 설계 → **✅ 완료됨**
2. Phase 0 완료 후 → **Phase 0.5** → `core-coder` — `unified-design-tokens.css` 토큰 추가
3. Phase 0.5 완료 후 → **Phase 1** → `core-coder` — AppShell 레이아웃 + 소셜 로그인 + 테넌트 아키텍처 구현
4. Phase 1 완료 후 → **Phase 2A + 2B 동시** → `core-coder` 2개 — 상담사·내담자 화면 병렬 구현
5. Phase 2 완료 후 → **Phase 3** → `core-coder` — P1 기능 + 푸시 알림 강화 + 추가 콘텐츠 구현
6. Phase 3 완료 후 → **Phase 4** → `core-coder` — 마음 날씨 AI + 상담사 옵트인 브릿지 + **마음 정원** 성장(게임화)
