# 매칭-스케줄 통합 원스톱 화면 기술 검토 및 설계서 (Technical Review & Design)

## 1. 문서 개요
본 문서는 디자이너 서브에이전트(core-designer)가 제안한 `MATCHING_SCHEDULE_INTEGRATION_SPEC.md` (스플릿 뷰 기반 매칭-스케줄 통합 화면)에 대하여, 프론트엔드 개발자(core-coder) 관점에서 기술적 타당성을 검토하고 실제 구현을 위한 아키텍처 및 라이브러리 설계안을 정리한 회의록 겸 기술 설계서입니다.

---

## 2. UI/UX 스펙 검토 총평
제안해주신 **'아이디어 A (스플릿 뷰 + 드래그 앤 드롭)'** 방식은 관리자의 업무 효율을 극대화할 수 있는 훌륭한 접근입니다. 
좌측 대기자 목록과 우측 캘린더를 한 화면에 배치하여 인지적 부하를 줄이고, 드래그 앤 드롭으로 즉각적인 액션을 유도하는 설계는 최신 B2B SaaS 어드민 트렌드와 잘 부합합니다. 제시된 마인드가든 디자인 토큰과 컬러 팔레트 역시 개발 시 CSS 상수(`unified-design-tokens.css`)와 `mg-v2-*` 클래스로 그대로 매핑하여 이질감 없이 구현 가능합니다.

---

## 3. 서브에이전트 기술 검토 안건에 대한 답변 및 설계안

### 3.1. 드래그 앤 드롭 라이브러리 선정 및 호환성
- **답변 및 설계안**: 
  - 캘린더 구현을 위해서는 압도적인 안정성과 기능을 자랑하는 **FullCalendar (`@fullcalendar/react`)**를 도입하는 것이 최적입니다.
  - 외부 DOM(좌측 대기자 리스트)에서 캘린더 내부로의 드래그 앤 드롭 연동은 FullCalendar에서 공식 지원하는 `@fullcalendar/interaction` 플러그인의 **`Draggable` (Third-Party Dragging)** 기능을 사용하여 구현합니다.
  - 이 방식을 사용하면 별도의 무거운 D&D 라이브러리(dnd-kit 등)를 억지로 캘린더와 동기화할 필요 없이, 좌측 리스트 아이템에 특정 클래스를 부여하고 `new Draggable()` 인스턴스를 연결하는 것만으로 자연스러운 드롭(Drop) 이벤트와 시간대 스냅(Snap) 효과를 얻을 수 있습니다.

### 3.2. 성능 및 렌더링 최적화
- **답변 및 설계안**:
  - D&D 동작 시 마우스 이동마다 전체 DOM이 리렌더링되는 것을 방지해야 합니다. 
  - 좌측 대기자 목록의 개별 카드는 `React.memo`로 감싸서 prop이 변경되지 않는 한 리렌더링되지 않도록 차단합니다.
  - 만약 대기자 목록이 수백 건 이상으로 길어질 경우, DOM 노드 개수를 제어하기 위해 **`@tanstack/react-virtual`**을 도입하여 화면에 보이는 항목만 렌더링(Virtualization)하는 방식을 적용합니다.
  - 캘린더 영역은 FullCalendar가 내부적으로 VDOM 최적화를 수행하므로 큰 성능 저하가 없으나, 이벤트 데이터를 주입할 때 매 렌더링마다 새로운 배열 참조를 생성하지 않도록 `useMemo`로 감싸서 전달합니다.

### 3.3. 캘린더 뷰의 가로/세로 스크롤 동기화
- **답변 및 설계안**:
  - 상담사가 여러 명일 경우 가로축에 상담사를 배치하고 세로축에 시간을 배치하는 `Resource TimeGrid` 구조가 필요합니다.
  - FullCalendar의 기본 구조는 상단 헤더(날짜/상담사)와 좌측 축(시간)을 `position: sticky` 형태로 자체 지원하므로, 스크롤을 내리거나 우측으로 이동해도 기준 축이 고정되어 깔끔한 UI가 유지됩니다.
  - 별도의 커스텀 스크롤 동기화 스크립트는 지양하고, 브라우저 네이티브 Sticky 동작에 기반한 FullCalendar의 내부 CSS를 마인드가든 디자인 토큰에 맞춰 오버라이딩(Overriding)하는 방식으로 스타일을 맞춥니다.

### 3.4. 상태 관리 구조 (Zustand 적용 및 낙관적 업데이트)
- **답변 및 설계안**:
  - 전역 상태 관리는 **Zustand**를 사용하며, 빠른 피드백을 위해 **낙관적 업데이트(Optimistic Update)**를 적극 도입합니다.
  - **작업 플로우**:
    1. 카드를 캘린더에 드롭하는 순간, 즉시 Zustand 상태에서 해당 내담자를 '대기자 배열'에서 제거하고 '캘린더 이벤트 배열'에 임시(가승인) 상태로 추가하여 UI에 즉각 반영합니다.
    2. 기획안대로 `UnifiedModal`을 띄워 상세 정보를 확인/수정합니다.
    3. 모달에서 [저장] 클릭 시 `StandardizedApi`를 통해 백엔드 API를 호출합니다.
    4. API 성공 시: 임시 ID를 DB의 실제 ID로 교체하고 확정 상태로 변경합니다.
    5. 모달 [취소] 또는 API 실패 시: 롤백 함수를 실행하여 캘린더에서 이벤트를 제거하고 다시 좌측 대기자 목록으로 복구한 뒤, 에러 토스트(Toast) 메시지를 노출합니다.

### 3.5. 모바일/태블릿 반응형 처리 정책
- **답변 및 설계안**:
  - 스플릿 뷰는 좌우 너비 확보가 필수적이므로, 화면 폭이 좁은 환경에서는 UX 훼손이 발생합니다. 다음과 같은 반응형 분기(Breakpoint: 1024px 기준)를 적용합니다.
  - **PC (1024px 초과)**: 기획안대로 **스플릿 뷰 (Split View)** 형태를 유지하며 드래그 앤 드롭을 지원합니다.
  - **태블릿/모바일 (1024px 이하)**: 
    - 화면 전체를 캘린더가 채웁니다.
    - 좌측 대기자 목록은 숨겨지고, 우측 하단 플로팅 버튼(FAB)이나 상단 툴바 버튼을 누르면 **바텀 시트(Bottom Sheet) 또는 슬라이딩 드로어(Drawer)** 형태로 나타납니다.
    - 터치 환경에서는 드래그 앤 드롭이 오작동하기 쉬우므로, **터치-투-셀렉트(Touch-to-Select)** 방식으로 전환합니다. (대기자 목록에서 카드를 '터치'하여 선택 상태로 만든 뒤, 캘린더의 빈 시간대를 '터치'하면 할당 모달이 뜨는 방식)
