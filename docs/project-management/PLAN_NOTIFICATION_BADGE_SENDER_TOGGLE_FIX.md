# 기획: 알림 배지·발신자 표기·목록 토글 점검·수정

> 사용자 리포트 기반 3가지 점검·수정. 탐색 → 디버거 분석·제안 → 코더 구현 → (선택) 문서 정리.

---

## 1. 목표·요구사항 요약

| # | 이슈 | 목표 |
|---|------|------|
| 1 | **알림 배지 미표시** | GNB 알림 버튼(`.mg-v2-notification-trigger`)에 읽지 않은 알림이 있으면 배지(숫자 또는 점) 표시. 기존 배지 컴포넌트/로직 재사용 또는 추가. |
| 2 | **발신자 표기 — 시스템 메시지** | 알림/메시지에서 시스템 발신 시 **"시스템 메시지"**로 표기(현재 "알 수 없음" 등으로 나오는 부분 수정). 메시지 카드·알림 드롭다운 등 발신자 텍스트 노출处 전부 공통 규칙 적용. |
| 3 | **목록 보기 전환 토글 공통 적용** | 사용자 관리 > 매핑 리스트 블록(`.mg-v2-mapping-list-block` 내 pill 토글)을 다른 목록 보기 전환(그리드/리스트)과 **동일한 공통 패턴**으로 통일. pill toggle / 보기 전환 컴포넌트·스타일·접근성·상태 관리 일치. |

---

## 2. 범위

- **포함**: GNB 알림 트리거·배지, 알림/메시지 발신자 표기(메시지 카드·드롭다운·관련 목록), 매핑 리스트 블록 및 다른 목록 보기 전환 토글 사용처.
- **제외**: 알림/메시지의 본문 내용 변경, 새 API 추가(기존 API·컨텍스트 활용 전제).
- **영향**: 프론트엔드 — `dashboard-v2`(NotificationDropdown, NotificationBadge), `admin`(AdminMessages, MappingListBlock), `common`(ViewModeToggle), 알림/메시지 발신자 표기 사용 컴포넌트 전부.

---

## 3. 의존성·순서

1. **Phase 0 (탐색)**: 세 이슈별로 코드 위치·데이터 흐름 파악. **병렬** 실행 가능(탐색 A·B·C 동시).
2. **Phase 1 (디버거)**: 탐색 결과 수신 후, 배지 미표시 원인·발신자 "알 수 없음 vs 시스템 메시지" 분기 누락·토글 비일관 지점 분석 → 수정 제안·체크리스트 작성.
3. **Phase 2 (코더)**: 디버거 제안 반영 — 배지 표시 수정, 발신자 "시스템 메시지" 공통 표기, pill 토글 공통화.
4. **Phase 3 (선택, 문서)**: 알림·메시지 표기 및 토글 공통 규칙 짧게 정리한 문서.

**Phase 0-A 전달문 (알림 배지)**  
```
GNB 알림 버튼은 div.mg-v2-notification-trigger-wrapper > button.mg-v2-notification-trigger 에 있음.
요구: 읽지 않은 알림이 있으면 해당 버튼에 배지(숫자 또는 점)가 표시되어야 하는데, 현재 미표시로 리포트됨.

다음을 탐색해 목록·요약으로 보고해 주세요.
1) 알림 배지가 그려지는 컴포넌트/파일 (예: NotificationBadge, NotificationDropdown 등)와 DOM 상 위치.
2) 알림 개수 또는 unread 상태를 어디서 가져오는지 (API, Context(useNotification), 로컬 state 등).
3) GNB에서 사용하는 알림 트리거/드롭다운이 위 데이터 소스와 어떻게 연결되어 있는지 (같은 Context 사용 여부, props 전달 경로).
4) 배지가 조건부 렌더링되는 경우 그 조건(예: unreadCount > 0)과 데이터가 채워지는 시점(마운트 시, 폴링, 이벤트 등).

산출: 파일 경로 목록, 데이터 흐름 1~2문장 요약, "배지가 안 나올 수 있는 원인 후보" 1~3가지 짧게.
```

**Phase 0-B 전달문 (발신자 표기)**  
```
알림/메시지에서 발신자가 "알 수 없음"으로 나온다는 리포트가 있음. 시스템이 보낸 메시지는 "시스템 메시지"로 표기되어야 함.

다음을 탐색해 목록·요약으로 보고해 주세요.
1) 발신자(sender / senderName / authorName 등) 텍스트를 표시하는 컴포넌트·파일 목록 (메시지 카드 mg-v2-message-card, 알림 드롭다운, 알림/메시지 목록 등).
2) 각 위치에서 발신자 문자열을 어떻게 결정하는지 (예: message.senderName || '알 수 없음', authorName || senderName || '시스템' 등). 시스템 발신 여부를 구분하는 필드(예: messageSource, senderType, isSystem 등)가 있는지.
3) 이미 "시스템" 또는 "시스템 메시지"로 표기하는 곳과 "알 수 없음"으로만 fallback 하는 곳을 구분한 목록.

산출: 발신자 표기 사용처 파일·라인 요약, 시스템 발신 구분 필드 정리, 공통 규칙을 적용해야 할 위치 목록.
```

**Phase 0-C 전달문 (목록 보기 전환 토글)**  
```
사용자 관리 > 매핑 리스트 블록(section.mg-v2-mapping-list-block) 내 pill 토글(div.mg-v2-ad-b0kla__pill-toggle.mg-v2-mapping-list-block__toggle, aria-label="목록 보기 전환")을 다른 목록 보기 전환(그리드/리스트)과 동일한 공통 패턴으로 맞춰야 함.

다음을 탐색해 목록·요약으로 보고해 주세요.
1) 목록 보기 전환(pill toggle, 그리드/리스트/카드 뷰 전환)을 쓰는 컴포넌트 목록과, 각각에서 사용하는 컴포넌트명(예: ViewModeToggle) 또는 직접 div.mg-v2-ad-b0kla__pill-toggle 마크업 사용 여부.
2) 매핑 리스트 블록(MappingListBlock)에서 토글을 어떻게 구현했는지(공통 컴포넌트 사용 여부, 클래스명, aria, 상태 관리).
3) 다른 화면(예: 재무/급여/상담사·직원 관리 등)의 목록 보기 전환과 비교했을 때 차이점(스타일·접근성·상태 네이밍·공통 컴포넌트 사용 여부).

산출: ViewModeToggle 사용처 목록, div.mg-v2-ad-b0kla__pill-toggle 단독 사용처 목록, 매핑 리스트 블록과의 차이 요약 및 공통화 시 수정 대상 후보.
```

### Phase 1 — 원인 분석·수정 제안 (core-debugger)

| ID | 목적 | 담당 | 전달할 태스크 설명(프롬프트) 초안 |
|----|------|------|----------------------------------|
| 1 | 배지·발신자·토글 이슈 분석 및 수정 제안 | **core-debugger** | 아래 "Phase 1 전달문" 참조 |

**Phase 1 전달문**  
```
다음 세 가지 사용자 리포트에 대해, 탐색(explore) Phase 결과를 참고하여 원인 분석과 수정 제안을 해 주세요. 코드 수정은 하지 말고, core-coder가 적용할 수 있는 수정 제안·체크리스트만 작성합니다.

1) 알림 배지 미표시
- 현상: 알림이 있는데 GNB 알림 버튼(.mg-v2-notification-trigger)에 카운트(배지)가 표시되지 않음.
- 탐색 결과에서 확인된 데이터 소스(Context/API/로컬 state)와 배지 렌더 조건을 바탕으로, 배지가 안 나올 수 있는 원인(데이터 미연동, 조건 오류, 초기값 등)을 1~3가지로 정리하고, 각 원인에 대한 수정 방향(어디를 어떻게 바꿀지)을 1~2문장으로 제안.

2) 발신자 표기 — 시스템 메시지
- 현상: 알림/메시지에서 발신자가 "알 수 없음"으로 나옴.
- 요구: 시스템이 보낸 메시지는 "시스템 메시지"로 표기. 메시지 카드(mg-v2-message-card), 알림 드롭다운 등 발신자 텍스트 노출处 전부 공통 규칙 적용.
- 탐색 결과에서 확인된 발신자 표기 사용처와 시스템 발신 구분 필드를 바탕으로, "알 수 없음"이 나오는 분기와 수정할 위치를 나열하고, 공통 규칙(시스템 발신 시 표기 문자열 "시스템 메시지", 적용 위치 목록)을 제안.

3) 목록 보기 전환 토글 공통 적용
- 현상: 사용자 관리 > 매핑 리스트 블록의 pill 토글이 다른 목록 보기 전환과 동일한 공통 패턴이어야 함.
- 탐색 결과에서 확인된 ViewModeToggle 사용처·비사용처·차이점을 바탕으로, 매핑 리스트 블록 및 필요 시 다른 화면에서 공통 컴포넌트/스타일/접근성·상태 관리가 일치하도록 하기 위한 수정 제안(파일·변경 요약)과 체크리스트 작성.

산출: (1) 배지 수정 제안·체크리스트, (2) 발신자 "시스템 메시지" 공통 표기 적용 위치 및 규칙, (3) 토글 공통화 수정 제안·체크리스트. docs/debug/ 아래에 짧은 분석·제안 문서로 저장해도 됨.
```

**Phase 2 전달문**  
```
알림 배지·발신자 표기·목록 토글 공통화 수정을 적용해 주세요. core-debugger의 Phase 1 산출물(수정 제안·체크리스트·공통 규칙)을 반드시 참고하여 구현합니다. 표준: /core-solution-frontend, /core-solution-atomic-design, /core-solution-encapsulation-modularization.

1) 알림 배지 표시
- GNB 알림 버튼(.mg-v2-notification-trigger)에 읽지 않은 알림이 있으면 배지(숫자 또는 점)가 보이도록 수정.
- 기존 NotificationBadge(dashboard-v2/atoms) 또는 공통 알림 배지 로직이 있으면 재사용하고, 데이터 소스(useNotification 또는 알림 목록 기반 unread 개수)와 정확히 연동되게 할 것.

2) 발신자 "시스템 메시지" 공통 표기
- 시스템이 보낸 메시지/알림은 발신자 텍스트를 "시스템 메시지"로 표기 (알 수 없음이 아님).
- 메시지 카드(mg-v2-message-card), 알림 드롭다운, 기타 발신자 표기 사용처에서 공통 규칙 적용: 시스템 발신 판단 필드(예: messageSource, senderType 등)가 있으면 "시스템 메시지"로 표기.
- 가능하면 발신자 표기 문자열을 한 곳(상수 또는 유틸)으로 모아 재사용.

3) 목록 보기 전환 토글 공통화
- 사용자 관리 > 매핑 리스트 블록(section.mg-v2-mapping-list-block)의 pill 토글을 다른 목록 보기 전환과 동일한 공통 패턴으로 통일.
- 공통 컴포넌트 ViewModeToggle(common/ViewModeToggle) 사용, 동일한 스타일·접근성(aria-label 등)·상태 관리 적용. 이미 ViewModeToggle을 쓰는 경우 동작·스타일 일치 여부 확인 및 불일치 시 정리.

완료 시 기획에게 보고: 수정한 파일 목록, 적용한 규칙(발신자 표기·토글 공통화) 한 줄 요약.
```

## 6. 리스크·제약

- 알림/메시지 API·백엔드에서 시스템 발신 여부 필드가 없으면, 프론트에서 타입/소스로 추론해야 할 수 있음(탐색·디버거 결과 반영).
- GNB가 사용하는 레이아웃/페이지가 여러 개일 수 있음(DesktopGnb, dashboard-v2 등). 실제 배지가 붙는 트리거가 어느 레이아웃인지 탐색에서 명확히 할 것.
