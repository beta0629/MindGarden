# 상담일지 모달 — 블록 순서·메모 요약·편집 UI 화면설계서

**대상**: core-coder 구현 · core-component-manager 정합  
**맥락**: 상담사 콘솔 — 일정에서 진입하는 **상담일지 작성/수정** (`ConsultationLogModal`, UnifiedModal fullscreen). 어드민 대시보드 샘플(https://mindgarden.dev.core-solution.co.kr/admin-dashboard-sample)과 동일한 **B0KlA 톤**(웜 뉴트럴 배경, 주조 `#3D5246`, 섹션 블록·좌측 악센트 바)을 적용하되, **전체 페이지가 아니라 모달 내부**에 압축된 콘솔 작업 흐름임을 전제한다.  
**참조**: `docs/design-system/PENCIL_DESIGN_GUIDE.md`, `frontend/src/styles/unified-design-tokens.css` (토큰명만 참고, 파일 직접 수정 금지)

---

## 1. 목표

1. **정보 순서**: 사용자가 요구한 **순번 명시**, **2번째 블록에「상담 내용」**(A4 1장 분량의 긴 본문 영역)을 두어 작성 흐름을 고정한다.  
2. **메모 요약**: 프로필에서 쓰는 **메모 요약**을 스크롤과 무관하게 **항상 인지 가능**하게 하고, **일지 저장 맥락에서 편집 가능**해야 한다.  
3. **구현 일원화**: 색·간격·섹션 블록은 **펜슬 가이드 + `var(--mg-*)` / `mg-v2-*`** 로만 정의한다.

## 2. 정보 구조 (블록 순서 — 상→하)

모달 본문(`.mg-v2-modal-body` 등 기존 스크롤 컨테이너) 안에서 **아래 순서**를 고정한다. (헤더/푸터는 UnifiedModal 고정 영역 유지.)

| 순서 | 블록명 | 역할 |
|------|--------|------|
| **0** | 모달 헤더 메타 | 일정·내담자 식별, **회기(순번)** 표시 (아래 §4) |
| **1** | 내담자 컨텍스트 | 기존 프로필 패널 계열: 식별 정보 + **메모 요약(항상 노출·편집 가능)** + (기존과 동일) 등급에 따른 비노출 규칙 |
| **2** | **상담 내용** | 긴 본문 — **제품 요구상 핵심 블록** (§3) |
| **3** | 구조화 부가 기록 | 엔티티의 나머지 TEXT 필드가 이미 상담 내용에 포함되지 않은 경우만 별 블록으로 유지하거나, §3 권장안에 따라 통합 |
| **4** | 주의·필수·메타 | 주의사항 아코디언, 필수 필드 안내, 우선순위·완료 상태 등 (기존 순서 재배치 시 이 구간은 **상담 내용 아래**로) |
| **5** | 푸터 액션 | 저장·취소 (기존 패턴) |

## 3. 와이어: 「상담 내용」2안 및 권장

### 안 A — 단일 대형 textarea (한 덩어리 본문)

- **구성**: 섹션 블록 1개, 제목「상담 내용」+ 좌측 악센트 바, 본문은 **단일 `textarea`** (min-height: 약 A4 1장 분량 — 예: `min-height: 28rem`~`32rem`, 토큰 기준 `var(--mg-spacing-*)` 조합으로만 명시).
- **장점**: 서술형 일지·인쇄·복사 시 **한 흐름**으로 읽기 좋음.  
- **단점**: 백엔드 `client_condition` / `main_issues` 등 **다수 컬럼**과 1:1 매핑이 어려워 코더·기획이 **분리/병합 규칙**을 추가로 정의해야 함.

### 안 B — 섹션 묶음 (엔티티 필드별 서브블록)

- **구성**: 하나의 부모 섹션「상담 내용」아래 **세로 스택**으로 서브 섹션(각각 좌측 악센트 + 라벨 + textarea). 예: 내담자 상태(`client_condition`), 주호소(`main_issues`), 개입(`intervention_methods`), 내담자 반응(`client_response`) — 실제 라벨은 도메인 용어에 맞게 확정.  
- **장점**: DB·목록·검색과 **구조 정합**, 상담 기록 품질(누락 방지)에 유리.  
- **단점**: 시각적으로 블록 수가 늘어나 스크롤 길이 증가 — **gap**은 `var(--mg-spacing-md)`~`var(--mg-spacing-lg)` 로 통일.

### 권장 (디자인 관점)

**안 B(섹션 묶음)** 를 권장한다. 이유: 엔티티가 이미 다수 TEXT로 쪼개져 있어 **표시·검증·접근성(라벨-컨트롤 대응)** 이 명확하고, 「2번째 블록 = 상담 내용」을 **하나의 시각적 슈퍼블록**(배경 `var(--mg-color-surface-main)`, 테두리 `1px solid var(--mg-color-border-main)`, `border-radius: 16px`, 패딩 `24px`)으로 묶으면 안 A와 동일한 **단일 초점**을 유지할 수 있다.  
**보완**: 최상단 서브필드 하나에 `min-height`를 크게 두어 「대표 본문」 느낌을 주거나, 접이식「한 줄 요약」 입력을 선택적으로 둘 수 있다(기획 옵션).

## 4. 순번(session) UI — 디자인 옵션

전제: 폼에서 **회기 번호는 시스템 부여·비활성**일 수 있음.

| 옵션 | 설명 | 시각 |
|------|------|------|
| **R1 — 읽기 전용 칩** | 헤더 또는 블록1 상단에 **칩/배지**로 `N회기` 표시 | 배경 `var(--mg-color-surface-main)`, 테두리 `var(--mg-color-border-main)`, 텍스트 `var(--mg-color-text-main)`, 높이 ≥44px 터치 권장 |
| **R2 — 정의 목록 행** | 「회기」라벨 + 값만 텍스트 | 라벨 `12px`, `var(--mg-color-text-secondary)` / 값 `var(--mg-color-text-main)` |
| **R3 — 비활성 입력 형태** | 기존처럼 input 비활성 | `background: var(--mg-color-surface-main)`, `color: var(--mg-color-text-secondary)`, `cursor: not-allowed` — **편집 불가**가 가장 분명 |

**권장**: **R1(칩) + R2(라벨 병행)** — 헤더 보조 영역에 칩으로 한눈에 보이게 하고, 본문 첫 블록에는 중복 없이 헤더만 사용하거나, 모바일에서만 블록1 상단에 동일 칩을 **스티키**로 반복할지 결정(열린 결정 참조).

## 5. 메모 요약 — 항상 보이게 (1안 권장 + 대안)

### 권장안: 데스크톱 **2열 고정**, 모바일 **상단 스티키 카드**

- **≥1024px (데스크톱/넓은 태블릿 가로)**: 모달 본문을 `flex` 2열 — **좌 320~360px** 고정(또는 `minmax(280px, 32%)`): 프로필 + **메모 요약** + 편집. **우측 flex:1**: 상담 내용·나머지 폼. 좌열은 세로 스크롤과 무관하게 **열 전체가 뷰포트 높이에 맞춰 고정(`position: sticky; top: 0` + `max-height: 100%`)** 되게 하여, 우측만 길게 스크롤되어도 메모가 사라지지 않게 한다.  
- **768~1023px**: 2열 유지 가능 시 동일. 좁으면 **단일 열 전환**.  
- **≤767px**: 단일 열. **메모 요약**을 **상단 스티키**(`sticky`, 배경 `var(--mg-color-background-main)` 또는 `var(--mg-color-surface-main)`, 하단 `1px solid var(--mg-color-border-main)`, `z-index` 적절히)으로 고정해 스크롤 시에도 항상 최소 2~3줄 + 「펼치기」가 보이게 한다.

**토큰**: 스티키 바 그림자는 과하면 `box-shadow` 대신 **하단 보더만** (`var(--mg-color-border-main)`).

### 대안 1 — 좌측 드로어(모바일)

- 햄버거/「내담자」버튼으로 프로필+메모 슬라이드 — **항상 보임** 요구와 상충 가능 → 권장안보다 **한 단계 낮음**.

### 대안 2 — 하단 고정 바(FAB 스타일)

- 메모 한 줄 + 편집 아이콘 — 가독성은 좋으나 키보드/textarea 포커스와 겹칠 수 있어 **접근성 검증 필수**.

## 6. 일지에서 메모 편집 — 인라인 vs 하단

| 방식 | 설명 | 접근성·시각 |
|------|------|----------------|
| **P1 — 프로필 블록 내 서브섹션** | 「메모 요약」제목 + 좌측 악센트 + `textarea` | 라벨 `for` 연결, 영역에 `aria-labelledby`. 편집 중 테두리 `2px solid var(--mg-color-primary-main)` 또는 `outline` (포커스 링 토큰 통일) |
| **P2 — 하단 고정 편집 영역** | 본문 끝에 별도 카드 | 시각적으로 본문과 분리되나 스크롤 거리 증가, 모바일에서 메모와 멀어질 수 있음 |

**권장**: **P1**. 메모가 **항상 보이는** 정책과 같이 움직이며, **같은 블록 안**에서 읽기→편집 전환이 자연스럽다. 저장 전 미저장 상태는 **보조 텍스트** `var(--mg-color-text-secondary)` 또는 작은 배지로만 표시(과한 경고색 사용 금지).

## 8. 토큰·클래스 (코더 전달용)

**색상 (PENCIL_DESIGN_GUIDE 2.1 정합)**

- 배경: `var(--mg-color-background-main)`  
- 카드/블록: `var(--mg-color-surface-main)`  
- 테두리: `1px solid var(--mg-color-border-main)`  
- 본문: `var(--mg-color-text-main)`  
- 보조: `var(--mg-color-text-secondary)`  
- 주조(악센트 바·포커스·주 버튼): `var(--mg-color-primary-main)`  

**간격·라디우스**

- 섹션 내부 gap: `var(--mg-spacing-md)` ~ `var(--mg-spacing-lg)`  
- 섹션 패딩: 24px 수준(프로젝트에 정의된 `--mg-spacing-*` 중 동일 의미 선택)  
- 블록 코너: `16px` (가이드 섹션 블록)  
- 악센트 바: 폭 `4px`, `border-radius: 2px`, `background: var(--mg-color-primary-main)`  

**클래스**

- 모달·스케줄 B0KlA: 기존 `ScheduleB0KlA.css` 및 `UnifiedModal` 패턴의 **`mg-v2-modal-*`**, **`mg-v2-modal-body`** 유지.  
- 신규 래퍼가 필요하면 네이밍 예: `mg-v2-consultation-log__layout`, `mg-v2-consultation-log__sidebar`, `mg-v2-consultation-log__main`, `mg-v2-consultation-log__memo-sticky` — **실제 클래스명은 기존 컨벤션에 맞춰 한 곳에만 정의**(중복 금지 원칙은 core-component-manager와 합의).

**타이포**

- 섹션 제목: 16px, `font-weight: 600`, `var(--mg-color-text-main)`  
- 라벨/캡션: 12px, `var(--mg-color-text-secondary)`  
- 본문 입력: 14~16px, Noto Sans KR

## 9. 열린 결정 5가지 — 디자인 관점 권고 (한 줄씩)

1. **상담 내용: 단일 textarea vs 다중 섹션** → **다중 섹션을 하나의 시각적 슈퍼블록으로 묶는 방식(안 B)** 이 데이터·접근성·일관성에 유리하다.  
2. **순번: 읽기 전용 표시 vs 비활성 입력 유지** → **헤더 칩(R1) 위주**로 바꾸고 비활성 입력 의존은 줄이는 것이 인지성이 좋다.  
3. **메모 항상 노출: 좌 sticky 2열 vs 상단 sticky vs 하단 바** → **데스크톱 2열 sticky 좌측 + 모바일 상단 sticky** 조합을 권장한다.  
4. **메모 편집: 프로필 인라인 vs 하단 분리** → **프로필 블록 내 전용 서브섹션(P1)** 이 맥락 유지에 가장 적합하다.  
5. **등급(visibilityTier)으로 메모 숨김 vs 편집 가능 정책 충돌 시** → **숨김이 우선**이며, 편집 가능은 노출 등급에서만; 숨김 시에는 해당 영역 자체를 렌더하지 않고 대체 안내 문구만 `var(--mg-color-text-secondary)` 로 짧게 표시하는 것이 혼선을 줄인다.

## 10. core-coder 전달용 구현 체크리스트 (코드 아님)

- [ ] 모달 본문 블록 순서가 §2와 일치하는가(특히 **상담 내용이 2번째 주요 블록**).  
- [ ] 모든 배경·테두리·텍스트가 §8 토큰만 사용하는가(임의 hex 없음).  
- [ ] 상담 내용 슈퍼블록이 시각적으로 한 덩어리인가(배경·테두리·radius·악센트 제목).  
- [ ] `session_number` UI가 비편집 정책과 모순 없이 표시되는가(칩/라벨/비활성 중 택1~2).  
- [ ] 데스크톱에서 메모가 우측 스크롤과 독립적으로 **항상 보이는가**(sticky/2열).  
- [ ] 모바일에서 메모가 **상단 sticky**로 가려지지 않는가(헤더·키보드 대응 포함).  
- [ ] 메모 `textarea`에 라벨·포커스 링·대비가 WCAG에 맞는가(주조 포커스 토큰).  
- [ ] `visibilityTier` 등으로 메모 비노출 시 편집 UI가 노출되지 않는가.  
- [ ] 기존 `UnifiedModal`·`ScheduleB0KlA` 패턴을 깨지 않는가(풀스크롤 본문 구조).  
- [ ] **core-tester**: 회기 표시, 메모 sticky, 상담 블록 순서, 등급별 노출 스냅샷·E2E 시나리오 반영.
