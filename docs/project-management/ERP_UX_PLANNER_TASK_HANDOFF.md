# ERP UX 기획 — 역할별 위임 태스크 (실행 지시문)

**작성일**: 2026-03-14  
**참조**:  
- `docs/project-management/ERP_UX_INVISIBLE_ERP_PLAN.md` — "이게 ERP인 줄 모를 정도로" UX 기획  
- `docs/project-management/ERP_RENEWAL_AUDIT.md` — 리뉴얼 갭 점검  

**목표**: Phase 1~5 적용 순서를 유지한 채, core-designer / core-publisher / core-coder에게 넘길 구체 태스크와 실행 지시문(프롬프트 초안)을 정리한다. 코드·디자인 파일 수정 없이 **문서만** 작성.

## 1. 기획 요약

### 1.1 목표·원칙

- **목표**: 사용자가 ERP를 의식하지 않고, **업무/결과 중심**으로 상담소 운영·수입·지출·정산을 쓰는 경험 제공.
- **원칙**: 업무 중심 언어, 결과 중심 표현, **메뉴·타이틀에서 "ERP" 노출 최소화**, 전문 용어(분개·원장·전기) 완화(라벨 병기 또는 대체).

### 1.2 적용 순서 (Phase 1~5)

| Phase | 범위 | 주요 산출/파일 |
|-------|------|----------------|
| **1** | LNB 메뉴 라벨, 운영 현황(ErpDashboard) 타이틀·카드 라벨 | menuItems.js, ErpDashboard.js |
| **2** | 통합 재무 화면 타이틀·탭 라벨, 섹션 제목 | IntegratedFinanceDashboard.js |
| **3** | 분개/원장 내부 문구(거래 정리, 계정별 내역 탭 내) | JournalEntriesTab, LedgersTab 등 |
| **4** | 토스트·에러 메시지, 모달 제목 | IntegratedFinanceDashboard, 공통 notification |
| **5** | AdminDashboard 위젯·카드, RefundManagement 등 | 해당 컴포넌트 |

### 1.3 유지·제외

- **URL**: `/erp/*`, `/admin/erp/*` 변경 없음.
- **권한 코드**: `ERP_ACCESS`, `INTEGRATED_FINANCE_VIEW` 등 **내부 코드 유지**, 화면 **노출 라벨만** 변경(예: "운영·재무 접근", "운영 현황").

### 1.4 기획서 보완 제안 (빠진 항목 정리)

아래 항목은 기획서(ERP_UX_INVISIBLE_ERP_PLAN.md)에 일부만 있거나 미정리되어 있어, **기획서 보완 시 반영**을 권장한다.

| 구분 | 항목 | 제안 |
|------|------|------|
| **권한 관리 화면** | 권한 설정/역할별 권한 UI에서 ERP 관련 **노출 라벨** | 기획 §2.4: "ERP_ACCESS" → 사용자에게 보이는 라벨만 "운영·재무 접근", "ERP 대시보드" → "운영 현황". **권한 관리 화면 목록·설명 문구**에서 이 라벨을 쓰는 위치를 열거해 기획서 §2.4에 추가할 것. |
| **다른 ERP 연동 문구** | 환불·결제·설정 등 다른 화면의 "ERP 연동", "ERP 설정" 문구 | RefundManagement 부제만 기획에 있음. **전역 검색**으로 "ERP"가 노출되는 라벨·부제·툴팁을 찾아 기획서 "기타 화면" 테이블에 추가할 것. |
| **위젯/카드 제목** | AdminDashboard 외 다른 대시보드·위젯에서 "ERP" 노출 | 기획 §2.4 AdminDashboard 카드만 명시. 위젯 이름·카드 제목에 "ERP"가 있는 다른 컴포넌트가 있으면 동일 원칙으로 라벨 매핑 추가. |
| **에러/안내 문구** | "분개 실패", "전기 오류" 등 서버/API 메시지 | 기획 §3은 토스트 위주. **백엔드/API 응답 메시지**에서 사용자에게 그대로 노출되는 ERP·회계 용어가 있으면 목록화해 "메시지·토스트" 테이블에 포함할 것. |

— **실행**: 위 보완 제안은 기획서(ERP_UX_INVISIBLE_ERP_PLAN.md)를 **수정할 때** 반영하면 된다. 본 위임 문서에서는 designer/coder 태스크에 "권한 관리 화면 노출 라벨", "다른 ERP 노출 문구 검색·적용"을 포함해 두었다.

**【core-designer 실행 지시】**

다음 문서를 읽고 아래 작업을 수행하라.

**참조 문서**  
- `docs/project-management/ERP_UX_INVISIBLE_ERP_PLAN.md`  
- `docs/project-management/ERP_UX_PLANNER_TASK_HANDOFF.md` (본 문서 §2)

**요청 사항**  
1. **라벨 적용 스펙 문서 작성**  
   - ERP_UX_INVISIBLE_ERP_PLAN.md의 §2(메뉴·화면 라벨 매핑)를 **화면·영역 단위**로 재구성한 단일 스펙 문서를 작성하라.  
   - 대상: LNB, 운영 현황(ErpDashboard), 통합 재무(IntegratedFinanceDashboard), 탭 내부(거래 정리/계정별 내역), AdminDashboard 위젯·카드, RefundManagement 부제.  
   - 저장 위치: `docs/design-system/` 또는 `docs/project-management/` (예: `ERP_UX_LABEL_APPLICATION_SPEC.md`).  
   - 코드 작성 없음. 마크다운 스펙만 작성.

2. **권한 관리 화면 노출 라벨 보완**  
   - 권한 설정/역할별 권한 UI에서 사용자에게 보이는 "ERP_ACCESS", "ERP 대시보드" 등 **노출 라벨**을 "운영·재무 접근", "운영 현황" 등으로 바꿀 목록을 정리하라.  
   - 기획서 §2.4 및 ERP_UX_PLANNER_TASK_HANDOFF.md §1.4 보완 제안을 반영하라.  
   - 위 스펙 문서에 "권한 관리 화면" 섹션으로 포함하거나, 기획서 §2.4 보완 초안을 별도로 제안하라.

3. **메시지·모달 문구 정리**  
   - 기획서 §3(메시지·토스트)에 더해, **모달 제목·버튼·안내 문구** 중 "분개/원장/전기/ERP"가 노출되는 부분을 찾아 대체 문구를 제안하라.  
   - 스펙 문서에 "메시지·모달 문구" 섹션 또는 표로 넣어라.

4. **표시 제약 검토**  
   - B0KlA·어드민 대시보드 샘플 기준으로, "운영 현황", "수입·지출 관리" 등 새 라벨이 타이틀·카드에 적용될 때 **줄바꿈·길이·가독성** 제약이 있으면 스펙에 짧게 명시하라.

**산출물**  
- 라벨 적용 스펙 문서 1종(경로 명시).  
- 권한 화면 라벨 목록 및 메시지·모달 문구 표.  
- 완료 후 기획(core-planner)에게 산출물 경로와 요약을 보고하라.

## 3. core-publisher 전달 태스크 및 프롬프트

### 3.1 담당 범위

- 이번 UX 기획은 **대부분 기존 컴포넌트의 텍스트(라벨) 변경**이므로, **새 마크업이 필요한 부분**만 퍼블리셔 담당.
- "데이터 동기화" 접이식 섹션(운영 현황 페이지) 구조·시맨틱이 없을 경우 마크업 설계.
- 라벨이 바뀌는 영역의 **시맨틱·접근성**(제목 레벨, aria 등) 점검 요청.

### 3.2 전달 태스크 목록

| 번호 | 태스크 | 산출물 |
|------|--------|--------|
| P1 | 운영 현황(ErpDashboard) 페이지의 **"데이터 동기화"** 섹션이 접이식(collapsible)으로 설계된 경우, 해당 블록의 **HTML 마크업 스펙**(구조, 시맨틱, BEM) 작성. 이미 구현되어 있으면 "기존 마크업 준수"만 명시 | `docs/design-system/` 또는 기존 마크업 스펙에 섹션 추가 |
| P2 | ERP_UX_INVISIBLE_ERP_PLAN.md §2에서 변경되는 **타이틀·카드 제목·탭 라벨**이 들어갈 영역의 **시맨틱 태그 권장**(h1/h2, role, aria-label 등) 정리. 기존 컴포넌트 구조를 바꾸지 않는 범위에서 제안 | 위 스펙 또는 ERP_UX_PLANNER_TASK_HANDOFF.md 연계 문서에 짧은 "마크업·접근성 가이드" |
| P3 | designer 산출물(라벨 적용 스펙)을 받은 뒤, **새로 문구가 들어가는 블록** 중 마크업 수정이 필요한 곳이 있으면 목록화하고 필요 시 HTML 예시만 제공(React 코드 작성 없음) | 목록 + 선택적 HTML 스니펫 |

### 3.3 실행 지시문(프롬프트 초안) — core-publisher

아래 문단을 **그대로** core-publisher 호출 시 프롬프트에 포함하여 전달한다.

---

**【core-publisher 실행 지시】**

다음 문서를 읽고 아래 작업을 수행하라.

**참조 문서**  
- `docs/project-management/ERP_UX_INVISIBLE_ERP_PLAN.md`  
- `docs/project-management/ERP_UX_PLANNER_TASK_HANDOFF.md` (본 문서 §3)  
- `/core-solution-publisher` 스킬(아토믹·BEM·시맨틱 HTML)

**요청 사항**  
1. **"데이터 동기화" 섹션 마크업**  
   - 운영 현황(ErpDashboard) 페이지의 "ERP 회계 초기화·동기화" → "데이터 동기화" 접이식 섹션이 **이미 구현되어 있는지** 코드베이스에서 확인하라.  
   - 없으면: 해당 블록의 **HTML 마크업 스펙**(구조, 시맨틱, BEM)을 작성하라.  
   - 있으면: "기존 마크업 유지, 라벨만 변경"을 스펙에 명시하라.  
   - 저장: `docs/design-system/` 또는 관련 마크업 스펙 문서.

2. **라벨 영역 시맨틱·접근성**  
   - 기획서 §2에서 변경되는 **페이지 타이틀, 카드 제목, 탭 라벨**이 들어가는 영역에 대해 **시맨틱 태그 권장**(h1/h2, role, aria-label 등)을 짧게 정리하라.  
   - 기존 컴포넌트 DOM 구조를 바꾸지 않는 범위에서만 제안하라.  
   - 코드 작성 없음. 마크다운 가이드만.

3. **designer 산출물 연계**  
   - core-designer가 작성한 **라벨 적용 스펙**을 받은 뒤, 새 문구가 들어가는 블록 중 **마크업 수정이 필요한 곳**이 있으면 목록화하고, 필요 시 **HTML 예시만** 제시하라. React/JSX 작성은 하지 않는다.

**산출물**  
- 데이터 동기화 섹션 마크업 스펙(필요 시).  
- 라벨 영역 시맨틱·접근성 가이드(짧은 문단 또는 표).  
- 완료 후 기획(core-planner)에게 산출물 경로와 요약을 보고하라.

## 4. core-coder 전달 태스크 및 프롬프트

- URL·권한 코드·라우트는 변경하지 않음.

| Phase | 태스크 | 대상 파일/영역 |
|-------|--------|----------------|
| **1** | LNB 메뉴 라벨 변경: "ERP 관리" → "운영·재무", "ERP 대시보드" → "운영 현황" 등. 운영 현황(ErpDashboard) 페이지 타이틀·부제·카드 라벨 변경. "ERP 회계 초기화·동기화" → "데이터 동기화"(접이식 권장) | menuItems.js, ErpDashboard.js |

| **3** | 거래 정리·계정별 내역 탭 **내부** 문구(헤더, 테이블 컬럼명, 빈 상태 문구 등)에서 "분개/원장/전기" 최소화, designer 스펙 반영 | JournalEntriesTab, LedgersTab 등 관련 컴포넌트 |

| **5** | AdminDashboard 위젯·카드: "ERP 대시보드" → "운영 현황", "ERP 보고서" → "운영 리포트". RefundManagement 부제: "ERP 연동 관리" → "환불·결제 연동". 권한 관리 화면에서 ERP 관련 **노출 라벨**만 "운영·재무 접근", "운영 현황" 등으로 변경 | AdminDashboard 관련 컴포넌트, RefundManagement, 권한 설정/역할별 권한 UI |

### 4.3 실행 지시문(프롬프트 초안) — core-coder

**【core-coder 실행 지시】**

다음 문서를 읽고 Phase 1~5 순서대로 **라벨·문구 변경만** 구현하라. URL·권한 코드·라우트는 변경하지 않는다.

**참조 문서**  
- `docs/project-management/ERP_UX_INVISIBLE_ERP_PLAN.md`  
- `docs/project-management/ERP_UX_PLANNER_TASK_HANDOFF.md` (본 문서 §4)  
- designer 산출: 라벨 적용 스펙(경로는 기획 또는 designer 보고 확인)  
- publisher 산출: 데이터 동기화 섹션·시맨틱 가이드(해당 시만 참조)  
- `/core-solution-frontend`, `/core-solution-atomic-design` 스킬

**Phase 1**  
- **menuItems.js**: LNB 메뉴 라벨 변경. "ERP 관리" → "운영·재무", "ERP 대시보드" → "운영 현황". 기획 §2.1 표 전부 반영.  
- **ErpDashboard.js**(및 관련 CSS/상수): 페이지 타이틀 "ERP 대시보드" → "운영 현황", 부제 "통합 자원 관리 및 회계 시스템" → "수입·지출·구매를 한눈에". 카드 라벨: 통합 회계 시스템→수입·지출 한눈에, 분개 관리→거래 정리, 원장 조회→계정별 내역, 정산 관리→정산, 현금흐름표→현금 흐름. "ERP 회계 초기화·동기화" 섹션 제목 → "데이터 동기화"(접이식 구조는 publisher 스펙 있으면 따름).

**Phase 2**  
- **IntegratedFinanceDashboard.js**: 페이지 타이틀 "통합 회계 관리 시스템" → "수입·지출 관리", 부제 유지 또는 "거래·손익·정산을 한곳에서". 탭 라벨: 분개 관리→거래 정리, 원장 조회→계정별 내역, 대차대조표→자산·부채 현황(또는 기획 대로), 손익계산서→손익 현황, 현금흐름표→현금 흐름, 정산 관리→정산, 일/월/년간 리포트 유지. 기획 §2.3 표 준수.

**Phase 3**  
- **JournalEntriesTab, LedgersTab** 등 거래 정리·계정별 내역 탭 **내부** 문구: "분개", "원장", "전기" 등 사용자 노출 문구를 designer 라벨 스펙에 맞게 "거래 정리", "계정별 내역", "반영 완료" 등으로 변경. 테이블 컬럼명, 빈 상태 메시지, 툴팁 포함.

**Phase 4**  
- 토스트·에러 메시지: "분개가 승인되었습니다" → "거래가 반영되었습니다", "분개 생성에 실패했습니다" → "거래 반영에 실패했습니다. 다시 시도해 주세요." 등 기획 §3 및 designer "메시지·모달 문구" 스펙 반영. 모달 제목 중 ERP·분개 용어도 동일 원칙으로 변경.

**Phase 5**  
- **AdminDashboard**(위젯·카드): "ERP 대시보드" → "운영 현황", "ERP 보고서" → "운영 리포트".  
- **RefundManagement**: 부제 "ERP 연동 관리" → "환불·결제 연동".  
- **권한 관리 화면**(역할별 권한 설정 등): 사용자에게 보이는 라벨만 변경. "ERP_ACCESS" → "운영·재무 접근", "ERP 대시보드" → "운영 현황". 내부 권한 코드·API는 그대로 유지.

**제약**  
- `/erp/*`, `/admin/erp/*` URL 및 권한 코드(ERP_ACCESS 등) 값은 수정하지 않는다.  
- 라벨·문구·노출 문자열만 변경한다.

**산출물**  
- Phase 1~5 반영된 프론트엔드 수정.  
- 완료 후 기획(core-planner)에게 변경 파일 목록과 요약을 보고하라.

## 5. 완료 기준 체크리스트

아래는 **전체 UX 기획(Phase 1~5)** 이 끝났을 때 충족할 기준이다. 기획 담당이 최종 검수 시 참고한다.

### 5.1 메뉴·화면

- [ ] LNB에서 "ERP" 문구가 보이지 않음(운영·재무, 운영 현황 등으로 대체).
- [ ] 운영 현황(ErpDashboard) 페이지 타이틀 "운영 현황", 부제 "수입·지출·구매를 한눈에".
- [ ] 운영 현황 카드 라벨: 수입·지출 한눈에, 거래 정리, 계정별 내역, 정산, 현금 흐름, 데이터 동기화(섹션).
- [ ] 통합 재무 첫 화면이 "수입·지출 관리"로 인식됨(타이틀·탭 라벨).
- [ ] 탭: 거래 정리, 계정별 내역, 자산·부채 현황/손익 현황/현금 흐름, 정산 등 §2.3 반영.

### 5.2 문구·메시지

- [ ] "분개", "원장", "전기"가 일반 사용자 노출 문구에서 최소화됨(거래 정리, 계정별 내역, 반영 완료 등).
- [ ] 입금 확인·거래 등록 후 메시지가 업무 결과 중심("반영되었습니다")으로 통일됨.
- [ ] 에러 메시지가 "거래 반영에 실패했습니다" 등으로 통일됨.

### 5.3 기타 화면

- [ ] AdminDashboard 위젯·카드: "운영 현황", "운영 리포트".
- [ ] RefundManagement 부제: "환불·결제 연동".
- [ ] 권한 관리 화면에서 ERP 관련 노출 라벨만 "운영·재무 접근", "운영 현황" 등으로 변경됨.

### 5.4 유지 확인

- [ ] URL 경로 `/erp/*`, `/admin/erp/*` 변경 없음.
- [ ] 권한 코드 `ERP_ACCESS`, `INTEGRATED_FINANCE_VIEW` 등 값·API 변경 없음(노출 라벨만 변경).
