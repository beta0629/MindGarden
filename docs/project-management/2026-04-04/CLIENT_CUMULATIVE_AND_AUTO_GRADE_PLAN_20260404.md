# 내담자 누적 가시성 + 회원등급 자동 승급 통합 계획

## 배경·목표

- **배경**: 관리자 ClientModal 등에서 내담자의 누적 이용·세션 지표와 회원등급이 함께 보여야 운영·상담 의사결정이 가능하다. 현재 데이터·코드베이스에는 등급 정의와 집계 경로가 여러 갈래로 존재하고, 자동 승급 로직은 명확히 정리되지 않은 상태다.
- **목표**: UI 단일 소스(with-stats)를 1차 기준으로 누적 KPI를 일관되게 노출하고, 회원등급 자동 승급은 별도 PR로 설계·구현하여 리스크를 분리한다. 동시에 등급·집계 관련 공통 코드와 완료 세션 집계 정합을 후속으로 정리한다.

## 핵심 발견

- **`users.grade` 정본**: 사용자 등급의 DB·도메인 기준은 `users.grade`에 두는 것이 타당하나, 코드·시드·상수 명명이 한 줄로 정리되지 않음.
- **자동 승급 없음**: 현 시점에서 등급을 규칙에 따라 자동으로 올리는 일관된 백엔드 플로우는 부재하거나 분산되어 있음 — E1에서 별도 설계 필요.
- **`UserGrade` vs `UserGrades` vs `USER_GRADE` 시드 혼재**: 엔티티·복수형·시드/상수 명이 혼용되어 유지보수·검색 시 혼선 유발.
- **`ClientStatsServiceImpl` total/completed 동일 카운트 이슈**: 집계 로직에서 total과 completed가 동일하게 잡히는 등 정합 이슈가 보고됨 — 완료 세션 정의와 쿼리·서비스 로직 재검증 필요.

## 권고

- **1차 UI**: **with-stats 단일 소스**로 KPI·누적 표시를 통일한다. 프론트는 가능한 한 해당 API 응답 필드에만 의존하고, 임의 하드코딩·중복 집계를 피한다.
- **자동 승급**: **별 PR**로 분리한다. UI 노출과 승급 규칙·배치/이벤트를 한 PR에 묶으면 롤백·검증 비용이 커진다.

### 미결 질문 목록 (SSOT·운영)

- **SSOT**: 등급 메타(라벨·순서·임계값)의 단일 저장 위치 — DB만 vs 코드 상수 vs 설정 테이블.
- **트리거**: **배치(스케줄)** vs **이벤트(세션 완료 시 즉시)** — 지연 허용 범위·부하·일관성.
- **강등**: 자동 강등 여부, 수동만 허용 여부, 유예 기간.
- **감사**: 등급 변경 이력(누가·언제·이유·이전값) 보관 범위.
- **수동 고정**: 관리자가 등급을 고정했을 때 자동 규칙과의 우선순위·해제 조건.
