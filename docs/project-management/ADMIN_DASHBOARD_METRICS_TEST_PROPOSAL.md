**목적**: 검증 포인트·시나리오·E2E/스냅샷 관점 제안 (테스트 코드 작성 없음)

## 1. 데이터 상태별 UI 검증 포인트

### 1.1 데이터 없음(Empty)

| 영역 | 검증 포인트 |
|------|-------------|
| **KPI 카드 3개** | `stats` 초기값(0)일 때 총 사용자·예약된 상담·완료율이 "0" 또는 "0%" 등으로 표시되는지. 배지(성장률·완료율)는 조건부 렌더이므로 미표시 시 레이아웃 깨짐 없는지. |
| **CoreFlowPipeline** | `stats`가 모두 0일 때 5개 단계 카드가 "0건", "매칭됨/대기중/부여됨/의견수렴중", "배치/일지작성" 등으로 표시되는지. 빈 배열/undefined 전달 시 기본 5단계가 그려지는지. |
| **상담 현황 추이 차트** | `stats.consultationStats?.monthlyData`가 없거나 길이 0일 때 "차트 영역" 플레이스홀더 또는 빈 메시지가 보이는지. |
| **차트 데이터 전부 0** | `monthlyData`는 있으나 `completedCount`가 모두 0일 때 "기간 내 완료된 상담이 없습니다." 문구가 노출되는지. |
| **우수 상담사 평점** | `topConsultantsData`가 빈 배열일 때 "평가 데이터가 없습니다." 문구가 노출되는지. |
| **ManualMatchingQueue / DepositPendingList** | `items`가 빈 배열일 때 빈 상태 메시지 또는 빈 테이블이 기대대로 표시되는지. |

### 1.2 로딩 중(Loading)

| 영역 | 검증 포인트 |
|------|-------------|
| **전체 지표 영역** | `loadStats` 호출 중(`loading === true`)일 때, KPI·파이프라인·차트 영역에 스켈레톤/스피너가 노출되는지, 또는 기존 숫자가 유지되는지(표준·디자인에 따라). |
| **CoreFlowPipeline** | stats를 비동기로 채우는 경우, 로딩 중에 "0건" 등 초기값이 잠깐 보였다가 실제 값으로 교체되는지, 깜빡임/레이아웃 시프트가 없는지. |
| **ManualMatchingQueue** | `loading={true}`일 때 대기열 영역에 로딩 UI가 표시되는지. |
| **시스템 상태/시스템 도구** | 해당 섹션의 `loading` prop 전달 시 버튼 비활성화·로딩 표시가 기대대로 동작하는지. |

### 1.3 에러 시(Error)

| 영역 | 검증 포인트 |
|------|-------------|
| **통계 API 실패** | `loadStats` 중 일부/전체 API 실패 시 토스트/인라인 에러 메시지가 노출되는지. KPI·파이프라인은 이전 값 유지 vs 0/에러 플레이스홀더 중 어떤 정책인지. |
| **개별 지표 API 실패** | consultants/clients/mappings/rating/vacation/consultation 등 일부만 실패했을 때 해당 영역만 에러 처리되는지, 전체가 깨지지 않는지. |
| **입금 대기/미배정 목록 실패** | `notificationManager.error` 호출 여부 및 사용자에게 에러 메시지가 전달되는지. |
| **차트/상담사 데이터 실패** | consultation 또는 rating API 실패 시 차트·우수 상담사 영역에 에러 메시지 또는 빈 상태가 표시되는지. |

## 2. 반응형 검증 포인트

### 2.1 브레이크포인트별

| 구간 | 검증 포인트 |
|------|-------------|
| **모바일 (예: &lt; 768px)** | KPI 카드 3개가 세로 쌓임 또는 가로 스크롤 시 가독성·터치 스크롤. CoreFlowPipeline `overflow-x: auto`로 가로 스크롤 가능한지, 스크롤바/스크롤 영역이 가려지지 않는지. 차트+상담사 카드가 세로 배치되는지. 헤더 검색·아이콘 그룹이 줄바꿈/오버플로우 없이 보이는지. |
| **태블릿 (예: 768px ~ 1279px)** | KPI 카드 배치(2+1 또는 3열). 파이프라인 단계가 일부만 보이고 가로 스크롤로 나머지 확인 가능한지. 차트·상담사 카드가 2열 또는 1열로 전환되는지. |
| **데스크톱 (예: ≥ 1280px)** | `CoreFlowPipeline.css`의 `justify-content: center` 적용 시 5단계가 중앙 정렬·균형 배치되는지. KPI 3열, growth-row 2열 레이아웃 유지. 컨테이너 최대 너비·패딩으로 오버플로우 없음. |

### 2.2 스크롤·오버플로우

| 항목 | 검증 포인트 |
|------|-------------|
| **파이프라인** | `core-flow-pipeline__steps`의 `overflow-x: auto`, `-webkit-overflow-scrolling: touch` 동작. 좁은 뷰포트에서 연결선(connector)이 잘리지 않고 카드와 함께 스크롤되는지. |
| **페이지 전체** | 지표 영역이 길 때 세로 스크롤로 전체 접근 가능한지. 고정 헤더/사이드바와 겹치지 않는지. |
| **카드 내부** | 차트 바·상담사 리스트가 카드 안에서만 스크롤(필요 시)되고 카드 경계를 넘지 않는지. |

## 3. 접근성(A11y) 검증 포인트

### 3.1 ARIA·시맨틱

| 항목 | 검증 포인트 |
|------|-------------|
| **CoreFlowPipeline** | `<section aria-label="5단계 핵심 파이프라인">` 존재. 스크린 리더가 섹션을 "5단계 핵심 파이프라인"으로 인식하는지. |
| **연결선** | `aria-hidden`로 장식용 연결선이 읽히지 않는지. |
| **PipelineStepCard** | 아이콘에 `aria-hidden`. 카드가 `<article>`, 제목이 `<h3>`인지. 단계별 제목·배지가 문서 구조에 반영되는지. |
| **PipelineStepBadge** | `aria-label`로 "배지 값 + 라벨" 의미 전달 여부(예: "3건 매칭됨"). |
| **KPI 카드** | 각 카드가 제목(총 사용자/예약된 상담/완료율)과 값을 스크린 리더가 순서대로 읽을 수 있는지. 필요 시 `aria-label` 또는 `role="group"` + `aria-labelledby` 검토. |
| **차트** | 차트 영역에 `role="img"` 또는 `aria-label`로 요약(예: "최근 6개월 상담 완료 추이") 제공 여부. |
| **버튼/토글** | 통계·캘린더·알림·테마 버튼에 `aria-label` 존재(이미 "통계", "캘린더" 등 적용된 부분 확인). |

### 3.2 키보드·포커스

| 항목 | 검증 포인트 |
|------|-------------|
| **포커스 순서** | 헤더 → KPI 카드들 → 파이프라인(포커스 가능한 요소 있을 경우) → 차트 토글 버튼 → 상담사 목록 → 하단 대기열 순으로 논리적 순서인지. |
| **포커스 가시성** | `:focus-visible` 등으로 포커스 링이 명확한지(예: `mg-v2-ad-b0kla__admin-card:focus-visible`). |
| **인터랙션** | 월간/주간 토글, 버튼들이 키보드(Tab/Enter/Space)로 활성화 가능한지. |
| **스크롤 영역** | 파이프라인 가로 스크롤이 키보드(예: 방향키)로 제어 가능한지, 포커스 트랩이 생기지 않는지. |

### 3.3 스크린 리더

| 항목 | 검증 포인트 |
|------|-------------|
| **숫자 읽기** | "1,234"처럼 포맷된 숫자가 "1234" 또는 "1천2백34" 등 자연스럽게 읽히는지. 필요 시 `aria-label`로 읽기 형태 보정 검토. |
| **동적 갱신** | stats 갱신 시 라이브 리전(`aria-live`) 사용 여부는 정책에 따름. 과도한 알림이 없는지. |
| **에러/빈 상태** | 에러 메시지·"데이터가 없습니다" 문구가 스크린 리더로 노출되는지. |

## 4. 스냅샷·E2E 시나리오 제안

### 4.1 스냅샷 테스트(선택 도구: Jest + React Testing Library 또는 Playwright 스냅샷)

| 시나리오 | 검증 목적 |
|----------|-----------|
| **지표 영역 - 데이터 없음** | stats·todayStats 초기값(0)으로 KPI 3개 + CoreFlowPipeline 5단계 + 차트 빈 상태 + 상담사 빈 메시지가 기대한 DOM/스냅샷과 일치하는지. |
| **지표 영역 - 데이터 있음** | totalMappings/activeMappings 등 숫자와 차트·상담사 리스트가 채워진 상태의 스냅샷으로 회귀 방지. |
| **CoreFlowPipeline 단독** | steps 또는 stats를 props로 넘긴 경우 5개 단계 카드와 연결선이 고정된 스냅샷과 일치하는지. |

### 4.2 E2E 시나리오 (Playwright, `tests/e2e/tests/**/*.spec.ts`)

| 시나리오 | 검증 목적 |
|----------|-----------|
| **관리자 로그인 후 대시보드 로드** | 관리자 계정으로 로그인 → 대시보드 URL 도달 → "대시보드 개요" 또는 동일한 헤딩 노출. |
| **대시보드 로드 후 지표 카드 5개(파이프라인) 렌더** | 로드 완료 후 `.core-flow-pipeline` 내부에 단계 카드가 5개 존재하는지(`.pipeline-step-card` 5개). |
| **대시보드 로드 후 KPI 카드 3개 렌더** | `.mg-v2-ad-b0kla__kpi-card` 3개 노출, "총 사용자", "예약된 상담", "완료율" 라벨(또는 해당 텍스트) 포함. |
| **차트 영역 표시** | "상담 현황 추이" 제목과 월간/주간 토글이 보이고, 데이터 유무에 따라 차트 바 또는 "기간 내 완료된 상담이 없습니다" / "차트 영역" 중 하나 노출. |
| **우수 상담사 영역 표시** | "우수 상담사 평점" 제목과 리스트 또는 "평가 데이터가 없습니다." 노출. |
| **반응형: 모바일 뷰포트에서 파이프라인 가로 스크롤** | 뷰포트를 375px 등으로 설정한 뒤 파이프라인 영역이 가로 스크롤 가능한지, 스크롤 후 5번째 단계까지 도달 가능한지. |
| **접근성: 파이프라인 섹션 aria-label** | `section[aria-label="5단계 핵심 파이프라인"]` 존재 여부. |

위 시나리오는 `docs/standards/TESTING_STANDARD.md`의 E2E 비중(10%), Given-When-Then, 한글 describe/test 사용 원칙과 `core-solution-testing` 스킬을 따른다. 테스트 코드 구현은 별도 작업으로 진행한다.

## 6. 옵션 C 확장 시나리오 (AdminMetricsVisualization 구조 도입 시)

옵션 C 구조(상단 KPI 그리드 + 하단 파이프라인)가 도입될 경우 아래 시나리오를 검증한다.

| 시나리오 | 검증 목적 |
|----------|-----------|
| **상단 KPI 타일 수** | 상단 그리드에 기대하는 KPI 타일 개수(예: 3개 또는 4개)가 렌더되는지. |
| **하단 5단계 동시 렌더** | 하단 파이프라인에 CoreFlowPipeline과 동일하게 5개 단계 카드가 한 번에 렌더되는지. |
| **반응형: 그리드 열 수 전환** | 모바일에서 KPI 1열, 태블릿 2열, 데스크톱 3열(또는 디자인 스펙에 따른 열 수)로 전환되는지. |
| **로딩 시 상단/하단 일괄 스켈레톤** | `loading=true`일 때 상단 KPI 그리드와 하단 파이프라인 영역 모두 스켈레톤/로딩 문구가 표시되는지. |
| **에러 시 영역별 처리** | metrics API 실패 시 상단·하단 중 해당 영역만 에러 UI가 나오는지, 전체가 깨지지 않는지. |
| **접근성: 상단 그리드 region/label** | 상단 KPI 그리드에 `role="region"` 및 `aria-label`(예: "핵심 지표") 존재 여부. |

## 7. AdminMetricsVisualization 도입 시 추가할 테스트

`AdminMetricsVisualization` 컴포넌트가 새로 도입되면 아래 테스트를 추가한다. (현재 해당 컴포넌트는 미도입 상태이므로 체크리스트만 정리.)

### 7.1 도입 시 추가할 테스트 체크리스트

- [ ] **metrics prop 전달 시**: 상단 KPI 타일 + 하단 파이프라인이 metrics 데이터로 렌더되는지 단위 테스트.
- [ ] **loading 전달 시**: 스켈레톤 또는 "로딩 중" 문구가 노출되는지 단위 테스트.
- [ ] **error 전달 시**: 인라인 에러 메시지 또는 에러 UI가 노출되는지 단위 테스트.
- [ ] **접근성**: 상단 그리드·하단 파이프라인 섹션의 `aria-label`/`role`, 배지·숫자 의미 전달 검증 (jest-axe 또는 수동 assertion).
- [ ] **통합**: AdminDashboard(또는 AdminDashboardV2)에서 AdminMetricsVisualization에 stats/loading/error를 전달하는 경로가 기대대로 동작하는지 통합 테스트(선택).

### 7.2 jest-axe 활용 (선택)

접근성 자동 검사가 필요하면 `jest-axe`를 설치 후 `AdminMetricsVisualization`·`CoreFlowPipeline` 렌더 결과에 `axe()`를 실행해 위반 사항이 없는지 검증할 수 있다.
