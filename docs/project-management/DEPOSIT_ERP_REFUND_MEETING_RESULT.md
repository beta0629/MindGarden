# 예약금·결재확인·ERP 등록·환불 회의 결과

**작성일**: 2025-03-16  
**담당**: core-planner (서브에이전트 회의 오케스트레이션)  
**목적**: 예약대기(결재확인) → 상담완료/전액수령 → ERP 등록 시점·방법, 미참석/취소 시 예약금 환불 절차 정리 및 문서화. **본 회의에서는 코드 수정 없음.**

## 2. 비즈니스 플로우 정리

### 2.1 흐름 개요

```
[예약금 50% 수령] → [결재확인 = 예약대기] → 예약 가능
                          │
         ┌────────────────┴────────────────┐
         ▼                                   ▼
  [경로 A: 상담 진행]                  [경로 B: 미참석/취소]
         │                                   │
  [상담완료] → [전액 수령] → [ERP 등록]    [예약금 환불] → [환불완료]
```

- **예약금**: 상담 예약 시 **회기당 50%**를 예약금으로 수령한다는 비즈니스 규칙 (현재 코드/DB에는 미반영).
- **결재확인(예약대기)**: "결재확인" 상태를 **예약대기**로 두고, 이 상태에서 **예약(스케줄 배정) 가능**. 코드 상 `PAYMENT_CONFIRMED`가 해당 역할 수행 중.
- **경로 A**: 상담 완료 후 잔금 수령 → 전액 수령 시점 → **ERP 등록** (시점·방법은 §3에서 제안).
- **경로 B**: 미참석 또는 취소 시 **예약금 환불** → 상태·금액·ERP 반영 방법은 §4에서 제안.

### 2.2 현재 코드 기준 상태 매핑 (참고)

| 비즈니스 용어 | 코드 상태 (MappingStatus 등) | 비고 |
|---------------|------------------------------|------|
| 예약대기 | PENDING_PAYMENT | 결제 전 |
| 결재확인(예약 가능) | PAYMENT_CONFIRMED | 결제 확인됨, 미수금 가능 |
| 입금 확인 후 승인 대기 | DEPOSIT_PENDING | confirm-deposit 후 |
| 활성(상담 진행) | ACTIVE | approve 후 |
| 종료/환불 | TERMINATED 등 | terminate / partial-refund |

- "전액 수령" 전용 상태/API는 **현재 없음**. 입금 확인(confirm-deposit) 또는 결제 확인 4인자(confirm-payment) 시점에 한 번에 INCOME 등록되는 구조.

---

## 3. ERP 등록 시점 제안

### 3.1 현행

- **입금 확인(confirm-deposit)**: INCOME 거래 생성 + `updateMappingInfo` 프로시저 호출.
- **결제 확인 4인자(confirm-payment, 금액 포함)**: INCOME 거래 생성. **updateMappingInfo 미호출** (갭 문서화됨).
- **승인(approve)**: ERP 등록/프로시저 없음. 입금 확인에서만 ERP 반영한다는 설계와 일치.

### 3.2 제안 (옵션)

| 옵션 | ERP 등록 트리거 시점 | 장점 | 비고 |
|------|----------------------|------|------|
| **A** | **전액 수령 시점**에만 ERP 등록 | 예약금·잔금 구분 시 정산 시점이 명확함 | 2단계 수령 모델 도입 시 "전액 수령" 상태/API 필요 |
| **B** | **상담 완료 시점**에 ERP 등록 | 상담 수행 후 매출 인식과 맞출 수 있음 | 상담 완료 이벤트와 ERP 생성 연동 필요 |
| **C** | **둘 다 조건**(전액 수령 **및** 상담 완료 후) ERP 등록 | 수익 인식 기준이 가장 보수적 | 두 조건 만족 시점에 트리거 설계 필요 |
| **D** | **현행 유지**(입금 확인/결제 확인 시점에 INCOME 1회) | 변경 최소. 2단계 수령 없이 한 번에 입금하는 플로우에 적합 | 예약금 50% 규칙·전액 시점 구분 없음 |

**권장**: 비즈니스에서 "예약금 → 잔금" 2단계 수령을 공식 플로우로 쓸 경우 **A** 또는 **C** 검토. 당분간 1단계 입금만 쓸 경우 **D** 유지 후, confirm-payment 4인자 경로에 **updateMappingInfo** 호출 추가만 권장(기존 갭 해소).

## 4. 환불 처리 제안 (미참석/취소 시 예약금 환불)

### 4.1 현행

- **미참석/취소 전용 플로우 없음**. `ConsultationStatus.NO_SHOW` 등 개념은 있으나 "미참석 시 예약금 환불" 자동/전용 API 없음.
- **현재**: 관리자가 **강제 종료(terminate)** 또는 **부분 환불(partial-refund)**로 수동 처리.  
  → `terminateMapping` / `partialRefundMapping` → `sendRefundToErp` → EXPENSE 거래 생성.  
  → 외부 ERP 전송(`sendToErpSystem`)은 **모의 구현** 상태.

### 4.2 제안

1. **상태·절차**
   - 미참석/취소 시 **"예약금 환불"** 전용 액션(버튼/API)을 두고, **환불 금액 = 예약금(회기당 50% 또는 정책값)**으로 제한할 수 있도록 옵션 검토.
   - 상태: **환불완료**에 대응하는 값(기존 TERMINATED 또는 전용 서브상태) 명확화. 디자이너 제안의 "환불완료" 라벨과 매핑.

2. **금액 기록**
   - 환불 금액·사유(미참석/취소)·처리 시점을 **DB·이력**에 남기기. (기존 refund-history 등 확장 여지 검토.)

3. **ERP 반영**
   - **내부 재무(EXPENSE)**: 현재처럼 `createConsultationRefundTransaction` / `createPartialConsultationRefundTransaction` 유지.
   - **외부 ERP**: 실제 연동이 필요하면 `sendToErpSystem` 실구현 추가. (회의 범위 외, 추후 작업.)

4. **UX**
   - 관리자/상담사 화면에서 "결재확인" 상태일 때 ** [예약금 환불] ** 버튼 노출 → 클릭 시 예약금 환불 플로우 진입. (core-designer 제안 반영.)

## 6. 추후 core-coder 작업 체크리스트 (참고용)

구현 시점에 아래를 참고. **본 회의에서 구현하지 않음.**

- [ ] **용어·라벨 통일**: "결재확인" vs "입금확인" vs "결제 확인" — UI/문서에서 한 가지로 통일 (또는 역할별 정의 정리).
- [ ] **confirm-payment 4인자 경로**: `updateMappingInfo` 프로시저 호출 추가 여부 결정 후 반영. (참고: `docs/project-management/ERP_PROCEDURE_CALL_MISSING_TASK.md`.)
- [ ] **예약금 50% 규칙**: 비즈니스 채택 시 예약금/잔금 구분 필드·상태·정책값 도입 검토.
- [ ] **전액 수령 시점**: 2단계 수령 플로우 채택 시 "전액 수령" 상태/이벤트 및 ERP 등록 트리거 설계·구현.
- [ ] **미참석/취소 예약금 환불**: 전용 API 또는 기존 partial-refund에 "예약금만 환불" 옵션 추가, 상태·이력·안내 문구 반영.
- [ ] **환불 시 외부 ERP**: `sendToErpSystem` 실구현 필요 시 연동 스펙 확정 후 구현.
- [ ] **상태 전환 UX**: `docs/design-system/RESERVATION_STATE_TRANSITION_UX_PROPOSAL.md` 확정안에 맞춰 버튼·뱃지·문구 반영.
