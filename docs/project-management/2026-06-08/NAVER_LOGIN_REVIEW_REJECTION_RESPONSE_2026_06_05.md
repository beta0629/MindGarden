## 0. 결정 분기

휴대전화번호(`mobile`) 권한을 **(C) 미사용 + 사용자 입력 + SMS OTP 본인 검증**으로 결정합니다 (2026-06-09 정책 변경).

> 본 결정은 "OAuth 휴대폰 매칭 SSOT 일원화 (사용자 입력 + SMS OTP)" 정책에 따른 것이며, 신규 핸드오프 문서 [`../2026-06-09/OAUTH_PHONE_SSOT_OTP_UNIFICATION_HANDOFF.md`](../2026-06-09/OAUTH_PHONE_SSOT_OTP_UNIFICATION_HANDOFF.md)에 전체 정책·범위·절차가 정리되어 있습니다.

| 근거 | 내용 |
|---|---|
| SSOT 일원화 | 모든 OAuth (Apple/Google/Kakao/Naver) → 사용자 휴대폰 입력 + SMS OTP 본인 검증으로 통일. Apple SIWA P1 OTP 인프라를 provider-agnostic 으로 일반화하여 재사용 |
| 검수 친화 | Naver 콘솔 "사용 API"에서 휴대전화번호 체크 해제 → mobile scope 권한 비요청 상태로 재검수 제출, 재검수 부담 최소화 |
| 운영 안정성 | OAuth 응답 휴대폰 의존 시 입력 오타·허위 입력 등으로 인한 계정 탈취·분리 위험 존재 → OTP 본인 검증으로 본인 소유 회선 검증 후 매칭 |
| 어드민 사전 등록 내담자 자동 매칭 | 사용자 OTP 검증 완료 후 동일 휴대폰 매칭 로직 적용 (휴대폰을 SSOT 로 보존). OAuth 응답 mobile 필드를 사용하지 않을 뿐 매칭 자체는 유지 |

### 대안 비교

| 대안 | 단점 |
|---|---|
| **(A) `mobile` scope 유지** | 검수원이 활용처 캡처를 추가 요구할 수 있고, OAuth 응답 휴대폰의 변조 위험·미인증 회선 매칭 위험 잔존 |
| **(B) `mobile` scope 단순 해제 (OTP 무)** | 어드민 사전 등록 내담자 자동 매칭 신뢰성 하락 (사용자 입력 휴대폰의 본인 확인 없음) |
| **(C) `mobile` 미사용 + OTP 본인 검증 (채택)** | 인프라 일반화 작업 필요 (Apple SIWA P1 자산 재사용으로 비용 최소). 본인 검증 1단계 추가로 사용자 마찰 소폭 증가하나 보안·매칭 정확도 우위 |
