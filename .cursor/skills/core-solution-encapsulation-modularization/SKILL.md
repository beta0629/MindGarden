# 캡슐화·모듈화 원칙 (Encapsulation & Modularization)

Core Solution(MindGarden)에서 **기획·디자이너·퍼블리셔·코더** 등 모든 코어 에이전트는 작업 시 **캡슐화**와 **모듈화**를 적용하여, 경량화·단순화·단순 반복 작업 제거를 목표로 합니다.

## 목표

- **경량화**: 불필요한 중복·과도한 추상화를 줄이고, 필요한 단위만 명확히 둔다.
- **단순화**: 한 단위는 한 책임만 가지도록 하고, 의존 관계를 단순하게 유지한다.
- **반복 제거**: 동일·유사한 패턴은 공통 모듈·컴포넌트·스펙으로 한 번만 정의하고 재사용한다.

## 원칙 (모든 코어 에이전트 공통)

### 1. 캡슐화

- **경계 명확화**: 기능·UI·API·데이터 단위별로 **경계**를 두고, 외부에는 필요한 인터페이스(Props, API, 스펙)만 노출한다.
- **내부 구현 숨김**: 구현 상세는 해당 단위 내부에 두고, 다른 단위가 내부 구현에 직접 의존하지 않도록 한다.
- **스펙·문서**: 기획·디자이너·퍼블리셔는 “어떤 단위가 어떤 책임을 가지는지”를 스펙에 명시하고, 코더는 그 단위 단위로 모듈/컴포넌트를 나눈다.

### 2. 모듈화

- **단위 분리**: 화면·기능·API·스타일을 **재사용 가능한 모듈/컴포넌트** 단위로 나눈다. 아토믹 디자인(Atoms → Molecules → Organisms)과 프로젝트 표준 구조를 따른다.
- **의존성 최소화**: 모듈 간 의존은 한 방향·필요 최소로 두고, 순환 의존을 만들지 않는다.
- **공통 추출**: 두 곳 이상에서 쓰이는 로직·UI·스타일은 **공통 모듈·컴포넌트·스킬/문서**로 한 번만 정의하고 참조한다.

### 3. 단순 반복 제거

- **기획**: 유사한 요구사항·Phase가 반복되면 “공통 Phase 템플릿” 또는 “공통 체크리스트”로 정리해 재사용한다.
- **디자이너**: 동일·유사 컴포넌트·토큰은 한 스펙에 모아 정의하고, 새 스펙에서 “기존 XXX 스펙 확장/참조”로만 기술한다.
- **퍼블리셔**: 동일 마크업 패턴은 공통 HTML 조각·클래스 세트로 정의하고, 문서화해 코더가 재사용하도록 한다.
- **코더**: 동일·유사 코드는 공통 함수·훅·컴포넌트·서비스로 추출하고, 중복 구현을 하지 않는다.
- **컴포넌트 관리**: 화면 컴포넌트·서버 모듈의 목록·재사용·중복은 **core-component-manager** 역할과 협업해 정리한다.

## 작업 시 체크리스트

1. **이 작업을 더 작은 캡슐화된 단위로 나눌 수 있는가?** 나눌 수 있으면 단위별 책임을 명시한다.
2. **이미 있는 모듈/컴포넌트/스펙을 재사용할 수 있는가?** 있으면 새로 만들지 않고 참조·확장만 한다.
3. **동일·유사한 작업이 다른 곳에서 반복되고 있는가?** 반복이면 공통화 후 한 곳만 유지한다.
4. **의존 관계가 단순한가?** 순환 의존·과도한 결합이 생기지 않도록 한다.
