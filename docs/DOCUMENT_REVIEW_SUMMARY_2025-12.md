## 2. 핵심 표준 원칙 요약

### 2.1 테넌트 기반 아키텍처 (최우선 원칙)

**핵심 원칙**:
- ✅ 모든 데이터는 테넌트별로 격리 (`tenant_id` 필수)
- ✅ 모든 조회는 테넌트 컨텍스트 기반
- ❌ 브랜치(branch) 개념 사용 금지
- ❌ 전역 데이터 접근 금지

**적용 범위**:
- 모든 엔티티는 `tenant_id` 필드 보유
- 모든 Repository 쿼리에 `tenant_id` 필터 필수
- 모든 Service 메서드에서 테넌트 검증 필수

### 2.2 하드코딩 금지 원칙

**핵심 원칙**:
- ✅ 모든 코드값은 공통코드 테이블에서 동적 조회
- ✅ 모든 설정값은 환경변수 또는 설정 파일 사용
- ❌ 코드 내 하드코딩된 값 금지
- ❌ 상수 클래스 사용 금지 (백엔드)

**적용 범위**:
- 공통코드 시스템: 모든 코드값 동적 조회
- 환경변수: 비밀키, 도메인, 포트 등
- API 엔드포인트: 상수 파일에 정의

### 2.3 표준화 원칙

**핵심 원칙**:
- ✅ API 버전 관리: `/api/v1/` 필수
- ✅ 표준 응답 구조: `{ success, data, message }`
- ✅ 표준 에러 처리: GlobalExceptionHandler
- ✅ CSS 변수 시스템: `var(--mg-...)` 사용

**적용 범위**:
- 모든 API는 `/api/v1/` 접두사 사용
- 모든 에러는 표준 형식으로 응답
- 모든 스타일은 CSS 변수 사용

### 2.4 보안 원칙

**핵심 원칙**:
- ✅ 최소 권한 원칙 (Principle of Least Privilege)
- ✅ 심층 방어 (Defense in Depth)
- ✅ 개인정보 암호화 (AES-GCM)
- ✅ 보안 감사 필수

**적용 범위**:
- JWT 기반 인증
- 개인정보 암호화 저장
- 보안 이벤트 로깅
- 세션 관리

### 2.5 소프트 삭제 원칙

**핵심 원칙**:
- ✅ `is_deleted` BOOLEAN 필드 필수
- ✅ `deleted_at` TIMESTAMP 필드 필수
- ❌ 하드 삭제 금지 (법적 요구 시 제외)

**적용 범위**:
- 모든 엔티티에 소프트 삭제 필드 포함
- 모든 조회 쿼리에 `is_deleted = false` 필터
