# DB 우선 개발 순서 (Database-First Development)

Core Solution(MindGarden)에서 **데이터베이스 관련 기능**을 추가·수정할 때는 반드시 아래 순서를 따릅니다.

## 1. 개발 순서 (필수)

```
1. DB 테이블·스키마 조회 및 설계  →  2. 쿼리·프로시저 수정  →  3. 백엔드 개발  →  4. 프론트엔드 개발
```

- **DB 설계 우선**: 백엔드·프론트엔드 코드를 작성하기 **전에** 먼저 DB를 조회하고 설계합니다.
- **추가·수정 필요 시**: 테이블/칼럼/인덱스 생성 또는 변경을 먼저 수행한 뒤, 그 결과를 기준으로 백엔드·프론트엔드를 개발합니다.

## 2. DB 조회 및 설계 단계

### 2.1 기존 스키마 조회

1. **관련 테이블 조회**: 기능과 연관된 테이블들을 먼저 조회합니다.
2. **칼럼·타입·제약조건 확인**: `information_schema`, `SHOW CREATE TABLE` 등으로 현재 스키마를 파악합니다.
3. **기존 쿼리·프로시저 확인**: 관련 뷰, 프로시저, 트리거 존재 여부와 내용을 확인합니다.

### 2.2 필요 시 수정 사항 도출

- **쿼리/프로시저 수정**: 기존 SQL 또는 프로시저가 요구사항을 충족하지 않으면, 이 단계에서 수정 계획을 세웁니다.
- **테이블·칼럼 추가**: 새 테이블 또는 칼럼이 필요하면 먼저 DDL(생성·수정) 스크립트를 작성하고, 적용 후 스키마가 확정된 상태에서 백엔드 개발을 시작합니다.

### 2.3 DB 적용 후 진행

- DB 스키마 변경이 끝난 뒤에만 **백엔드(Entity, Repository, Service)** 를 작성합니다.
- 백엔드 API가 준비된 뒤 **프론트엔드** 개발을 진행합니다.

## 3. 금지 사항

- ❌ 테이블·칼럼 존재 여부 확인 없이 백엔드 Entity·Repository를 작성하지 않습니다.
- ❌ DB 스키마를 확정하지 않은 채 프론트엔드에서 새 데이터 구조를 가정하고 화면을 만들지 않습니다.
- ❌ 기존 프로시저/쿼리 내용을 확인하지 않고 API를 구현하지 않습니다.

## 4. 작업 체크리스트

| 단계 | 작업 | 완료 |
|------|------|------|
| 1 | 관련 테이블·칼럼 조회 | ☐ |
| 2 | 필요 시 DDL(테이블·칼럼) 생성/수정 | ☐ |
| 3 | 쿼리·프로시저 확인 및 수정 | ☐ |
| 4 | 백엔드 Entity·Repository·Service 구현 | ☐ |
| 5 | API 개발 및 테스트 | ☐ |
| 6 | 프론트엔드 개발 및 연동 | ☐ |

## 5. 관련 서브에이전트

- **core-coder**: DB 조회 결과를 바탕으로 Entity, Repository, Service 구현 시 이 순서를 따릅니다.
- **core-tester**: DB 스키마 확정 후, 백엔드 단위·통합 테스트를 작성합니다.
