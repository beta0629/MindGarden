## 1. 운영 직전 필수 테스트·검증 체크리스트

### 1.1 단위·통합·E2E (운영 직전 우선 수행)

| 구분 | 항목 | 체크 | 비고 |
|------|------|:----:|------|
| **단위** | 백엔드 Service/Controller 단위 테스트 전체 통과 (`mvn test`) | [ ] | 커버리지 80%+ 유지, 실패 케이스 0 |
| **단위** | 변경된 모듈 단위 테스트 (SystemNotification, Admin, Schedule 등) | [ ] | 최근 수정 영역 우선 |
| **통합** | API 통합 테스트 통과 (인증·X-Tenant-ID 헤더 포함) | [ ] | `*IntegrationTest.java` |
| **통합** | 주요 API 스모크 (로그인, 알림/메시지, 대시보드, ERP 연동) | [ ] | 스테이징 또는 동일 빌드로 검증 |
| **E2E** | 로그인·대시보드·관리자 메뉴 접근 (`auth.spec.ts`, `branch-management.spec.ts` 등) | [ ] | Playwright 스펙 실행 |

| **E2E** | 상담사 대시보드·심리검사 후 로그인 플로우 (해당 스펙) | [ ] | 주요 시나리오만 선택 실행 가능 |

### 1.3 접근성·UI 검증 (대비)

| 항목 | 체크 | 비고 |
|------|:----:|------|
| 대시보드 등 주요 화면 대비(contrast) — 문서화된 이슈 반영 여부 | [ ] | B0KlA 등 분석 문서 참고 |
| 키보드 포커스·모달 포커스 트랩 (UnifiedModal) | [ ] | 필요 시 체크리스트만 명시 |
| 공통 스피너·로딩 상태 표시 | [ ] | 빈 차트/목록 시 로딩/에러 구분 |

### 1.4 최근 변경 영역 집중 검증

- **알림·메시지**: AdminNotifications 통합 UI, SystemNotification 관리, NotificationDropdown, 위젯(ClientMessageWidget, SystemNotificationWidget).
- **공통 컴포넌트**: Badge, PipelineStepBadge, UnifiedModal, LoadingSpinner, ContentHeader.
- **대시보드**: AdminDashboard V1/V2, 차트 데이터(빈 데이터 대비), AdminDashboardStats.
- **백엔드**: SystemNotificationController/Service, AdminController, Schedule 관련 변경.

위 영역에 대해 **스테이징(또는 개발)에서 1회 이상** 수동 시나리오 + 해당 E2E(있는 경우) 실행 권장.

### 1.5 환경(스테이징 vs 운영) 차이 점검

| 항목 | 체크 | 비고 |
|------|:----:|------|
| 운영 전용 설정 (application 프로파일, DB URL, 도메인) | [ ] | 프로덕션만 다른 값 확인 |
| 환경 변수 (JWT, 암호화, OAuth2 등) — 운영 값 적용 여부 | [ ] | 표준 배포 문서와 일치 |
| CORS/쿠키 도메인 (운영 도메인) | [ ] | 로그인/세션 정상 동작 |
| API baseURL / 프론트 빌드 시 환경 구분 | [ ] | 운영 API 호출 주소 |
| DB 마이그레이션 (Flyway) — 운영 DB에 미적용 스크립트 없음 | [ ] | 배포 브랜치에 마이그레이션 포함 여부 |
| 운영 배포 시 `-DskipTests` 사용 시, 테스트는 CI/로컬에서 이미 통과했는지 | [ ] | 배포 전 빌드·테스트 완료 확인 |

## 2. 테스트 관점 리스크 (요약)

- **미검증 구간**
  - Admin 알림/메시지 통합 UI, 시스템 알림 CRUD, 배지·모달 등 **신규/변경 UI**에 대한 E2E 스펙이 없을 수 있음. 수동 스모크로 보완 필요.
  - 스테이징이 없고 개발→운영 직행 시, 환경 차이(설정·도메인·DB)로 인한 오동작을 배포 전에 발견하기 어려움.

- **자동화되지 않은 검증**
  - 스모크·회귀 대부분이 수동. 배포 후 “로그인 → 대시보드 → 알림/메시지” 등 최소 시나리오를 Playwright로 자동화하면 운영 직후 검증에 유리함.
  - 접근성(대비, 포커스)은 수동 체크에 의존. 자동화(axe 등) 도입 시 이후 배포부터 활용 가능.

- **데이터·테넌트**
  - 운영 데이터로 테스트 금지(표준). 테넌트 격리·멀티테넌트 시나리오는 스테이징/통합 테스트에서 검증하고, 운영 반영 전 체크리스트에 “테넌트 격리 통합 테스트 통과” 항목 명시 권장.

- **롤백 후 검증**
  - 롤백 시 “이전 빌드 기준” 스모크를 다시 수행할 수 있도록, 롤백 체크리스트에 테스트 항목(최소 스모크 1~2개) 포함 권장.

---

## 3. 제안 요약 (회의 시 공유)

1. **운영 배포 직전**: 위 1.1~1.5 체크리스트를 순서대로 수행하고, 실패 항목은 해소 후 배포 진행.
2. **스테이징 활용**: 가능하다면 스테이징에서 1.2·1.4 항목을 먼저 수행한 뒤 운영 배포.
3. **자동화 보강**: “운영 직후 스모크” 1~2개 시나리오를 Playwright로 추가해, 배포 후 자동 실행 검토.
4. **리스크 공유**: 2장 리스크를 팀과 공유하고, 미검증 구간은 수동 범위와 담당을 명시.
