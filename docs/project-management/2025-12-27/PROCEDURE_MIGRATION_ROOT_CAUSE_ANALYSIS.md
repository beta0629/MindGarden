# 프로시저 마이그레이션 근본 원인 분석 및 해결책

## 📋 문제 요약

**현상:**
- Flyway 마이그레이션이 성공으로 기록되지만 프로시저가 실제로 생성되지 않음
- 테스트할 때마다 프로시저가 없어지는 문제 발생
- `CreateOrActivateTenant` 프로시저가 없어서 승인 프로세스 실패

## 🔍 근본 원인 분석

### 1. Flyway의 구문 분리 메커니즘
- Flyway는 기본적으로 **세미콜론(;)**으로 구문을 분리합니다
- 프로시저 본문 내부에도 세미콜론이 많아 충돌 발생
- `DELIMITER //`를 사용해도 Flyway는 이를 인식하지 못함

### 2. 현재 마이그레이션 파일 구조
```sql
DELIMITER //

DROP PROCEDURE IF EXISTS CreateOrActivateTenant //

CREATE PROCEDURE CreateOrActivateTenant(...)
BEGIN
    SET v_var = 'value';  -- 세미콜론
    SELECT * FROM table;  -- 세미콜론
    ...
END //

DELIMITER ;
```

**문제점:**
- Flyway가 `DELIMITER //`를 무시하고 세미콜론으로 구문을 분리
- `CREATE PROCEDURE` 문이 중간에 끊어짐
- 프로시저 본문이 제대로 생성되지 않음

### 3. Flyway 히스토리 vs 실제 상태
- Flyway는 마이그레이션 파일을 읽고 실행 시도
- 구문 오류가 발생해도 일부는 성공으로 기록될 수 있음
- 실제로는 프로시저가 생성되지 않았지만 Flyway는 성공으로 기록

## ✅ 근본적인 해결책

### 방안 1: Flyway 설정 변경 (권장하지 않음)
- `splitStatements: false` 설정
- **단점**: 모든 마이그레이션에 영향을 미침, 다른 SQL 구문과 충돌 가능

### 방안 2: 프로시저 생성 마이그레이션을 두 파일로 분리 (권장)
1. **DROP 파일**: `V20251227_003__drop_create_or_activate_tenant.sql`
   ```sql
   DROP PROCEDURE IF EXISTS CreateOrActivateTenant;
   ```

2. **CREATE 파일**: `V20251227_004__create_create_or_activate_tenant.sql`
   - DELIMITER 없이 작성
   - 프로시저 본문의 세미콜론을 유지하되, Flyway가 전체를 하나의 구문으로 처리하도록 함
   - **핵심**: 프로시저 본문을 동적으로 생성하여 실행

### 방안 3: Java 코드에서 프로시저 생성 (현재 백업 메커니즘)
- 애플리케이션 시작 시 프로시저 존재 여부 확인
- 없으면 마이그레이션 파일에서 읽어서 생성
- **장점**: Flyway 문제를 우회
- **단점**: Flyway 마이그레이션과 중복, 유지보수 복잡
