# 매칭 시스템 + 스케줄 통합 — 요구사항 정리

## 1. 잘못 이해했던 점 (이전 구현)

- **했던 것**: “대기 목록 + 캘린더 드래그” 형태의 **새 화면**을 만들고, **목업 데이터**만 사용
- **실제 요청**: **기존 매칭 관리**와 **기존 스케줄** 기능을 **한 화면/한 흐름으로 통합**하는 것

즉, **새로운 개념의 UI**가 아니라 **현재 있는 두 시스템을 묶는 것**이 목표입니다.

---

## 2. 현재 시스템 정리

### 2.1 매칭 시스템 (Mapping Management)

| 항목 | 내용 |
|------|------|
| **경로** | `/admin/mapping-management` |
| **컴포넌트** | `MappingManagement` → `MappingManagementPage` |
| **API** | `GET/POST /api/v1/admin/mappings` (목록, 생성, 승인, 종료 등) |
| **역할** | 상담사–내담자 **매칭** 관리 (결제 대기, 활성, 종료 등 상태 관리) |
| **UI** | KPI, 검색, 매칭 목록(카드/테이블/캘린더 뷰), 매칭 생성·상세·결제·환불 모달 등 |

### 2.2 스케줄 (Schedule)

| 항목 | 내용 |
|------|------|
| **경로** | `/admin/schedule`, `/admin/schedules` |
| **컴포넌트** | `SchedulePage`, `UnifiedScheduleComponent` (캘린더 + 모달) |
| **API** | `GET /api/v1/schedules/admin`, `POST /api/v1/schedules/consultant` 등 |
| **역할** | **일정(상담 세션)** 등록·조회·수정 (날짜/시간 단위) |
| **UI** | 캘린더 뷰, 상담사 필터, 스케줄 등록/상세 모달, 휴무 등 |

- **매칭** = “누가 누구와 매칭되어 있는지” (관계)
- **스케줄** = “언제 어떤 상담이 잡혀 있는지” (일정)

---

## 3. “통합”의 올바른 방향 (제안)

다음 중 어떤 형태에 가깝은지 확인이 필요합니다.

### 방안 A: 한 화면에 두 기능 배치 (스플릿/탭)

- **한 페이지**에서:
  - 좌측(또는 상단): **기존 매칭 목록** (`/api/v1/admin/mappings` 사용)
  - 우측(또는 하단): **기존 스케줄 캘린더** (`/api/v1/schedules/admin` 등 사용)
- 매칭 목록에서 행/카드 선택 시 → 해당 상담사·내담자로 스케줄 캘린더/필터 연동 또는 스케줄 등록 모달에 **자동 채움**

### 방안 B: 매칭 화면에서 스케줄 등록까지 한 흐름

- **매칭 관리 페이지**는 그대로 두고,
- 매칭 카드/행에 **“스케줄 등록”** 버튼 추가
- 클릭 시 **기존 스케줄 등록 모달**을 띄우고, 해당 매칭의 상담사·내담자 정보로 **자동 세팅**
- (필요 시) 등록 후 스케줄 페이지로 이동 또는 같은 페이지 내 캘린더 영역 갱신

### 방안 C: 메뉴만 통합 (진입점 통일)

- LNB에서 “매칭 관리”와 “스케줄”을 **하나의 메뉴**(예: “매칭·스케줄”)로 묶고,
- 클릭 시 **방안 A**처럼 한 화면에 매칭 + 스케줄을 같이 보여 주거나,
- 또는 **방안 B**처럼 매칭 화면에 “스케줄 등록” 액션을 강화

---

## 4. 현재 잘못 만들어 둔 것 (정리 대상)

- **경로**: `/admin/integrated-schedule`
- **구현**: `IntegratedMatchingSchedule.js`, `useMatchingScheduleStore.js` (목업 전용)
- **역할**: 위 “통합”과 무관한 **별도 목업 화면**이므로,  
  → **제거**하거나, **기존 매칭 API + 기존 스케줄 API**를 붙여 “실제 매칭·스케줄 통합”으로 **재구성**해야 함

---

## 5. 다음 단계 제안

1. **요구 확인**: 위 방안 A/B/C 중 어떤 것이 “매칭 시스템과 스케줄 통합”에 가장 가까운지 선택 (또는 조합).
2. **현행 유지**: 통합 화면에서는 **반드시**  
   - `StandardizedApi.get('/api/v1/admin/mappings')` 등 **실제 매칭 API**,  
   - `apiGet('/api/v1/schedules/admin')`, `StandardizedApi.post('/api/v1/schedules/consultant', ...)` 등 **실제 스케줄 API**  
   를 사용하도록 구현.
3. **기존 통합 페이지 처리**:  
   - `/admin/integrated-schedule`은 **실제 매칭+스케줄 통합**으로 재구성하거나,  
   - 통합을 “매칭 관리 페이지 확장”(방안 B)으로 갈 경우 해당 라우트/메뉴는 제거 검토.

이 문서를 기준으로 “매칭 시스템과 스케줄 통합” 범위를 확정한 뒤, 그에 맞춰 설계·구현을 진행하는 것이 좋습니다.
