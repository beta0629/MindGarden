### 1. `/admin/integrated-schedule` (통합 일정 관리)
- **정보 계층**: 1순위는 내담자명, 예약 시간, 그리고 회기 잔여 및 결제 상태를 직관적으로 알 수 있는 상태 배지입니다. 2순위는 담당 상담사 이름과 진행 센터 위치 정보가 되어야 합니다. 그 외에 과거 노쇼 이력이나 세부 메모 등은 과감히 사이드바 목록에서 숨기고 상세 패널로 이관해야 합니다. 이를 통해 사용자가 스캔 단계에서 꼭 필요한 정보만 소화하도록 돕습니다.
- **밀도 권장**: Comfortable 뷰를 기본값으로 제공하여 카드당 충분한 여백과 가독성을 보장해야 합니다. 공간 절약이 필요한 사용자를 위해 Compact 옵션을 제공할 수 있으나, 이는 반드시 사용자의 명시적인 토글 선택에 의해서만 활성화되어야 합니다.
- **CTA 배치**: 사이드바 카드 인라인에는 가장 빈도가 높은 Primary 액션(예: 결제 확인 또는 출석 체크) 단 1개만 배치해야 합니다. 일정 변경이나 취소, 메모 작성 등 부차적인 기능은 오버플로우(⋯) 메뉴나 상세 패널 안으로 숨겨 인지 부하를 줄입니다.
- **Side peek 적합성**: Y. 캘린더라는 공간적 맥락을 유지하면서 특정 일정의 상세 정보(결제 현황, 회기 이력 등)를 확인하려면 우측 Side peek 패널이 필수적입니다.
- **Must fix (P0) / Can wait (P2)**: Must fix(P0)는 행 클릭 시 화면 전체 이동을 방지하는 Side peek 패널 도입과 Comfortable 기본 뷰 복구입니다. Can wait(P2)는 세부 필터 상태를 사용자별로 기억하는 뷰(Saved View) 저장 기능입니다.
- **오늘型 실수 금지**: 사용자의 선택권(Toggle)이나 충분한 UX 검증 없이 1줄짜리 Compact Row를 전역 기본값으로 강제 적용하여 정보 손실과 현장의 반발을 유발하지 말 것.

### 2. `/admin/mapping-management` (매칭 관리 / MappingListBlock)
- **정보 계층**: 1순위는 현재 매칭 상태(대기/완료), 내담자 및 상담사 이름, 매칭 신청 일시입니다. 2순위는 결제 수단, 첫 상담 예정일, 서비스 종류가 되어야 하며, 상담 신청 사유 등 긴 텍스트는 목록에서 숨겨야 합니다. 이를 통해 담당자가 당장 처리해야 할 매칭 건을 빠르게 식별할 수 있습니다.
- **밀도 권장**: Table 기반의 List 뷰를 기본으로 채택하여 20개 이상의 행을 한 번에 스크롤하며 스캔할 수 있도록(Comfortable) 밀도를 설정해야 합니다. 카드 뷰는 모바일 등 제한된 환경에서만 Fallback으로 사용합니다.
- **CTA 배치**: 현재 카드에 존재하는 4~5개의 인라인 버튼은 절대 금물이며, '매칭 확정' 같은 Primary 1개 외에는 모두 우측 끝의 오버플로우(⋯) 메뉴로 통합해야 합니다. 
- **Side peek 적합성**: Y. 매칭 대기열을 좌측에서 보면서 우측 패널로 내담자의 요구사항과 상담사 캘린더 일정을 대조하는 워크플로우에 최적화되어 있습니다.
- **Must fix (P0) / Can wait (P2)**: Must fix(P0)는 Card 기반의 기본 뷰를 Table 기반 뷰로 즉시 전환하고 다중 액션을 오버플로우로 정리하는 것입니다. Can wait(P2)는 매칭 리스트와 캘린더 뷰 간의 양방향 드래그 앤 드롭 동기화 기능입니다.
- **오늘型 실수 금지**: 매칭 카드(MappingCard)에 최대 5개의 ActionButton을 일렬로 나열하여 사용자의 클릭 실수를 유발하고 시각적 피로도를 극대화하지 말 것.

### 3. `/admin/clients` (내담자 종합 관리)
- **정보 계층**: 1순위는 내담자명, 연락처, 현재 활성화된 서비스(상담/코칭) 상태 뱃지입니다. 2순위는 최근 방문일, 담당 상담사, 누적 결제/노쇼 지표이며, 상세 상담 메모나 민감한 초기 호소 문제 텍스트는 숨김 처리하여 목록을 가볍게 유지해야 합니다. 
- **밀도 권장**: 정보량이 많은 B2B 시스템 특성상 Table 형태의 Comfortable 밀도가 기본이 되어야 하며, 행 높이는 48px~56px 수준으로 유지하는 것이 바람직합니다.
- **CTA 배치**: 목록 행마다 상세, 수정, 비번 변경, 삭제 등 4개의 버튼을 늘어놓는 것은 최악의 안티 패턴이므로, '프로필 보기' Primary 액션 하나와 오버플로우 메뉴로 깔끔하게 정리해야 합니다. 이를 통해 사용자가 핵심 액션에만 집중하도록 유도합니다.
- **Side peek 적합성**: Y. 내담자 목록을 탐색하면서 특정 내담자의 최근 결제 내역이나 예약 일정을 우측 패널로 가볍게 확인하면 전체 화면 전환 비용을 크게 줄일 수 있습니다.
- **Must fix (P0) / Can wait (P2)**: Must fix(P0)는 SmallCard 기반의 기본 뷰를 탈피하여 확장성 있는 완전한 Table 뷰로 이관하는 작업입니다. Can wait(P2)는 내담자 그룹별 타겟팅 일괄 발송(SMS/Email) 기능의 고도화입니다.
- **오늘型 실수 금지**: 화면을 압축한다는 명목으로 Compact 카드 안에 4개 이상의 버튼을 욱여넣어 컴포넌트 레이아웃이 깨지거나 터치 영역이 겹치게 만들지 말 것.

### 4. `/admin/dashboard` (어드민 대시보드 위젯)
- **정보 계층**: 1순위는 금일 발생한 핵심 지표(신규 예약, 미결제 건수, 취소/노쇼 알림)와 경고 뱃지입니다. 2순위는 주간 상담 추이 그래프나 상담사별 가동률 요약이며, 개별 내담자의 구체적 개인정보는 숨기고 철저히 숫자로만 요약해야 합니다. 대시보드의 본질은 빠르고 정확한 '현황 파악'과 '문제 인지'이기 때문입니다.
- **밀도 권장**: 대시보드의 파편화된 위젯 형태이므로 Compact 밀도를 유지하여, 한 화면(Viewport) 내에 다양한 지표 블록이 스크롤 없이 최대한 많이 들어오도록 배치해야 합니다.
- **CTA 배치**: 개별 데이터 행에 대한 인라인 액션 버튼은 모두 제거하고, 위젯 상단이나 하단에 '전체 보기(View All)' 형태의 Primary 텍스트 링크 하나만 두어 해당 관리 페이지로 자연스럽게 유도해야 합니다.
- **Side peek 적합성**: N. 대시보드에서는 즉각적인 상세 작업보다는 개괄적 전체 지표 확인이 주 목적이므로, 상세 내용이 필요할 경우 Side peek보다는 해당 전용 기능 페이지로 완전 전환(Routing)하는 것이 구조적으로 맞습니다.
- **Must fix (P0) / Can wait (P2)**: Must fix(P0)는 단순 테이블로 표시되어야 할 지표를 굳이 ProfileCard 목록으로 나열하여 공간을 낭비하는 위젯들을 Table/List 포맷으로 롤백하는 것입니다. Can wait(P2)는 사용자가 직접 위젯 간의 레이아웃을 편집할 수 있는 커스텀 드래그 앤 드롭(DnD) 뷰 기능입니다.
- **오늘型 실수 금지**: '상세한 정보를 주겠다'는 이유로 좁은 대시보드 위젯 안에 과도한 카드 UI와 다중 인라인 버튼을 배치하여 화면의 복잡도만 높이는 실수를 범하지 말 것.
